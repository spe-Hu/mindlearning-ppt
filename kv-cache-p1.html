KV Cache 横纵深度分析

纵轴：从 Transformer 诞生到 2026 年的演进史

第一纪元：注意力的原罪（2017-2019）

2017 年，Vaswani 等人在划时代的论文 "Attention Is All You Need" 中提出了 Transformer 架构。这个架构的核心——自注意力机制（Self-Attention）——彻底改变了序列建模的范式。然而，这个设计在带来并行训练优势的同时，也埋下了一个深远的隐患：在自回归解码时，每一步生成新 token 都需要重新计算之前所有 token 的注意力。

自回归语言模型（如 GPT 系列）的解码过程是逐步生成的。在时间步 t，模型需要计算：

Attention(Q_t, K_{1:t}, V_{1:t}) = softmax(Q_t · K_{1:t}^T / √d_k) · V_{1:t}

其中 Q_t 是当前 token 的查询向量，K_{1:t} 和 V_{1:t} 分别是之前所有 token（包括当前 token）的键向量和值向量。如果不做任何缓存，在生成第 t 个 token 时，模型需要对前 t-1 个 token 重新进行前向传播以获得它们的 K 和 V 表示——这意味着每一步的计算量随序列长度线性增长，整体生成 N 个 token 的总计算量为 O(N^3)，极其低效。

KV Cache 的诞生正是为了解决这一问题：既然之前 token 的 K 和 V 在后续步骤中不会改变（因为自回归的因果掩码保证了每个 token 只能看到它之前的 token），那么我们可以把这些 K、V 向量缓存起来，避免重复计算。这一优化将解码阶段的计算复杂度从 O(N^3) 降低到 O(N^2)，是一个质的飞跃。

然而，这个"用空间换时间"的策略也带来了新的问题——内存消耗随序列长度线性增长。对于一个具有 L 层、h 个注意力头的 Transformer 模型，每个 token 需要缓存的 KV 空间为：

KV_cache_size_per_token = 2 × L × h × d_k × sizeof(dtype)

以 LLaMA-2 70B 为例（80 层，8 个 KV 头，d_k=128，FP16）：

每 token KV = 2 × 80 × 8 × 128 × 2 bytes = 327,680 bytes ≈ 320 KB

当上下文长度从 4K 增长到 128K 时，单个请求的 KV Cache 从约 1.28 GB 膨胀到约 40.96 GB——直接涨了 32 倍。这正是文章中提到的"Token 消耗呈指数级增长，KV Cache 随之线性激增"的核心含义。

第二纪元：大规模模型的内存危机（2020-2022）

随着 GPT-3（175B 参数，2020）的发布和模型规模的指数级膨胀，KV Cache 的内存问题从"需要注意"升级为"系统性瓶颈"。

关键数据对比（2020-2022 年典型模型）：

模型 参数量 层数 KV头数 d_k 4K上下文KV Cache 8K上下文KV Cache
GPT-3 175B 175B 96 96 128 ~48 GB ~96 GB
PaLM 540B 540B 118 48 128 ~135 GB ~270 GB
Chinchilla 70B 70B 80 8 128 ~1.3 GB ~2.6 GB

这个阶段暴露了一个深刻的矛盾：模型参数本身是静态的，可以放在 GPU 显存中不动；但 KV Cache 是动态的，随并发请求数和上下文长度线性增长。在 GPU 显存有限的约束下，系统能同时服务的请求数（batch size）被 KV Cache 的内存占用严重制约。

这催生了一个关键洞察：Transformer 推理的瓶颈正在从"计算密集型"转向"内存密集型"。GPU 的算力其实并未被充分利用，真正卡脖子的是 HBM（高带宽内存）的容量和带宽。

第三纪元：优化方案的爆发（2023-2024）

2023 年是 KV Cache 优化的分水岭。多项突破性技术在这一年涌现：

1. Multi-Query Attention (MQA) - Shazeer 2019/2023 实践

MQA 的核心思想极其简洁：让所有注意力头共享同一组 K 和 V，只保留不同的 Q。这将 KV Cache 的大小从 h × d_k 降低到 1 × d_k，减少了 h 倍。

标准 MHA: 每 token KV = 2 × L × h × d_k
MQA:     每 token KV = 2 × L × 1 × d_k    # 减少 h 倍

代价是模型表达能力的轻微下降，但在实践中，MQA 对模型质量的影响微乎其微，而对吞吐量的提升却非常显著。

2. Grouped-Query Attention (GQA) - Ainslie 2023

GQA 是 MQA 的推广版本，将 h 个查询头分成 g 组（1 < g < h），每组共享一组 KV。当 g=1 时退化为 MQA，当 g=h 时退化为标准 MHA。

GQA:     每 token KV = 2 × L × g × d_k    # g 为 KV 头数，1 < g < h

LLaMA-2 70B 使用 g=8（8 个 KV 头对应 64 个查询头），在模型质量和推理效率之间取得了优秀的平衡。GQA 已经成为 2023-2024 年大模型的标配。

3. PagedAttention - Kwon 2023 (vLLM)

如果说 MQA/GQA 减少了每个 token 的 KV 大小，那么 PagedAttention 则从内存管理层面彻底革新了 KV Cache 的分配方式。

传统系统中，KV Cache 需要预分配连续的显存空间，且大小按最大序列长度预留。这导致了严重的内存碎片化和内部浪费——实际序列可能只有 1000 个 token，但预分配了 2048 个 token 的空间，浪费率高达 50%。

vLLM 借鉴了操作系统虚拟内存的经典思想，将 KV Cache 切分为固定大小的"页"（block），每页存储固定数量 token（如 16 个）的 KV 向量。这些页不需要连续存储，通过一个"块表"（block table）来映射逻辑关系。

传统方式: [====连续分配====][浪费][====连续分配====][浪费]
PagedAttention: [页1][页3][页7][页2][页5]... → 块表映射逻辑顺序

PagedAttention 的效果是革命性的： - 内存浪费从 ~50% 降低到 <4%（仅最后一页的内部碎片） - 同等显存下的吞吐量提升 2-4 倍 - 支持动态的 copy-on-write，实现 beam search 和 parallel sampling 的高效 KV 共享

4. Multi-head Latent Attention (MLA) - DeepSeek 2024

DeepSeek-V2 提出的 MLA 是一种更加激进的压缩策略。它不是简单地减少 KV 头数，而是将 K 和 V 通过一个低秩联合压缩矩阵投影到一个更低维度的潜在空间中进行缓存：

压缩: c_t = W_DKV · h_t     # W_DKV ∈ R^{d_c × d_model}, d_c << d_model
解压: K_t = W_UK · c_t, V_t = W_UV · c_t   # 在需要计算注意力时解压

这使得 KV Cache 的大小进一步降低，DeepSeek-V2 报告 KV Cache 减少了约 93.3%，同时模型质量几乎无损。

第四纪元：长上下文时代与 2026 现状

2025-2026 年，长上下文窗口成为大模型的标配能力。Claude 3.5 支持 200K 上下文，Gemini 1.5 Pro 支持 1M 上下文，GPT-4o 支持 128K。上下文长度的爆炸式增长让 KV Cache 问题变得更加尖锐。

2026 年 KV Cache 优化的主要方向：

  1. KV Cache 量化：将 KV Cache 从 FP16 量化到 INT8 甚至 INT4，直接减半或再减半内存占用。关键技术包括 KIVI、KVQuant 等。

  2. KV Cache 驱逐策略：并非所有 token 的 KV 都同等重要。基于注意力分数的驱逐策略（如 H2O、Scissorhands、StreamingLLM）可以动态丢弃不重要的 KV 条目。

  3. 稀疏注意力 + KV Cache 协同：滑动窗口注意力、局部注意力等稀疏模式天然减少了需要缓存的 KV 数量，与 KV Cache 优化形成协同效应。

  4. ** speculative decoding 与 KV Cache**：推测性解码中的 draft model 和 target model 之间的 KV Cache 复用和验证机制。

  5. 硬件协同设计：Groq 的 LPU、Cerebras 的 Wafer Scale Engine 等新型硬件从架构层面重新思考 KV Cache 的存储和访问模式。

横轴：KV Cache 与关键变量的关系

1. KV Cache 与模型规模

KV Cache 的大小与模型参数量的关系并非简单的线性关系，而是取决于架构选择：

KV_cache_total = batch_size × seq_len × 2 × L × n_kv × d_k × dtype_size

其中 n_kv 是 KV 头数（MHA 时等于注意力头数 h，GQA 时等于组数 g，MQA 时为 1）。

关键洞察：模型参数量增长主要影响的是"静态显存"（模型权重本身），而 KV Cache 影响的是"动态显存"。在推理服务中，动态显存往往是更大的瓶颈，因为它随并发请求数线性增长。

配置 模型权重 (FP16) 单请求 KV (128K, FP16) KV/权重比
LLaMA-2 7B, MHA 14 GB ~20 GB 1.43
LLaMA-2 70B, GQA 140 GB ~41 GB 0.29
DeepSeek-V2, MLA 42 GB (FP8) ~2.8 GB 0.07

可以看到，GQA 和 MLA 等优化技术极大地改变了 KV Cache 与模型权重的比例关系。

2. KV Cache 与上下文长度

KV Cache 与上下文长度是严格的线性关系：

KV_cache ∝ seq_len

这意味着： - 4K → 128K：32 倍增长 - 4K → 1M：256 倍增长

在百万级上下文的场景下，单个请求的 KV Cache 就可能超过单卡显存。这推动了： - 多卡 KV Cache 分布式存储：将 KV Cache 切分到多个 GPU 上 - KV Cache offloading：将部分 KV Cache 卸载到 CPU 内存或 NVMe - 分层 KV Cache：对不同层使用不同的缓存策略

3. KV Cache 与吞吐量

吞吐量（tokens/s）与 KV Cache 的关系通过显存容量约束来体现：

max_batch_size = (GPU_memory - model_weights) / (seq_len × per_token_kv)

在显存固定的情况下，KV Cache 越大，能同时处理的请求数越少，吞吐量越低。这就是为什么 KV Cache 优化本质上是一个吞吐量优化问题。

不同优化技术对吞吐量的影响：

技术 KV Cache 缩减比 吞吐量提升 质量影响
MQA (h=32→1) 32x ~2.5x <0.5% PPL 退化
GQA (h=32→8) 4x ~1.8x <0.1% PPL 退化
MLA (DeepSeek) ~15x ~2.2x 几乎无损
PagedAttention ~1.3x (利用率) ~2-4x 无损
KV INT8 量化 2x ~1.5x <0.3% PPL 退化
StreamingLLM (窗口256) ~500x (128K→256) ~10x+ 任务相关

4. KV Cache 与延迟

KV Cache 不仅影响吞吐量，也影响延迟。在 decode 阶段，每一步都需要读取整个 KV Cache 来计算注意力：

decode_latency ∝ KV_cache_size / memory_bandwidth

这就是所谓的"内存墙"问题——GPU 的计算能力远超其内存带宽，导致 decode 阶段成为 memory-bound 操作。以 A100 为例： - FP16 算力：312 TFLOPS - HBM 带宽：2,039 GB/s - 读取 40 GB KV Cache 的时间：~20 ms - 对应的"有效算力"：仅约 2 TFLOPS（利用率的不到 1%）

核心矛盾：KV Cache 是推理阶段的"内存税"

矛盾的本质

KV Cache 的核心矛盾可以用一句话概括：它是 Transformer 自回归解码的"内存税"——你无法避免缴纳，只能通过架构优化来降低税率。

这个矛盾的根源在于自回归生成的本质：每一步生成都依赖于之前所有步骤的输出。在 Transformer 中，这种依赖通过注意力机制实现，而注意力机制需要访问所有之前 token 的 K 和 V 向量。KV Cache 是避免重复计算这些 K/V 的唯一方式。

矛盾的三个维度

维度一：质量 vs 效率 - 保留完整的 KV Cache → 模型质量最优，但内存消耗最大 - 压缩/量化/驱逐 KV Cache → 节省内存，但可能损失质量 - 最优平衡点取决于具体应用场景

维度二：延迟 vs 吞吐 - 大 batch size → 高吞吐，但 KV Cache 总量增大，可能超出显存 - 小 batch size → 低延迟，但 GPU 利用率低 - Continuous batching（如 vLLM）通过动态调度来缓解这一矛盾

维度三：训练 vs 推理 - 训练时 KV Cache 不是问题（可以并行计算所有位置） - 推理时 KV Cache 是核心瓶颈（必须自回归逐步生成） - 这导致训练和推理的最优架构选择可能不同

破局方向

2026 年，几个方向正在从根本上挑战 KV Cache 的必要性：

  1. 线性注意力（Linear Attention）：通过核方法或状态空间模型将注意力的 O(n^2) 复杂度降到 O(n)，从根源上消除 KV Cache 的需求。代表工作：Mamba、RWKV、RetNet。

  2. 无 KV Cache 架构：如 DeepSeek 的 DualPipe 等方案，通过重新设计注意力计算流程来减少或消除 KV Cache。

  3. 硬件革命：近存计算（Processing-in-Memory）、光计算等新技术可能从根本上改变内存带宽的约束。

2026 年现状总结

截至 2026 年，KV Cache 优化的格局如下：

  1. GQA 已成为默认选择：几乎所有新发布的大模型都采用 GQA 或 MQA。
  2. PagedAttention 是推理引擎的标配：vLLM、TensorRT-LLM、SGLang 等主流推理框架都实现了 PagedAttention。
  3. KV Cache 量化正在普及：INT8 KV Cache 已经成为生产环境的常见配置。
  4. 长上下文推动新需求：100K+ 上下文窗口让 KV Cache 管理从"优化项"变为"核心挑战"。
  5. SSM/线性注意力作为替代方案持续发展：Mamba-2、基于状态空间的方法在特定场景中展现出优势，但尚未撼动 Transformer 的主导地位。

KV Cache 的故事，本质上是整个大模型推理优化的缩影——在算力、内存、延迟、质量的多维约束中寻找最优解。这个搜索过程远未结束。
