KV Cache 知识卡片集

卡片 1：原理卡——KV Cache 是什么？

正面：一句话定义

KV Cache（Key-Value 缓存）是 Transformer 模型在自回归推理过程中，将已计算过的 Key 和 Value 向量存储在 GPU 显存中，避免重复计算的数据结构。

背面：核心原理

为什么需要 KV Cache？

Transformer 的自回归解码是逐步进行的。在生成第 t 个 token 时，模型需要计算：

output_t = Attention(Q_t, K_{1:t}, V_{1:t})
         = softmax(Q_t @ K_{1:t}^T / sqrt(d_k)) @ V_{1:t}

其中 K_{1:t} 和 V_{1:t} 包含了从位置 1 到 t 的所有 Key 和 Value 向量。

没有 KV Cache 的情况： - 生成 token 3 时，需要重新计算 token 1, 2 的 K, V - 生成 token 4 时，需要重新计算 token 1, 2, 3 的 K, V - 生成 token N 时，总计算量为 O(N^3)

有 KV Cache 的情况： - token 1 的 K, V 在第一步计算后缓存 - token 2 的 K, V 在第二步计算后缓存 - 生成 token N 时，只需计算 token N 的 K, V，然后与缓存的 K_{1:N-1}, V_{1:N-1} 拼接 - 总计算量降为 O(N^2)

KV Cache 的存储结构：

KV Cache 结构示意:

Layer 0:  K_cache = [K_1, K_2, ..., K_t]    # shape: [t, n_kv, d_k]
          V_cache = [V_1, V_2, ..., V_t]    # shape: [t, n_kv, d_v]

Layer 1:  K_cache = [K_1, K_2, ..., K_t]
          V_cache = [V_1, V_2, ..., V_t]
          ...

Layer L-1: K_cache = [K_1, K_2, ..., K_t]
           V_cache = [V_1, V_2, ..., V_t]

关键约束：KV Cache 的正确性依赖于因果掩码——因为每个 token 只能看到它之前的 token，所以之前 token 的 K/V 不会随新 token 的加入而改变。

卡片 2：公式卡——KV Cache 内存计算公式

核心公式

KV Cache 总内存 = batch_size × seq_len × 2 × num_layers × num_kv_heads × head_dim × dtype_size

变量说明

变量 含义 典型值
batch_size 并发请求数 1-128
seq_len 序列长度（上下文 + 生成） 512 - 1,000,000
num_layers Transformer 层数 32-128
num_kv_heads KV 头数 1 (MQA) / g (GQA) / h (MHA)
head_dim 每个头的维度 64-256
dtype_size 数据类型字节数 2 (FP16) / 1 (INT8) / 0.5 (INT4)

速算表

速算公式

快速估算（FP16, 单请求）:

KV Cache (GB) ≈ seq_len(K) × num_layers × num_kv_heads × head_dim × 4 / 1,000,000

简化版:
KV Cache (GB) ≈ seq_len(K) × per_token_KV (KB) / 1,000,000

例: LLaMA-2 70B, 128K 上下文
= 128,000 × 320 KB / 1,000,000
= 40.96 GB ≈ 41 GB

MQA vs GQA vs MHA 结构对比

Multi-Head Attention (MHA):
Q1 K1 V1    Q2 K2 V2    Q3 K3 V3    Q4 K4 V4
 ↓   ↓  ↓    ↓   ↓  ↓    ↓   ↓  ↓    ↓   ↓  ↓
head1       head2       head3       head4

Multi-Query Attention (MQA):
Q1 K  V    Q2 K  V    Q3 K  V    Q4 K  V
 ↓  ↓  ↓    ↓  ↓  ↓    ↓  ↓  ↓    ↓  ↓  ↓
head1       head2       head3       head4
KV Cache = 1 × d_k (每层)     # 所有头共享 K, V

Grouped-Query Attention (GQA, g=2):
Q1 Q2 K1 V1    Q3 Q4 K2 V2
 ↓   ↓  ↓  ↓    ↓   ↓  ↓  ↓
head1 head2    head3 head4
KV Cache = 2 × d_k (每层)     # 每 2 个头共享一组 K, V

卡片 4：内存计算卡——实际场景的 KV Cache 预算

场景 1：单卡部署 LLaMA-2 7B

硬件: 1 × A100 80GB
模型权重 (FP16): 14 GB
框架开销: ~5 GB
可用于 KV Cache: 80 - 14 - 5 = 61 GB

每 token KV (MHA, 32 头): 2 MB
最大总 token 数: 61 GB / 2 MB = 30,500 tokens

配置方案:
- 8 个并发请求 × 4K 上下文 = 32K tokens → 64 GB KV → 超出!
- 4 个并发请求 × 4K 上下文 = 16K tokens → 32 GB KV → 可行 ✓
- 16 个并发请求 × 2K 上下文 = 32K tokens → 64 GB KV → 超出!
- 8 个并发请求 × 2K 上下文 = 16K tokens → 32 GB KV → 可行 ✓

结论: 单 A100 部署 LLaMA-2 7B，最多支持 ~8 个并发短对话请求

场景 2：多卡部署 LLaMA-2 70B

硬件: 4 × A100 80GB = 320 GB 总显存
模型权重 (FP16): 140 GB (张量并行分布在 4 卡)
框架开销: ~20 GB
可用于 KV Cache: 320 - 140 - 20 = 160 GB

每 token KV (GQA, 8 头): 320 KB
最大总 token 数: 160 GB / 320 KB = 500,000 tokens

配置方案:
- 4 个并发 × 32K 上下文 = 128K tokens → 41 GB → 轻松 ✓
- 2 个并发 × 128K 上下文 = 256K tokens → 82 GB → 可行 ✓
- 1 个并发 × 500K 上下文 = 500K tokens → 160 GB → 极限 ✓

结论: 4 × A100 可以支持 1 个 500K 上下文的超长请求

场景 3：百万上下文部署

目标: 支持 1M 上下文, LLaMA-2 70B (GQA)
单请求 KV Cache: 328 GB

硬件需求:
- 模型权重: 140 GB
- KV Cache: 328 GB
- 总计: 468 GB + 框架开销 ~20 GB = ~490 GB

方案:
- 8 × A100 80GB = 640 GB → 可行 ✓ (剩余 ~150 GB 用于其他请求)
- 6 × A100 80GB = 480 GB → 极限 ✓ (几乎无余量)
- 4 × A100 80GB = 320 GB → 不可行 ✗

优化后 (MLA, DeepSeek 方案):
- 单请求 KV Cache: ~7.7 GB (1M × 60 KB)
- 总计: 140 + 7.7 + 20 = ~168 GB
- 2 × A100 80GB = 160 GB → 接近极限
- 4 × A100 80GB → 轻松 ✓

结论: MLA 让百万上下文从"需要 6-8 卡"变为"2-4 卡即可"

卡片 6：代码卡——KV Cache 实现要点

PyTorch 原生 KV Cache 实现

import torch
import torch.nn.functional as F

class KVCache:
    """简单的 KV Cache 实现"""

    def __init__(self, batch_size, max_seq_len, num_layers, num_kv_heads, head_dim, device):
        # 预分配 KV Cache 空间
        self.cache_k = torch.zeros(
            num_layers, batch_size, max_seq_len, num_kv_heads, head_dim,
            dtype=torch.float16, device=device
        )
        self.cache_v = torch.zeros(
            num_layers, batch_size, max_seq_len, num_kv_heads, head_dim,
            dtype=torch.float16, device=device
        )
        self.seq_lens = torch.zeros(batch_size, dtype=torch.long, device=device)

    def update(self, layer_id, batch_ids, new_k, new_v):
        """在指定位置写入新的 KV"""
        positions = self.seq_lens[batch_ids]
        for i, (batch_id, pos) in enumerate(zip(batch_ids, positions)):
            self.cache_k[layer_id, batch_id, pos] = new_k[i]
            self.cache_v[layer_id, batch_id, pos] = new_v[i]

    def get(self, layer_id, batch_id):
        """获取指定层和 batch 的完整 KV"""
        seq_len = self.seq_lens[batch_id]
        return self.cache_k[layer_id, batch_id, :seq_len], \
               self.cache_v[layer_id, batch_id, :seq_len]

    def increment(self, batch_ids):
        """序列长度 +1"""
        self.seq_lens[batch_ids] += 1


def attention_with_kv_cache(query, key_cache, value_cache, num_kv_heads, num_q_heads):
    """使用 KV Cache 的注意力计算（支持 GQA）"""
    # query: [batch, 1, num_q_heads, head_dim]
    # key_cache, value_cache: [batch, seq_len, num_kv_heads, head_dim]

    batch_size, seq_len, _, head_dim = key_cache.shape
    groups = num_q_heads // num_kv_heads

    # GQA: 将 KV 头扩展以匹配 Q 头
    if groups > 1:
        key_cache = key_cache.repeat_interleave(groups, dim=2)
        value_cache = value_cache.repeat_interleave(groups, dim=2)

    # 计算注意力分数
    # query: [batch, 1, num_q_heads, head_dim] -> [batch, num_q_heads, 1, head_dim]
    q = query.squeeze(1).transpose(1, 2)  # [batch, num_q_heads, 1, head_dim]
    k = key_cache.transpose(1, 2)         # [batch, num_q_heads, seq_len, head_dim]
    v = value_cache.transpose(1, 2)       # [batch, num_q_heads, seq_len, head_dim]

    # 缩放点积注意力
    scores = torch.matmul(q, k.transpose(-2, -1)) / (head_dim ** 0.5)
    attn_weights = F.softmax(scores, dim=-1)
    output = torch.matmul(attn_weights, v)  # [batch, num_q_heads, 1, head_dim]

    return output

关键实现要点

  1. 预分配 vs 动态分配：预分配避免运行时内存分配的开销，但可能造成浪费。PagedAttention 采用动态分配。

  2. GQA 的 KV 扩展：KV 头数少于 Q 头数时，需要通过 repeat_interleave 将 KV 扩展以匹配 Q 头数。

  3. RoPE 位置编码：使用 RoPE 时，K 和 Q 需要在计算注意力前应用旋转位置编码。缓存的 K 是已经应用了位置编码的版本。

  4. 内存连续性：KV Cache 的内存布局对性能影响很大。推荐使用 [batch, seq_len, heads, dim] 的布局，确保同一 token 的 KV 在内存中连续。

关键观察

  1. GQA 已成为绝对主流：2023 年后的模型几乎全部采用 GQA，KV 头数通常为 4-8。
  2. MLA 是效率之王：DeepSeek-V2 的每 token KV 仅 60 KB，是 GQA 模型的 1/5 到 1/2。
  3. head_dim 的变化：不同模型选择不同的 head_dim（64-256），直接影响每 token KV 大小。
  4. 模型大小不等于 KV Cache 大小：LLaMA-2 7B (MHA) 的 KV Cache 远大于 LLaMA-3 8B (GQA)，尽管参数量相近。
