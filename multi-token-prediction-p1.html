多Token预测 (Multi-Token Prediction / MTP) 横纵分析报告

研究时间：2026-06-28 | 所属领域：AI/模型训练与推理 | 研究对象类型：技术概念 关键词：Multi-Token Prediction, MTP, Next-Token Prediction, DSpark, Training-Inference Co-design

一、一句话定义

多Token预测（Multi-Token Prediction, MTP）是大语言模型训练和推理的范式革新：在训练阶段，模型不仅学习预测下一个 token，还同时学习预测未来第 2、3、...、N 个 token；在推理阶段，这些额外的预测头可以直接作为推测解码的草稿生成器，实现训练-推理的协同优化。 其本质是"从逐步猜测到批量预知"——将模型的视野从"只看下一步"扩展为"同时看到多个未来"。从标准 next-token prediction（1990s 语言模型的原点），到 DeepSeek-V3 首次在工业级模型中集成 MTP 模块（2024），到 DSpark 在 DeepSeek-V4 上实现 60-85% 推理加速（2025），MTP 正在改变大模型"训练时不想推理、推理时不管训练"的割裂状态。

二、纵向分析：从 next-token 到 multi-token（1990s-2026）

2.1 标准范式：Next-Token Prediction（1990s-2023）

自语言模型诞生以来，标准训练目标就是 next-token prediction（NTP）：给定前面的 token 序列，预测下一个 token。

标准 NTP：
输入: [token_1, token_2, ..., token_t]
目标: 最大化 P(token_{t+1} | token_1, ..., token_t)
每步只预测一个未来 token

这个范式极其成功——从 n-gram 语言模型（1990s）到 RNN/LSTM（2000s）到 Transformer（2017）到 GPT/BERT/LLaMA（2020s），所有模型都在优化同一个目标：预测下一个 token。

但 NTP 有一个根本性的低效：每次前向传播只产出 1 个 token 的监督信号。模型的第 t 步前向传播计算了所有层的隐藏状态，但只用了最后一个隐藏状态来预测 token_{t+1}。中间层的计算结果没有被充分利用。

2.2 早期探索：Multi-Step Prediction（2018-2022）

学术界很早就注意到了 NTP 的低效，并探索了多步预测的变体。

Teacher Forcing 的局限：标准训练中，模型在每一步都使用真实的 token 作为输入（teacher forcing）。但推理时，模型使用自己生成的 token（auto-regressive）。这种训练-推理不匹配（exposure bias）导致误差累积。多步预测可以缓解这个问题——如果模型能预测未来多个 token，它对自身生成的误差就更鲁棒。

Scheduled Sampling（Bengio et al., 2015）：在训练时随机选择使用真实 token 或模型自己生成的 token 作为输入，缓解 exposure bias。这可以看作多步预测的早期形式。

Blockwise Parallel Decoding（Stern et al., 2018）：训练浅层辅助模型并行预测多个 token，然后用目标模型验证。这是推测解码的前身，也是多步预测在推理加速中的首次应用。

但这些早期工作都没有将多步预测深度集成到模型训练目标中——它们要么是训练策略的调整（Scheduled Sampling），要么是推理时的外挂模块（Blockwise Parallel）。

2.3 DeepSeek-V3：MTP 的工业级首次集成（2024）

2024 年底，DeepSeek-V3 技术报告首次公开了 Multi-Token Prediction 在工业级模型中的集成。

DeepSeek-V3 的 MTP 设计： 1. 训练目标扩展：在标准 NTP 损失之外，增加了对未来第 2、3、...、N 个 token 的预测损失。 2. 模块化设计：MTP 模块是模型的附加组件，不影响主模型的架构。训练时，MTP 模块与主模型联合训练；推理时，MTP 模块可以选择性启用。 3. 损失加权：MTP 损失与 NTP 损失通过加权系数 alpha 组合：L_total = L_NTP + alpha * L_MTP。alpha 的选择影响 MTP 模块的预测质量。

DeepSeek-V3 的 MTP 模块在推测解码中作为草稿生成器使用，为后续的 DSpark 奠定了基础。

2.4 DSpark：MTP 驱动的推理加速（2025）

2025 年初，DeepSeek 发布 DSpark 推理加速框架（梁文锋署名 + 北京大学合作），这是 MTP 从训练优化到推理加速的关键跨越。

DSpark 的核心创新在于将 MTP 模块深度集成到推理流程中： 1. MTP 作为草稿生成器：推理时，MTP 模块直接利用主模型的隐藏状态来预测未来多个 token，无需独立的草稿模型。 2. 动态推测深度：根据上下文的复杂度动态调整 MTP 的预测深度（预测几个未来 token）。简单文本用深推测（5+ token），复杂文本用浅推测（1-2 token）。 3. 跨模型通用性：DSpark 的框架设计不依赖于 DeepSeek 的特定架构，在 Qwen3 等第三方模型上也能实现约 30% 的加速。

性能数据： | 模型 | 场景 | 基线速度 | DSpark 速度 | 提升 | |------|------|----------|-------------|------| | DeepSeek-V4-Flash | 通用对话 | 100 tok/s | 185 tok/s | 85% | | DeepSeek-V4-Pro | 代码生成 | 60 tok/s | 99 tok/s | 65% | | Qwen3-32B | 通用对话 | 80 tok/s | 104 tok/s | 30% |

2.5 MTP 的理论基础（2025-2026）

2025-2026 年，学术界开始系统性地研究 MTP 的理论基础。

为什么 MTP 有效？ 1. 信息共享：预测未来第 k 个 token 和预测未来第 1 个 token 共享大量的中间计算。如果模型已经计算了第 1 个 token 的隐藏状态，预测第 2 个 token 的额外成本很低。 2. 表示学习：多步预测迫使模型学习更丰富的表示——不仅要理解当前上下文，还要理解上下文的未来演化。这提高了模型的泛化能力。 3. 推测加速：训练时学会的多步预测能力，在推理时可以直接用于推测解码——MTP 模块生成的草稿质量远高于独立草稿模型，因为 MTP 模块与主模型共享表示。

MTP 的最优深度： 理论上，MTP 应该预测到未来多远？这取决于 token 之间的条件互信息。如果未来第 k 个 token 与当前 token 的条件互信息为零（给定中间 token），则预测第 k 个 token 没有额外价值。实际中，自然语言的 token 之间存在长程依赖，MTP 深度通常为 3-5 个 token。

2.6 现状与趋势（2026）

截至 2026 年中，MTP 正在成为大模型训练的标配组件：

  • DeepSeek-V4 全系列集成 MTP 模块
  • Qwen3 在最新版本中引入了 MTP 训练
  • LLaMA 4 探索了 MTP 与 MoE 的结合
  • 开源社区 的推理引擎（vLLM、SGLang）已支持 MTP 驱动的推测解码

技术趋势： 1. MTP + MoE：MTP 模块与 MoE 架构的结合——MTP 模块只激活部分专家，减少额外计算成本 2. 自适应 MTP 深度：根据输入复杂度自动调整 MTP 深度 3. MTP 与推理时 Scaling 的结合：MTP 预测多个未来 token，推理时 Scaling 在多个候选中选择最优——两者协同 4. 多模态 MTP：将 MTP 从文本扩展到视觉、语音等多模态

3.2 MTP vs 推测解码

MTP 和推测解码不是竞争关系，而是互补关系： - 推测解码是一个框架——定义了"草稿生成+验证"的两阶段流程 - MTP是一种草稿生成方法——用模型自身的多步预测能力生成草稿 - MTP 是推测解码的一种"高级实现"——草稿质量更高，因为草稿生成器和验证器共享表示

类比：推测解码是"用秘书起草+老板审批"的流程。MTP 相当于"老板自己先快速想一遍"——因为老板最了解自己的审批标准，所以草稿质量更高。

3.3 MTP vs 并行解码

并行解码（Stern 2018）也预测多个 token，但关键区别： - 并行解码的辅助模型是独立训练的浅层模型 - MTP 的预测模块与主模型深度集成，共享表示 - MTP 的预测质量更高，因为利用了主模型的深层表示

3.4 MTP vs Medusa

Medusa（2024）在目标模型上添加多个解码头，与 MTP 有相似之处： - 两者都在目标模型上添加额外的预测头 - 区别：Medusa 的解码头是在已有模型上微调的（推理时优化），MTP 的预测模块是在预训练时联合训练的（训练时优化） - MTP 的预测质量更高，因为从预训练开始就学习多步预测

3.5 MTP 的深层意义：训练-推理协同设计

MTP 最重要的意义不在于推理加速本身，而在于它开创了"训练-推理协同设计"的范式：

传统范式：
训练 → 模型 → 推理 → 各种优化（量化/蒸馏/推测解码）
训练和推理是割裂的——训练时不考虑推理优化

MTP 范式：
训练（含 MTP 目标）→ 模型（含 MTP 模块）→ 推理（MTP 直接用于推测解码）
训练和推理是一体的——训练时就为推理做准备

这种范式可以推广到其他推理优化： - 训练时加入量化感知（Quantization-Aware Training）→ 推理时量化损失更小 - 训练时加入蒸馏目标 → 推理时可以更早退出 - 训练时加入推测目标（MTP）→ 推理时推测解码更快

五、产业影响

5.1 对模型训练的影响

MTP 改变了模型训练的目标函数： - 从单一的 NTP 损失 → NTP + MTP 联合损失 - 训练成本略有增加（多步预测的额外计算），但推理效率大幅提升 - 训练时需要额外的超参数调优（MTP 深度、损失加权系数）

5.2 对推理效率的影响

MTP 是推测解码的最优草稿来源： - 独立草稿模型：接受率 60-70% - Medusa 解码头：接受率 70-80% - MTP 模块：接受率 80-90%（因为与主模型共享表示） - 更高的接受率 → 更大的加速比

5.3 对模型设计的影响

MTP 推动了"训练-推理协同设计"的范式： - 模型架构不再只考虑训练效率，还要考虑推理效率 - MTP 模块成为模型架构的一部分 - 未来的模型可能在训练时就集成多种推理优化（量化感知、蒸馏目标、推测模块）

5.4 核心洞见：从"逐步猜测"到"批量预知"

MTP 的深层意义在于改变了模型"看未来"的方式： - NTP：每次只看一步，像走迷宫时每走一步都要停下来看地图 - MTP：一次看多步，像走迷宫时提前规划好几步的路线

这种"批量预知"的能力不仅加速了推理，还可能提高模型的质量——因为模型在训练时就学会了考虑更长远的依赖关系。

六、总结

多Token预测（MTP）代表了大模型训练和推理的范式转变——从"训练时不想推理"到"训练时就为推理做准备"。DeepSeek 的 DSpark 首次在工业级模型上验证了 MTP 的价值：在 DeepSeek-V4-Flash/Pro 上实现 60-85% 的推理加速，在 Qwen3 上也实现了 30% 的加速。

MTP 的核心价值不在于某种具体的算法实现，而在于一个更深层的理念：训练和推理不应该是割裂的两个阶段，而应该是协同设计的统一体。训练时的每一个决策（目标函数、架构设计、损失加权）都应该考虑推理时的影响。

从更宏观的视角看，MTP 是"训练-推理协同设计"大趋势的开端。未来，我们可能会看到更多这样的协同设计：训练时集成量化感知、训练时集成蒸馏目标、训练时集成推测模块。每一次协同设计，都是训练效率和推理效率的双重提升。
