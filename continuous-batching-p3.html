  • 1 核心问题 + 挠痒处

连续批处理在答一个问题：为什么 GPU 同时处理 10 个请求只比处理 1 个慢一点点？更直白地说——GPU 在推理时 99% 的时间都在等显存搬运，能不能把这些等待时间利用起来？

挠痒处：2022-2026 年，大模型推理服务成为 AI 产业的核心基础设施。训练成本是一次性的，但推理成本与用户请求量成正比——用户越多，GPU 越不够用。连续批处理的诱惑在于它承诺了一种"免费的吞吐量提升"：不需要更快的 GPU，不需要更小的模型，只需要更聪明地调度请求。但"免费"二字总是可疑的。动态调度有开销，显存管理有复杂度，不同请求的生成长度不可预测——真正的"免费"在哪里？

答案藏在 GPU 的架构特性里。自回归解码时，每一步只处理一个 token，但需要把整个模型权重（几十 GB）从 HBM 读到计算单元。这个内存搬运操作占据了 99% 的时间，而实际的矩阵乘法只用了不到 1% 的时间。连续批处理的洞见是：既然内存搬运的时间已经付出了（读模型权重），那搬运期间闲置的计算单元不用白不用——让它同时处理多个请求的 token，反正不额外增加内存搬运。

这就是"搬一次也是搬，搬十次也是搬"的真正含义。不是真的增加了搬运量，而是每次搬运时"顺带"多做了计算。前提是：batch 中的 token 数量不能太多——当 token 数量超过某个阈值后，计算量本身开始成为瓶颈，延迟才会显著增加。

从 2019 年 FasterTransformer 的动态批处理雏形，到 2022 年 Orca 的迭代级调度，到 2023 年 vLLM 的 PagedAttention，到 2024 年 FastGen 的 SplitFuse、SGLang 的 RadixAttention，到 2025 年 PD 分离架构——连续批处理走过了一条从学术构想到产业标配的道路。每一代都在解决上一代遗留的问题：Orca 解决了调度问题但遗留了显存管理问题，vLLM 解决了显存管理问题但遗留了 prefill 阻塞问题，FastGen 解决了 prefill 阻塞问题但遗留了前缀匹配问题，SGLang 解决了前缀匹配问题。这条演进链本身就是一部"问题驱动"的技术史。

  • 2 基础假设

连续批处理立在三条不证之物上。

第一条：GPU 推理的瓶颈是显存带宽（memory bandwidth），而非浮点算力（FLOPs）。这是整条连续批处理的生命线。如果推理的瓶颈是算力而非带宽，那么增加 batch size 会线性增加延迟，批处理就没有"免费"的吞吐量提升。在自回归解码阶段，每一步只处理 1 个 token，需要读取整个模型权重（如 14 GB 对于 7B FP16 模型），但实际的矩阵乘法计算量极小（14 GFLOPs）。在 NVIDIA A100 上，读取 14 GB 需要 7 ms，而 14 GFLOPS 的计算只需 0.045 ms——GPU 利用率不到 1%。这个假设在当前 GPU 架构（HBM 带宽有限但计算单元充裕）下成立。但如果未来 GPU 的内存带宽大幅提升（如 CXL 内存池化、近存计算），或者模型架构发生根本变化（如非自回归模型成为主流），连续批处理的前提条件可能不再成立。

第二条：在 decode 阶段，每个请求每步的工作量相同（都是 1 个 token 的前向传播）。这使得在 iteration 边界动态调度请求成为可能——不需要任何额外的同步开销。如果不同请求的工作量差异很大（如某些请求需要处理更长的上下文），那么动态调度的效果会打折扣。但 decode 阶段的特性保证了这一点：无论请求的 prompt 多长，decode 阶段每步只处理 1 个 token。Prefill 阶段的工作量确实差异很大（取决于 prompt 长度），这正是 FastGen 的 Chunked Prefill 要解决的问题。

第三条：KV Cache 可以分页管理而不显著影响性能。这是 vLLM/PagedAttention 的核心假设。如果分页引入了大量的额外开销（如页表查找、非连续内存访问的 cache miss），那么 PagedAttention 的收益会被抵消。实际中，分页的开销很小（<2%），因为：(1) 页表本身很小（每个请求几 KB）；(2) 现代 GPU 的 TLB（Translation Lookaside Buffer）可以缓存页表映射；(3) 通过预取机制可以隐藏非连续访问的延迟。

还有一条隐含的：请求的生成长度服从某种可预测的分布。如果所有请求都极长（如 100K token），那么 KV Cache 的显存需求会远超 GPU 容量，连续批处理也无法在单 GPU 上服务足够多的请求。实际中，大多数在线推理请求的生成长度在几十到几百 token 之间，KV Cache 的显存需求是可管理的。

  • 3 分析框架

连续批处理的取景框是"三代演进"：静态批处理 → 动态批处理 → 连续批处理。

框架里有几件独占的东西：

调度粒度（scheduling granularity）——连续批处理的核心维度。静态批处理的粒度是"batch"（一批请求从开始到结束不可分割），动态批处理的粒度是"请求"（单个请求完成后可被替换），连续批处理的粒度是"iteration"（每次前向传播都可以调整 batch 组成）。调度粒度越细，GPU 利用率越高，但调度开销也越大。最优的调度粒度取决于调度开销与 GPU 闲置成本的平衡。

显存管理策略（memory management strategy）——连续批处理的物理基础。三代方案：连续分配（静态/动态批处理）→ 分页分配（PagedAttention）→ 智能缓存（RadixAttention）。显存管理决定了可同时服务的请求数量上限，从而决定了吞吐量的上限。

KV Cache 大小——连续批处理的物理约束。对于每个请求，KV Cache 的大小 = 2 × n_layers × d_model × seq_len × dtype_size。以 LLaMA-7B (32层, 4096维, FP16) 为例，每个 token 的 KV Cache 大小 = 2 × 32 × 4096 × 2 = 512 KB。1000 个 token 的 KV Cache = 512 MB。100 个并发请求各 1000 token = 51.2 GB——这已经超过了单张 A100 80GB 的显存容量（还需要减去模型权重的 14 GB）。因此 KV Cache 的显存管理是连续批处理的核心瓶颈。

Prefill vs Decode 的异构性——连续批处理的深层矛盾。Prefill 是计算密集型（GPU 利用率 80%+），Decode 是内存带宽密集型（GPU 利用率 <1%）。混合调度时，prefill 会阻塞 decode（长 prefill 导致 decode 延迟毛刺），decode 会"稀释" prefill（短 decode batch 降低了 prefill 的效率）。这个矛盾在 FastGen 的 SplitFuse 中得到了部分解决（Chunked Prefill），在 PD 分离中得到了彻底解决（不同阶段分配不同 GPU）。

  • 4 核心观点

连续批处理是 GPU 内存墙时代的必然产物。只要自回归解码仍然是内存带宽受限的操作，连续批处理就有存在的理由。它的核心价值不在于某种具体的算法实现，而在于一个更深层的洞见：GPU 在推理时大量闲置，连续批处理通过动态调度让多个请求共享 GPU 资源，将"闲置"转化为"产出"。

从 Orca 到 SGLang，连续批处理经历了三次范式转换：

  1. 请求级调度 → 迭代级调度（Orca）：调度粒度从"batch"降低到"iteration"
  2. 连续显存 → 分页显存（vLLM）：显存管理从"连续分配"变为"分页分配"
  3. 混合阶段 → 分离阶段（PD Disaggregation）：架构从"单 GPU 混合"变为"集群级分离"

    4. 每次范式转换都带来了显著的性能提升：从 2-4x 到 4-24x 再到集群级的 8-50x 吞吐量提升。

    连续批处理的"基础设施"特性值得深入思考。在推理优化的众多技术中（量化、蒸馏、剪枝、推测解码、FlashAttention），连续批处理是唯一一个"所有其他优化都建立在它之上"的技术。你可以不用量化、不用推测解码，但你不可能不用连续批处理——任何生产级推理服务都需要它。就像操作系统一样，你可能不理解它的工作原理，但你离不开它。

    PagedAttention 的类比思维值得学习。将 KV Cache 的显存管理类比操作系统的虚拟内存管理，这是一个极其优雅的洞见。操作系统解决的问题是"如何让进程的内存不需要连续物理空间"，PagedAttention 解决的问题是"如何让 KV Cache 不需要连续显存空间"。同样的分页、页表、写时复制——问题不同，但解决方案的结构完全相同。这种"跨领域的结构类比"是技术创新的重要源泉。

    PD 分离的经济学逻辑值得思考。Prefill 需要高算力 GPU（如 H100），Decode 可以用低算力但高带宽的 GPU（如 L40S）。如果 H100 的价格是 L40S 的 3 倍，那么将 decode 从 H100 迁移到 L40S 可以节省 60%+ 的推理成本。PD 分离不仅是技术优化，更是成本优化。

    从产业角度看，连续批处理正在成为推理引擎的"操作系统"。vLLM、TensorRT-LLM、SGLang 就是推理时代的 Linux/Windows/macOS——它们提供了资源管理、调度、安全隔离等基础设施功能。用户只需关注"模型"和"业务逻辑"，不需要关心"GPU 资源如何分配"。这种抽象层次的提升是技术成熟的标志。

    • 5 精神内核

    一个洞见——"连续批处理的核心不是'怎么调度得更聪明'，而是'为什么可以调度得这么自由'。"

    从 2019 年到 2026 年，连续批处理的技术路线在不断演化——从静态到动态到连续到分页到分离。但底层逻辑从未改变：GPU 在自回归解码时大量闲置（99% 的时间在等显存搬运），连续批处理利用这些闲置资源来同时服务多个请求。每次搬运模型权重的成本是固定的，"搬一个 token 也是搬，搬十个 token 也是搬"——只要计算量不超过内存搬运的等待时间，增加 batch size 就是"免费"的。

    这是一个资源利用的故事：在一个存在大量闲置资源的系统中，通过更聪明的调度来利用这些闲置资源，可以在不增加成本的情况下大幅提升产出。关键是理解"瓶颈在哪里"——瓶颈在显存带宽而非算力，所以增加计算（batch size）不增加瓶颈成本。

    带走了这一件，连续批处理的精神内核就在手：在资源闲置的系统中，调度的价值远大于堆砌资源——不是买更快的 GPU，而是让已有的 GPU 不再闲置。

    这个洞见超越了推理加速。在任何"固定成本 + 闲置资源"的系统中（云计算、物流、制造业、人力资源），"连续批处理"思维都适用：不要等凑满一车再发货（静态批处理），而是让每辆车都尽量装满（连续批处理）。搬一次也是搬，搬十次也是搬——关键是每次搬运都要物尽其用。
