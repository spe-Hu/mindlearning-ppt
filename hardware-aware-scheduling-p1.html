硬件感知调度与在线校准 — 横纵深度分析

**一句话定位**：DSpark 通过置信度头预估每个草稿位置的验证存活概率，结合预先测算的 GPU 吞吐量参考曲线，动态匹配最优验证长度；同时用顺序温度缩放做在线校准，将预期校准误差从 3%-8% 压到约 1%，整套调度逻辑完全在 GPU 内部执行，无需 CPU 参与。

一、纵轴：从固定长度到硬件感知自适应的演化叙事

1.1 起点：固定长度草稿

推测解码的最初设计采用固定长度的草稿。每次推测-验证循环中，草稿模型生成固定数量 $K$ 个候选 token，目标模型一次性验证。

$$

K_{\text{fixed}} = \text{const}

$$

这种设计的优点是简单，但存在严重的效率问题：

  1. **过度推测**：当草稿模型信心很低时（如生成罕见词汇），大部分草稿 token 会被拒绝，浪费验证计算
  2. **推测不足**：当草稿模型信心很高时（如生成常见短语），可以接受更多 token，但固定长度限制了收益
  3. **硬件不感知**：不考虑 GPU 当前的负载状态和吞吐特性

    4. 数学建模：设 GPU 在批次大小 $B$ 下的吞吐量为 $\Theta(B)$，验证 $K$ 个 token 的时间为：

    $$

    T_{\text{verify}}(K) = \frac{K \cdot d_{\text{model}} \cdot |V|}{\Theta(B)}

    $$

    当 $K$ 过大时，$T_{\text{verify}}$ 增加但接受率不增反降；当 $K$ 过小时，$T_{\text{verify}}$ 减少但每轮前进的 token 数也减少。存在一个最优的 $K^*$：

    $$

    K^* = \arg\max_K \frac{\mathbb{E}[\text{accept}(K)] + 1}{T_{\text{draft}}(K) + T_{\text{verify}}(K)}

    $$

    1.2 第一次跃迁：自适应长度草稿

    早期自适应方案（如 EAGLE）根据草稿模型的置信度动态调整草稿长度：

    $$

    K_{\text{adaptive}} = f(\text{confidence})

    $$

    当草稿模型的 top-1 概率高于阈值时继续生成，否则停止。这种方案比固定长度好，但存在两个问题：

    1. **只看草稿模型**：不考虑目标模型的验证行为
    2. **不看硬件状态**：不考虑 GPU 的实时吞吐特性

      3. 1.3 第二次跃迁：硬件感知调度

    DSpark 的核心创新之一是引入硬件感知调度。其核心思想是：

    1. **预先测算** GPU 在不同批次大小下的吞吐数据
    2. **生成吞吐量参考曲线** $\Theta(B)$
    3. **动态匹配**当前 GPU 状态下的最优验证长度

      4. 具体来说，DSpark 在系统启动时执行一个校准阶段：

      
def calibrate_throughput(model, device):
    """
    预先测算 GPU 在不同批次大小下的吞吐量
    
    输出：吞吐量参考曲线 Theta(B)
    """
    batch_sizes = [1, 2, 4, 8, 16, 32, 64, 128]
    throughputs = []
    
    for B in batch_sizes:
        # 测量不同草稿长度下的验证吞吐量
        for K in [4, 8, 12, 16, 24, 32]:
            # 构造批次
            input_ids = torch.randint(0, vocab_size, (B, K)).to(device)
            
            # 预热
            for _ in range(10):
                _ = model(input_ids)
            torch.cuda.synchronize()
            
            # 测量
            start = time.time()
            for _ in range(100):
                _ = model(input_ids)
            torch.cuda.synchronize()
            elapsed = (time.time() - start) / 100
            
            throughput = B * K / elapsed  # tokens/s
            throughputs.append((B, K, throughput))
    
    # 拟合吞吐量曲线
    throughput_curve = fit_curve(throughputs)
    return throughput_curve

      1.4 终态：置信度头 + 吞吐量曲线 → 动态最优

      DSpark 的最终方案是置信度头与吞吐量参考曲线的结合：

      1. **置信度头**（Confidence Head）：给每个草稿位置打分，预估验证存活概率
      2. **吞吐量曲线**：提供当前 GPU 状态下的验证成本
      3. **最优化求解**：在 GPU 内部实时求解最优验证长度

        4. $$

        K^* = \arg\max_K \frac{\sum_{i=1}^{K} P_{\text{survive}}(i) + 1}{T_{\text{verify}}(K, B) / \Theta(B)}

        $$

        其中 $P_{\text{survive}}(i)$ 是置信度头预估的第 $i$ 个位置的验证存活概率。

        二、核心创新 1：置信度头与动态调度

        2.1 置信度头的设计

        置信度头是一个轻量级的分类器，输入草稿模型在位置 $i$ 的隐藏状态，输出该位置的验证存活概率：

        $$

        P_{\text{survive}}(i) = \sigma(w_c^T \cdot h_i^{\text{draft}} + b_c)

        $$

        其中 $w_c \in \mathbb{R}^{d_{\text{hidden}}}$，$b_c \in \mathbb{R}$ 是可训练参数，$\sigma$ 是 sigmoid 函数。

        训练目标：最小化预估存活概率与实际存活情况的二元交叉熵：

        $$

        \mathcal{L}_{\text{conf}} = -\sum_{i=1}^{K} \left[ y_i \log \hat{y}_i + (1 - y_i) \log(1 - \hat{y}_i) \right]

        $$

        其中 $y_i \in \{0, 1\}$ 是实际验证结果（接受=1，拒绝=0），$\hat{y}_i = P_{\text{survive}}(i)$ 是预估概率。

        2.2 动态调度算法

        
def dynamic_schedule(confidence_head, throughput_curve, draft_hidden, 
                     current_batch_size, gpu_state):
    """
    动态调度算法
    
    输入：
    - confidence_head: 置信度头
    - throughput_curve: 吞吐量参考曲线
    - draft_hidden: 草稿模型的隐藏状态 [K_max, d_hidden]
    - current_batch_size: 当前批次大小
    - gpu_state: GPU 当前状态
    
    输出：
    - K_opt: 最优验证长度
    """
    K_max = draft_hidden.shape[0]
    
    # 1. 预估每个位置的存活概率
    survival_probs = confidence_head(draft_hidden)  # [K_max]
    
    # 2. 计算不同验证长度的期望收益
    expected_gains = []
    for K in range(1, K_max + 1):
        # 期望接受长度
        expected_accept = sum(survival_probs[:K])
        
        # 验证成本（从吞吐量曲线查询）
        verify_cost = throughput_curve.query(current_batch_size, K)
        
        # 净收益 = 期望接受长度 / 验证成本
        net_gain = (expected_accept + 1) / verify_cost
        expected_gains.append(net_gain)
    
    # 3. 选择最优验证长度
    K_opt = np.argmax(expected_gains) + 1
    
    return K_opt

        三、核心创新 2：在线校准与顺序温度缩放

        3.1 问题：校准误差

        推测解码的拒绝采样要求草稿模型和目标模型的概率分布尽可能一致。但在实际部署中，由于量化、近似等原因，两个模型的分布之间存在校准误差（calibration error）。

        预期校准误差（Expected Calibration Error, ECE）定义为：

        $$

        \text{ECE} = \sum_{m=1}^{M} \frac{n_m}{N} |o_m - e_m|

        $$

        其中 $o_m$ 是第 $m$ 个置信度区间的平均预估置信度，$e_m$ 是实际准确率，$n_m$ 是该区间的样本数。

        3.2 标准温度缩放 vs 顺序温度缩放

        标准温度缩放（Temperature Scaling）：

        $$

        p_{\text{calibrated}} = \text{softmax}(z / T)

        $$

        其中 $T$ 是在验证集上优化的单一温度参数。

        顺序温度缩放（Sequential Temperature Scaling）：

        DSpark 的创新在于为草稿序列中的每个位置使用不同的温度参数：

        $$

        p_{\text{calibrated}}^{(i)} = \text{softmax}(z^{(i)} / T_i)

        $$

        其中 $T_i$ 是第 $i$ 个位置的温度参数。

        为什么需要顺序温度缩放？

        1. 草稿序列中不同位置的分布偏差模式不同
        2. 位置越远，草稿模型和目标模型的分布差异越大
        3. 单一温度无法同时校准所有位置

          4. 数学分析：设位置 $i$ 的 logits 偏差为 $\Delta z^{(i)}$，则最优温度为：

          $$

          T_i^* = 1 + \frac{\mathbb{E}[\|\Delta z^{(i)}\|]}{\mathbb{E}[\|z^{(i)}\|]}

          $$

          3.3 在线校准流程

          
class OnlineCalibrator:
    """
    在线校准器
    
    使用顺序温度缩放，实时校准草稿模型的输出分布
    """
    
    def __init__(self, max_draft_len=32, initial_temp=1.0):
        self.max_draft_len = max_draft_len
        # 每个位置一个温度参数
        self.temperatures = nn.Parameter(
            torch.ones(max_draft_len) * initial_temp
        )
        # 在线统计
        self.running_stats = {
            "ece": 0.0,
            "num_samples": 0,
            "update_count": 0
        }
    
    def calibrate(self, draft_logits, position):
        """
        对指定位置的 logits 进行校准
        
        Args:
            draft_logits: [batch, vocab_size] 草稿模型 logits
            position: int 草稿位置索引
        
        Returns:
            calibrated_probs: [batch, vocab_size] 校准后的概率
        """
        T = self.temperatures[position]
        calibrated = torch.softmax(draft_logits / T, dim=-1)
        return calibrated
    
    def update_temperature(self, position, draft_logits, target_probs):
        """
        在线更新指定位置的温度参数
        
        使用 NLL 损失优化：
        L = -sum(target_probs * log(softmax(draft_logits / T)))
        """
        T = self.temperatures[position]
        calibrated_log_probs = torch.log_softmax(draft_logits / T, dim=-1)
        loss = -(target_probs * calibrated_log_probs).sum(dim=-1).mean()
        
        # 梯度更新
        loss.backward()
        with torch.no_grad():
            self.temperatures[position] -= 0.01 * self.temperatures[position].grad
            self.temperatures[position].grad = None
            # 温度约束在 [0.5, 3.0]
            self.temperatures[position].clamp_(0.5, 3.0)
    
    def compute_ece(self, calibrated_probs, predictions, targets):
        """计算预期校准误差"""
        confidences, predicted = torch.max(calibrated_probs, dim=-1)
        accuracies = (predicted == targets).float()
        
        n_bins = 15
        bin_boundaries = torch.linspace(0, 1, n_bins + 1)
        
        ece = 0.0
        for i in range(n_bins):
            in_bin = (confidences > bin_boundaries[i]) & \
                     (confidences <= bin_boundaries[i + 1])
            prop_in_bin = in_bin.float().mean()
            
            if prop_in_bin > 0:
                avg_confidence = confidences[in_bin].mean()
                avg_accuracy = accuracies[in_bin].mean()
                ece += torch.abs(avg_confidence - avg_accuracy) * prop_in_bin
        
        return ece.item()

          四、GPU 内部执行：零 CPU 开销

          4.1 设计目标

          DSpark 的整套调度逻辑完全在 GPU 内部执行，不需要 CPU 参与。这避免了 CPU-GPU 之间的数据传输开销。

          六、关键洞察

          6.1 三个核心洞察

          1. **硬件感知是必须的**：不同 GPU 的最优草稿长度不同，同一 GPU 在不同负载下的最优长度也不同。忽略硬件特性的调度方案无法达到最优。
          2. **在线校准是持续的**：分布偏差不是静态的，会随着输入分布、量化精度等因素变化。在线校准使得系统能够持续适应这些变化。
          3. **GPU 内部执行是关键**：CPU-GPU 传输是推测解码中被忽视的瓶颈。将调度逻辑完全放在 GPU 内部，消除了这一瓶颈。

            4. 6.2 工程审美

          硬件感知调度体现了一种"越跑越准"的工程审美：

          • 启动时校准，建立基线
          • 运行时在线学习，持续优化
          • 不需要完美的初始配置，只需要正确的学习方向

            • 七、总结

        硬件感知调度与在线校准是 DSpark 的两大核心创新。置信度头 + 吞吐量参考曲线实现了动态最优的验证长度匹配；顺序温度缩放将校准误差从 3%-8% 压到约 1%；GPU 内部执行消除了 CPU-GPU 传输瓶颈。整套方案使得 DSpark 在各种硬件和负载条件下都能保持最优的推理性能。
