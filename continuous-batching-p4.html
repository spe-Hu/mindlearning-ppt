连续批处理 (Continuous Batching)

一句话定义：在 GPU 执行每一次前向传播时，动态地将多个请求的不同位置 token 组合成一个 batch，让新请求随时插入、已完成的请求立即释放，实现 GPU 计算单元和显存的全时满载利用。

核心机制：

  • 调度粒度从"batch 级"降低到"iteration 级"
  • 每次前向传播都可以动态调整 batch 组成
  • 已完成请求立即释放资源，新请求立即插入
  • KV Cache 分页管理，按需分配，消除显存碎片

核心洞见：

  • GPU 推理的瓶颈是显存带宽，不是算力
  • 每次前向传播都需要读取整个模型权重（固定成本）
  • 增加 batch 中的 token 数量几乎不增加延迟（在阈值内）
  • "搬一次也是搬，搬十次也是搬"

数学表达：

#+BEGIN_EXAMPLE

GPU 利用率 = T_compute / (T_compute + T_memory)

T_compute = 2 × P × B / FLOPS_peak

T_memory = M_model / BW_memory

当 B 较小时：T_compute << T_memory → GPU 利用率 ≈ 0%

当 B 增大时：T_compute → T_memory → GPU 利用率 → 50%

当 B 很大时：T_compute >> T_memory → GPU 利用率 → 100%

#+END_EXAMPLE

关键数字：

  • 7B 模型在 A100 上，batch_size=1 时 GPU 利用率 < 1%
  • 7B 模型在 A100 上，batch_size=10 时 GPU 利用率 ≈ 6%
  • 7B 模型在 A100 上，batch_size=156 时 GPU 利用率 ≈ 50%
  • 7B 模型在 A100 上，batch_size=312 时 GPU 利用率 ≈ 67%

  • H100 的算力是 A100 的 3.2x，但利用率反而更低（因为搬运更快了，但计算也更快了，搬运仍然是瓶颈）
  • 增加 batch size 是提升利用率的唯一途径

静态 → 动态 → 连续批处理

#+BEGIN_EXAMPLE

=== 静态批处理 (Static Batching) ===

时间线：

Batch 1:

请求1: [prefill][decode]...[decode][EOS] ← 20步

请求2: [prefill][decode]...[decode]......[decode][EOS] ← 200步

GPU: |← 必须等所有请求完成 →|

← 请求1完成后闲置180步 →

特点：

✓ 实现简单

✗ 木桶效应（最慢请求决定 batch 时间）

✗ 新请求必须等待当前 batch 完成

✗ GPU 利用率: 10-30%

=== 动态批处理 (Dynamic Batching) ===

时间线：

请求1: [prefill][decode][decode][EOS] → 完成，替换为请求4

请求2: [prefill][decode]...[decode][EOS] → 完成，替换为请求5

请求3: [prefill][decode]...[decode]......[decode]...

请求4: [prefill][decode][EOS] → 完成

请求5: [prefill]...

特点：

✓ 请求完成后可立即替换

✓ 新请求不必等待整个 batch 完成

✗ 仍需 padding 对齐

✗ KV Cache 连续分配，碎片化严重

✗ GPU 利用率: 40-60%

=== 连续批处理 (Continuous Batching) ===

时间线：

Iteration 1: [请求1-token1, 请求2-token1, 请求3-token1]

Iteration 2: [请求1-token2, 请求2-token2, 请求3-token2, 请求4-prefill-chunk1]

Iteration 3: [请求1-EOS→移除, 请求2-token3, 请求3-token3, 请求4-prefill-chunk2, 请求5-新插入]

Iteration 4: [请求2-token4, 请求3-token4, 请求4-decode1, 请求5-prefill-chunk1]

特点：

✓ 每次 iteration 动态调整 batch

✓ 新请求随时插入，完成请求立即移除

✓ KV Cache 分页管理，零碎片

✓ Chunked Prefill 避免阻塞

✓ GPU 利用率: 70-95%

#+END_EXAMPLE

性能对比表：

指标 静态批处理 动态批处理 连续批处理
吞吐量 1x (baseline) 2-4x 4-24x

P99 延迟 极高 高 低

PagedAttention：KV Cache 的虚拟内存

核心类比：

#+BEGIN_EXAMPLE

操作系统虚拟内存 PagedAttention

───────────────── ──────────────

虚拟地址空间 逻辑 KV Cache

物理内存页框 物理显存块

页表 (Page Table) 块表 (Block Table)

4KB 页 16 token 块

按需分页 按需分配块

Copy-on-Write 前缀共享 (CoW)

LRU 页面替换 LRU 块替换

TLB 缓存映射 Block Table 缓存

内存碎片问题：

进程A: [████████░░░░░░] → 内部碎片 30%

进程B: [████████████████░░░░] → 内部碎片 20%

间隙: [░░░░] → 外部碎片

PagedAttention：

请求1: [块3][块7][块12] → 内部碎片 <5% (仅最后一块)

请求2: [块1][块5][块9][块14] → 无外部碎片 (块大小相同)

#+END_EXAMPLE

KV Cache 大小计算：

#+BEGIN_EXAMPLE

每个 token 的 KV Cache = 2 × n_layers × d_head × n_heads × dtype_size

LLaMA-7B (32层, 4096维, 32头, FP16):

= 2 × 32 × 128 × 32 × 2 bytes

= 524,288 bytes = 512 KB / token

LLaMA-70B (80层, 8192维, 64头, FP16):

= 2 × 80 × 128 × 64 × 2 bytes

= 2,621,440 bytes = 2.5 MB / token

显存占用示例 (LLaMA-7B):

模型权重: 14 GB

100 请求 × 1000 token × 512 KB = 51.2 GB

总计: 65.2 GB → 需要 A100 80GB

使用 PagedAttention 后:

显存浪费 < 5% → 可支持 ~140 个并发请求 (vs 连续分配的 ~80 个)

#+END_EXAMPLE

前缀共享 (Copy-on-Write)：

#+BEGIN_EXAMPLE

场景：多个请求共享 system prompt

System Prompt: "You are a helpful assistant..." (100 tokens)

请求1: [System Prompt][User: Hello][Response...]

请求2: [System Prompt][User: How are you][Response...]

请求3: [System Prompt][User: Tell me about AI][Response...]

传统方案：

请求1 的 KV Cache: [System Prompt KV = 51.2 MB] [User KV] [Response KV]

请求2 的 KV Cache: [System Prompt KV = 51.2 MB] [User KV] [Response KV]

请求3 的 KV Cache: [System Prompt KV = 51.2 MB] [User KV] [Response KV]

总 System Prompt KV: 153.6 MB (重复 3 次)

PagedAttention：

物理块: [块1-3: System Prompt KV] [块4: 请求1 User] [块5: 请求2 User] [块6: 请求3 User] ...

请求1 块表: [块1, 块2, 块3, 块4, ...]

请求2 块表: [块1, 块2, 块3, 块5, ...] ← 共享块1-3

请求3 块表: [块1, 块2, 块3, 块6, ...] ← 共享块1-3

总 System Prompt KV: 51.2 MB (只存 1 次)

节省: 102.4 MB (67%)

#+END_EXAMPLE

连续批处理 + 推测解码 = 双重加速

#+BEGIN_EXAMPLE

两种优化的互补关系：

连续批处理：

优化维度：请求间并行

核心思想：多个请求共享同一次模型权重读取

效果：吞吐量提升 4-24x

作用层面：请求级调度

推测解码：

优化维度：请求内并行

核心思想：用草稿模型猜测多个 token，目标模型一次验证

效果：单请求延迟降低 2-5x

作用层面：算法级优化

协同效果：

┌─────────────────────────────────────────┐

│ GPU 一次前向传播 │

│ │

│ 请求1: [draft: t1,t2,t3] [verify: ok] │ ← 推测解码

│ 请求2: [draft: t1,t2] [verify: ok] │ ← 推测解码

│ 请求3: [draft: t1,t2,t3,t4] [verify] │ ← 推测解码

│ 请求4: [decode: t1] │ ← 普通解码

│ ... │

│ 请求N: [decode: t1] │ ← 普通解码

│ │

│ ← 连续批处理：多个请求共享模型权重读取 → │

└─────────────────────────────────────────┘

叠加效果：

单独连续批处理: 吞吐量 4-24x

单独推测解码: 单请求速度 2-5x

两者叠加: 有效吞吐量 8-100x+ (理论值，实际受接受率和 batch size 限制)

#+END_EXAMPLE

协同约束：

  • 推测解码增加计算量（草稿模型 + 验证），可能使 batch 更快进入 compute-bound 区域
  • 连续批处理的 batch size 需要为推测解码"留出"计算预算
  • 最优配置：batch_size × draft_length ≈ B_optimal

    连续批处理核心公式速查

    #+BEGIN_EXAMPLE

    1. GPU 利用率公式：

      2. η = T_compute / (T_compute + T_memory)

      = (2PB/F) / (2PB/F + M/BW)

      P=参数量, B=batch_size, F=FLOPS_peak, M=模型大小, BW=带宽

      1. 最优 batch size (η=50%)：

        2. B_opt = M × F / (2P × BW)

        对于 7B + A100: B_opt ≈ 156

        对于 70B + A100: B_opt ≈ 156 (与参数量无关！)

        对于 7B + H100: B_opt ≈ 249

        1. KV Cache 大小：

          2. KV_size = 2 × L × d_h × n_h × dtype × seq_len

          L=层数, d_h=头维度, n_h=头数

          4. 最大并发请求数：

          N_max = (GPU_memory - M_model) / (KV_per_token × avg_seq_len)

          7B + A100 80GB: N_max ≈ (80-14) / (0.5MB × 1000) ≈ 132

          5. 吞吐量：

          Throughput = N_active × tokens_per_step / step_time

          step_time ≈ T_memory (当 B < B_opt 时)

          step_time ≈ T_compute (当 B > B_opt 时)

          6. 延迟 vs 吞吐量权衡：

          延迟敏感: B = 1-10 (低延迟, 低吞吐量)

          吞吐量优先: B = B_opt (高吞吐量, 中等延迟)

          过犹不及: B >> B_opt (高延迟, GPU 计算瓶颈)

          #+END_EXAMPLE
