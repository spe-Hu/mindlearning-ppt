+filetags: :concept:ai:inference:

+date: [2026-06-28]

  • 定锚

推测解码（Speculative Decoding）：大语言模型推理加速的核心技术。用轻量级草稿模型快速生成多个候选 token，目标大模型一次性并行验证，接受正确的、拒绝错误的，在数学上保证输出分布完全一致的前提下实现 2-5 倍加速。本质是"用廉价的猜测换昂贵的验证"——利用 GPU 自回归解码时 99% 闲置的算力。

常见误解： - 误解一："推测解码是有损的"——错。推测解码通过接受-拒绝采样在数学上保证输出分布与原始模型完全一致。不是近似，是精确等价。 - 误解二："推测解码就是用小模型替代大模型"——错。小模型只是"草稿"，最终决策权在大模型。小模型的角色类似于秘书起草文件，老板审批签字。 - 误解三："推测解码总是能加速"——错。当接受率太低（如 <30%）或 batch size 太大时，推测解码可能反而更慢。 - 误解四："推测解码需要额外 GPU"——不一定。Medusa 和 EAGLE 在目标模型自身上添加解码头，无需额外 GPU 内存来加载独立草稿模型。 - 误解五："推测解码只适用于文本生成"——错。2025 年已有工作将推测解码扩展到多模态模型（视觉-语言模型、语音模型）。

核心词素： - Speculative（推测的）：源自拉丁语 =speculari=（观察、推测），暗含"投机"的意味——先猜再说 - Decoding（解码）：从模型的概率分布中生成 token 序列的过程 - 隐含词素：Draft Model（草稿模型）——负责快速猜测的小模型 - 隐含词素：Verification（验证）——目标模型对草稿的检验 - 隐含词素：Acceptance Rate（接受率）——推测解码的核心指标 - 隐含词素：Memory-Bound（内存受限）——推测解码存在的根本原因

  • 八刀

** 历史

2018 年，Google Brain 的 Stern 等人发表《Blockwise Parallel Decoding for Deep Autoregressive Models》，首次提出用辅助模型加速自回归解码的思想。他们训练一个浅层的"jam model"来并行预测多个 token 的候选块，然后让目标模型一次性评分。这埋下了推测解码的种子。但 2018 年的模型还不够大，推理延迟不是痛点，这篇论文没有引起广泛关注。

2022 年底，两个独立团队几乎同时重新发现了这一思想。Google 的 Leviathan 等人正式命名为"Speculative Decoding"，给出了接受-拒绝采样的无损保证和加速比公式。DeepMind 的 Chen 等人从采样角度提出了等价的"Speculative Sampling"框架。两篇论文的同时出现说明推测解码不是某个人的灵光一现，而是技术成熟到一定程度的必然产物——当大模型推理延迟成为产业痛点时，"用辅助模型加速"的想法自然浮现。

2024 年是推测解码的爆发年。Medusa（Cai et al.）消除了独立草稿模型，用目标模型自身的解码头做草稿。EAGLE（Han et al.）将推测从 token 空间转移到特征空间，显著提高了接受率。SpecInfer 引入多草稿模型池和树状推测。REST 提出非参数化的检索式草稿。

2025 年，DeepSeek 的 DSpark 将推测解码推向工业级部署。MTP 模块在预训练阶段就集成到模型中，实现了训练-推理的协同设计。DSpark 在 DeepSeek-V4-Flash/Pro 上实现了 60-85% 的速度提升，在 Qwen3 上也实现了 30% 的加速。

更深层的历史脉络：推测解码的思想可以追溯到计算机科学中的"投机执行"（speculative execution）——CPU 的分支预测就是一种推测执行。CPU 在条件判断完成之前就猜测分支方向并提前执行，猜错了就回滚。推测解码是这一思想在 AI 推理领域的再现：草稿模型猜测 token 序列，目标模型验证，猜错了就拒绝。

** 辩证

正面：推测解码——用闲置资源加速推理，免费午餐。 反面：推测解码有代价——草稿模型占内存，验证有开销，接受率不总是高。 碰撞后的更高理解：推测解码的真正价值不在于"免费"，而在于"不对称"——输了不亏（拒绝的开销几乎为零），赢了大赚（一次验证通过多个 token）。这种不对称性使得即使接受率不是 100%，推测解码仍然是理性的策略。

更深层的辩证：推测解码揭示了 AI 推理的一个根本矛盾——算力和内存带宽的不匹配。GPU 有巨大的计算能力，但内存带宽限制了这些能力的利用。推测解码巧妙地利用了这一矛盾：既然内存带宽是瓶颈，那就在等待内存搬运的同时用闲置的算力做有用的事。但这也意味着，如果未来 GPU 的内存带宽大幅提升（如 CXL 内存池化），推测解码的前提条件可能不再成立。推测解码是"内存墙时代"的产物，它的命运与内存墙紧密相连。

再一层辩证：推测解码的"无损"特性看似完美，但它暗含了一个假设——输出分布不变就等于质量不变。在某些场景下（如代码生成、数学推理），用户关心的不是分布，而是"最终答案的正确性"。推测解码保证了分布不变，但单次运行的结果可能因为推测的随机性而有所不同。这是一个微妙但重要的区别。

** 现象

想象一个餐厅厨房。主厨（目标模型）做菜很慢，因为每道菜他都要亲自从头到尾做一遍。但主厨大部分时间其实是在等——等水烧开、等烤箱预热、等食材解冻。这些等待时间就是"内存搬运"。

推测解码相当于给主厨配了一个助手（草稿模型）。助手先快速准备好食材的初步处理（切菜、腌制），然后主厨只需要快速检查确认就行。助手的准备工作大部分时候是对的（接受率高），主厨的确认时间几乎为零（验证成本极低）。结果是主厨的有效产出翻倍——不是因为他做得更快了，而是因为他不用从零开始了。

但如果助手经常做错（接受率低），主厨花时间检查反而更慢。所以关键不是"有没有助手"，而是"助手靠不靠谱"。Medusa 的做法相当于让主厨自己提前准备——因为最了解主厨口味的就是主厨自己。EAGLE 的做法相当于让助手在"味道空间"里猜测（这道菜大概是什么口味），而不是在"食材空间"里猜测（具体用哪片菜叶）——前者更模糊但也更不容易错。

** 语言

  • Speculative：源自拉丁语 =speculari=（观察、推测），与 speculation（投机）同源。暗含的隐喻是"金融市场中的投机"——先下注，再看结果。在金融中，投机有风险；在推测解码中，"风险"被接受-拒绝采样消除了。
  • Decoding：源自拉丁语 =decodare=（解码），在 AI 语境中指从模型的概率分布中生成 token 序列。
  • 语义网：推测解码与以下概念相邻：
  • Speculative Execution（投机执行）：CPU 分支预测的同源思想
  • Rejection Sampling（拒绝采样）：推测解码的数学基础
  • Knowledge Distillation（知识蒸馏）：草稿模型的一种构建方式
  • Memory Wall（内存墙）：推测解码存在的根本原因
  • Auto-Regressive Decoding（自回归解码）：推测解码要加速的对象
  • 暗含隐喻："猜测-验证"。这个隐喻挡住了一个问题：推测解码不仅仅是"猜测"，它是一种有数学保证的"受控投机"。也挡住了另一个问题：为什么 GPU 推理是内存受限的？
  • 相关术语：
  • Acceptance Rate（接受率）：推测解码的核心指标
  • Draft Model（草稿模型）：负责快速猜测的模型
  • Draft Length（草稿长度）：每次推测生成的候选 token 数
  • Tree-based Verification（树状验证）：同时验证多条候选路径
  • Feature Space（特征空间）：EAGLE 推测的领域

** 形式

核心公式：

接受概率：=P(accept x_i) = min(1, p_target(x_i) / p_draft(x_i))=

加速比：=speedup = 1 / (1 - alpha)=，其中 =alpha= 是平均接受率

最优草稿长度：=K* = -1 / ln(alpha)=

每轮推测的期望产出：=E[output] = sum_{i=1}^{K} alpha^i = alpha(1 - alpha^K) / (1 - alpha)=

失效边界： - 当 batch size 较大时，计算量成为瓶颈，验证 K 个 token 的成本不再约等于验证 1 个 token 的成本。此时加速比下降。 - 当接受率太低（alpha < 0.3）时，推测解码的开销（草稿生成 + 验证）可能超过收益。此时应关闭推测解码。 - 在 prefill 阶段（处理长 prompt），输入很长但输出还没有开始，推测解码不适用。推测解码只在 decode 阶段（逐 token 生成）有效。

进一步的形式化分析：

推测解码的加速比可以从信息论角度理解。自回归解码每步产出 1 个 token，信息量为 H(X)（token 的熵）。推测解码每步期望产出 E[output] 个 token，信息量为 E[output] * H(X)。加速比 = E[output] / 1 = E[output]。

但这是理想情况。实际中，草稿模型的生成也有成本（虽然远小于目标模型）。设草稿模型生成 1 个 token 的成本为目标模型的 c 倍（c << 1，因为草稿模型小得多），则净加速比为：

=net_speedup = E[output] / (1 + c * K)=

当 c → 0（草稿模型成本忽略不计）时，net_speedup → E[output] = alpha(1-alpha^K)/(1-alpha)。 当 c 不可忽略时，需要优化 K 使 net_speedup 最大化。

从博弈论角度看，推测解码是草稿模型和目标模型之间的"合作博弈"。草稿模型的目标是最大化接受率（减少目标模型的拒绝），目标模型的目标是保证输出质量（不接受错误的 token）。接受-拒绝采样机制确保了两个目标的兼容：草稿模型越"像"目标模型，接受率越高，双方合作越顺畅。

** 存在

推测解码改变了 AI 推理的经济学。在它出现之前，推理成本几乎与请求量成正比——用户越多，GPU 越不够用。推测解码使得相同的 GPU 资源可以服务更多请求，直接降低了 AI 服务的边际成本。对于 API 服务商（如 OpenAI、Anthropic、DeepSeek），推测解码意味着 30-50% 的成本下降——这不是小数目，它决定了 AI 服务能否盈利。

推测解码也改变了 AI 工程师的活法。在它出现之前，推理优化主要靠量化、蒸馏、剪枝——都是有损的，需要在速度和质量之间权衡。推测解码提供了第一个无损选项——你可以无脑开启它，不需要担心"加速后模型变笨了"。这简化了推理优化的决策流程。

从更宏观的视角看，推测解码揭示了 AI 产业的一个结构性问题：GPU 的算力和内存带宽严重不匹配。自回归解码时，GPU 99% 的时间在等待内存搬运，只有 1% 的时间在做有用的计算。推测解码利用了这 99% 的闲置时间，但也暴露了一个更深的问题：我们花了大量金钱购买的 GPU 算力，绝大部分被浪费了。这推动了硬件设计的调整——HBM 的容量和带宽成为比 TFLOPS 更重要的指标。

** 美感

美在不对称。推测解码是一个"输了不亏、赢了大赚"的赌局。拒绝的开销几乎为零（一次前向传播已经在做了），接受的收益很大（一次产出多个 token）。这种不对称性使得即使草稿模型不完美，推测解码仍然是理性的策略。意象：一个精明的商人，每笔交易都先低价买入（草稿），再高价验证（目标模型）。即使有些买入被验证为不合格（拒绝），合格的交易带来的利润足以覆盖所有成本。

美在必然。推测解码不是某个天才的灵光一现，而是技术成熟到一定程度的必然产物。当大模型推理延迟成为产业痛点时，"用辅助模型加速"的想法自然浮现——两个独立团队几乎同时提出就是证据。就像微积分被牛顿和莱布尼茨同时发明一样，推测解码被 Leviathan 和 Chen 同时发现。好的想法不是发明出来的，是在条件成熟时"长出来"的。

美在简洁。推测解码的核心只需要一个公式：accept = min(1, p_target/p_draft)。这个简单的公式保证了输出分布的精确等价——不是近似，是数学上的完美。从这一个公式出发，可以推导出加速比、最优草稿长度、接受率上界等所有重要结论。简洁是深层真理的标志。

  • 缝合

推测解码与 Scaling Law 的关系：Scaling Law 推动模型越来越大，大模型推理越来越慢，推测解码的需求越来越迫切。Scaling Law 是"因"，推测解码是"果"——没有 Scaling Law 驱动的大模型时代，推测解码不会成为产业焦点。

推测解码与 MoE 的关系：MoE 通过稀疏激活减少每次推理的计算量，推测解码通过减少前向传播次数来加速。两者互补——MoE 让每次前向传播更快，推测解码让前向传播的次数更少。DeepSeek-V3 同时使用了 MoE 和 MTP/推测解码，两者协同实现了极致的推理效率。

推测解码与 Flash Attention 的关系：Flash Attention 优化单次前向传播中注意力计算的内存访问模式，推测解码优化前向传播的总次数。两者解决不同层面的问题，完全可以组合使用。

推测解码与自动驾驶的关系：自动驾驶的端到端模型要求 <50ms 的推理延迟，车端 GPU 资源有限。推测解码在 batch size=1 时加速效果最好，恰好匹配车端场景。MTP 训练使车端模型天生具备推测能力，是实现实时推理的关键技术之一。
