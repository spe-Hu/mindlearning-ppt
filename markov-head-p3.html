马尔可夫头 — 概念解剖（ljg-book 风格）

**核心问题**：并行猜完了，怎么用最 cheap 的方式修正？

**精神内核**："只看前一步，修正后一步"

一、这本书（这个概念）在答什么问题？

1.1 问题的起源

大语言模型的推理速度一直是部署的核心瓶颈。自回归解码（autoregressive decoding）的本质限制是：每一步只能生成一个 token，即使 GPU 有巨大的并行计算能力，也只能串行等待。

2023 年，推测解码（Speculative Decoding）横空出世，提出了一种"先猜后验"的范式：

  1. **草稿阶段**：用小模型快速生成 $K$ 个候选 token
  2. **验证阶段**：用大模型一次性并行验证这 $K$ 个 token
  3. **接受/拒绝**：保留被接受的 token，从拒绝位置重新生成

    4. 这个框架的数学基础是拒绝采样（rejection sampling），保证输出分布与目标模型完全一致。

    1.2 问题的深化

    推测解码的瓶颈在于接受率。如果草稿模型和目标模型差距太大，大部分草稿 token 会被拒绝，加速效果大打折扣。

    传统的解决思路是"提升草稿模型质量"——用更大的草稿模型、更复杂的草稿策略。但这直接增加了草稿阶段的成本，形成了一种跷跷板效应。

    DSpark 提出了一个全新的思路：不提升草稿模型，而是在验证阶段添加一个极轻量的修正模块，把草稿分布"推"向目标分布。

    1.3 问题的极致

    修正模块的设计面临一个核心矛盾：

    • 修正能力越强 → 需要的计算越多 → 增加的延迟越大
    • 延迟增加越大 → 推测加速的效果越差

    如何在"修正能力"和"计算成本"之间找到最优折中？这就是马尔可夫头要回答的问题。

    答案出人意料地简洁：只看前一个 token，用 rank-256 的低秩矩阵做修正。

    二、立在什么不证之命题上？

    2.1 第一公理：局部性假设

    马尔可夫头建立在一个核心假设之上：在短草稿序列中，token 之间的修正模式主要是局部的。

    这意味着：要修正第 $i$ 个位置的预测，只需要知道第 $i-1$ 个位置的 token 是什么。更早的 token 对修正的贡献可以忽略。

    用信息论的语言：

    $$

    I(q_i; q_{1:i-2} | q_{i-1}) \approx 0

    $$

    即给定前一个 token 后，更早的 token 与当前位置的条件互信息近似为零。

    这个假设在自然语言中并不严格成立。但在推测解码的典型草稿长度（K=4~16）下，它是一个足够好的近似。实验数据表明，一阶马尔可夫修正能解释 ~70%-91% 的修正模式（取决于文本类型）。

    2.2 第二公理：低秩性假设

    第二个不证之命题是：修正矩阵的有效秩很低。

    修正函数 $f: \mathbb{R}^{d_{\text{embed}}} \to \mathbb{R}^{|V|}$ 虽然输出维度高达 128K，但其有效秩只有 ~256。这意味着修正模式实际上分布在一个 256 维的子空间中。

    这个假设的依据来自对大语言模型内部表示的经验观察：

    • 模型的隐藏状态具有高度的冗余性
    • 有效的语义变化集中在低维子空间中
    • 修正模式（作为两个分布的差异）比原始分布更低维

      • 2.3 第三公理：成本敏感性假设

第三个隐含假设是：在推测解码系统中，额外延迟的边际成本远大于修正质量的边际收益。

这意味着：即使一个更复杂的修正模块能带来 2% 的额外接受率提升，但如果它增加了 5% 的延迟，净效果仍然是负的。

这个假设在 GPU 推理场景下尤其成立，因为 GPU 的计算资源已经被主模型充分利用，任何额外计算都会直接转化为延迟。

三、用什么框架看世界？

3.1 成本-收益帕累托前沿

马尔可夫头的分析框架是一个二维优化问题：

$$

\text{Pareto}(r) = \{(\text{cost}(r), \text{benefit}(r)) : r \in \mathcal{R}\}

$$

其中 $r$ 是修正模块的设计参数（如 rank、输入范围等），cost 是额外延迟，benefit 是接受率提升。

帕累托最优解是那些"不可能在不增加成本的情况下提升收益"的点。马尔可夫头（rank 256）恰好位于帕累托前沿上。


收益（接受率提升）
    ^
    |                          * 完整修正头
    |                     * Rank-1024
    |                 * Rank-512
    |            * Rank-256 ← 马尔可夫头（帕累托最优点）
    |        * Rank-128
    |    * Rank-64
    | * Rank-32
    +----------------------------------------> 成本（额外延迟）

3.2 修正-预测对偶框架

马尔可夫头引入了一个新的分析维度：修正 vs 预测。

范式 目标 输入 输出 代表方法
预测 生成新 token 完整上下文 新 token logits Medusa, EAGLE
修正 调整已有草稿 前一个 token 修正增量 马尔可夫头

预测范式需要生成完整的 logits 分布，而修正范式只需要生成增量。增量的内在维度远低于完整分布，这是马尔可夫头能如此轻量的根本原因。

3.3 信息瓶颈分析框架

用信息瓶颈理论分析马尔可夫头的信息流：

$$

\underbrace{\text{embed}(q_{i-1})}_{X \in \mathbb{R}^{4096}} \xrightarrow{\text{bottleneck}} \underbrace{z_i}_{Z \in \mathbb{R}^{256}} \xrightarrow{\text{expand}} \underbrace{\Delta \log p_i}_{Y \in \mathbb{R}^{128000}}

$$

信息瓶颈的宽度（256 维）决定了修正头能传递多少信息。256 维恰好是一个"甜蜜点"：

  • 足够宽以捕获主要的修正模式（95% 能量）
  • 足够窄以限制计算成本（~67.6 MFLOPs）
  • 天然起到正则化作用（防止过拟合）

    • 四、得出什么结论？

    4.1 核心结论：极简修正的优越性

    马尔可夫头的核心结论是：在推测解码的短草稿场景下，最简单的一阶马尔可夫修正就已经足够好。

    "足够好"的含义是：

    1. 接受长度提升 ~30%，接近理论最优（完整修正头 ~32%）
    2. 额外延迟仅 0.2%-1.3%，几乎可以忽略
    3. 参数量仅 33.8M，远小于其他方法

4. 不需要修改主模型架构，部署友好

4.2 推论一：修正比预测便宜一个数量级

修正已有草稿比从头预测新 token 容易得多。数学上：

$$

\text{cost}(\text{修正}) = O(r \cdot |V|) \ll \text{cost}(\text{预测}) = O(d_{\text{model}} \cdot |V|)

$$

因为 $r \ll d_{\text{model}}$（256 vs 4096），修正的计算成本约为预测的 1/16。

4.3 推论二：低秩是天然的效率-质量折中

低秩分解不仅是一种计算优化手段，更是一种天然的正则化和效率-质量折中机制。rank 的选择直接决定了修正头的"分辨率"：

Rank 分辨率 成本 质量 适用场景
32 低 极低 一般 资源极度受限
128 中 低 良好 边缘设备
**256** **高** **极低** **优秀** **通用场景**
512 极高 低 微幅提升 追求极致
1024 过饱和 中 饱和 不推荐

4.4 推论三：工程审美的胜利

马尔可夫头的深层意义在于它体现了一种工程审美：在系统设计中，最优解往往不是最复杂的解，而是恰好足够好的解。

这种审美可以推广到更广泛的系统设计领域：

  • 不需要完美的模型，只需要足够好的近似
  • 不需要最大的容量，只需要恰好匹配需求
  • 不需要最复杂的架构，只需要最简洁的有效方案

    • 五、萃出一个 Takeaway

    取景框（Frame）

    "一阶修正，零阶成本"

    这六个字浓缩了马尔可夫头的全部精髓：

    • **一阶修正**：只看前一个 token 来修正当前位置，这是信息论上的"一阶"近似
    • **零阶成本**：计算成本几乎为零（相对于主模型），这是工程上的"零阶"开销

    这个取景框的力量在于它的可迁移性。在任何需要在"质量"和"成本"之间做折中的系统设计场景中，都可以问自己：

    "我能不能用一个'一阶修正，零阶成本'的方案，而不是追求'高阶修正，高阶成本'的方案？"

    模型（Model）

    马尔可夫头可以形式化为一个通用的修正框架：

    $$

    \hat{y} = y_{\text{base}} + \underbrace{W_B \cdot \sigma(W_A \cdot x_{\text{prev}})}_{\text{低秩修正}}

    $$

    其中：

    • $y_{\text{base}}$ 是基础预测（草稿模型的输出）
    • $x_{\text{prev}}$ 是前一个位置的输入
    • $W_A, W_B$ 是低秩修正矩阵
    • $\sigma$ 是非线性激活

    这个框架可以应用到任何需要"轻量修正"的场景：

    • 语音识别中的语言模型修正
    • 机器翻译中的目标语言流畅度修正
    • 推荐系统中的上下文修正
    • 控制系统中的参考轨迹修正

      • 洞见（Insight）

      "完美的修正是不必要的；足够好的修正 + 几乎为零的成本 = 最优的系统设计"

      这个洞见挑战了工程界常见的"追求完美"倾向。在资源受限的场景下（实时推理、边缘计算、嵌入式系统），一个"足够好"的轻量方案往往比一个"完美"的重量方案更有价值。

      六、代码示例：完整的马尔可夫头训练与推理

      
"""
马尔可夫头完整实现：训练 + 推理 + 集成到推测解码管线
"""
import torch
import torch.nn as nn
import torch.nn.functional as F
from torch.utils.data import DataLoader, Dataset

class MarkovHead(nn.Module):
    """马尔可夫修正头"""
    
    def __init__(self, embed_dim=4096, vocab_size=128000, rank=256):
        super().__init__()
        self.down_proj = nn.Linear(embed_dim, rank, bias=False)
        self.act = nn.SiLU()
        self.up_proj = nn.Linear(rank, vocab_size, bias=False)
        
        # 小尺度初始化
        nn.init.normal_(self.down_proj.weight, std=0.02)
        nn.init.normal_(self.up_proj.weight, std=0.02)
    
    def forward(self, prev_embed):
        """
        Args:
            prev_embed: [batch, seq_len, embed_dim] 前一个 token 的嵌入
        Returns:
            delta_logits: [batch, seq_len, vocab_size] 修正增量
        """
        hidden = self.act(self.down_proj(prev_embed))
        return self.up_proj(hidden)


class MarkovHeadTrainer:
    """马尔可夫头训练器"""
    
    def __init__(self, target_model, draft_model, embed_dim, vocab_size, rank=256):
        self.target_model = target_model
        self.draft_model = draft_model
        self.markov_head = MarkovHead(embed_dim, vocab_size, rank)
        self.optimizer = torch.optim.AdamW(
            self.markov_head.parameters(), lr=1e-4, weight_decay=0.01
        )
    
    def compute_training_loss(self, batch):
        """
        训练损失：最小化修正后的分布与目标分布的 KL 散度
        
        L = KL(p_target || p_draft + delta)
        """
        input_ids = batch["input_ids"]
        
        # 获取目标模型和草稿模型的 logits
        with torch.no_grad():
            target_logits = self.target_model(input_ids).logits
            draft_logits = self.draft_model(input_ids).logits
        
        # 获取前一个 token 的嵌入
        with torch.no_grad():
            prev_embed = self._get_embedding(input_ids)
        
        # 计算修正增量
        delta_logits = self.markov_head(prev_embed)
        
        # 修正后的 logits
        corrected_logits = draft_logits + delta_logits
        
        # KL 散度损失
        target_probs = F.softmax(target_logits, dim=-1)
        corrected_log_probs = F.log_softmax(corrected_logits, dim=-1)
        loss = F.kl_div(corrected_log_probs, target_probs, reduction="batchmean")
        
        return loss
    
    def train_step(self, batch):
        self.optimizer.zero_grad()
        loss = self.compute_training_loss(batch)
        loss.backward()
        self.optimizer.step()
        return loss.item()


class SpeculativeDecoderWithMarkov:
    """集成马尔可夫头的推测解码器"""
    
    def __init__(self, target_model, draft_model, markov_head, 
                 tokenizer, max_draft_len=16):
        self.target_model = target_model
        self.draft_model = draft_model
        self.markov_head = markov_head
        self.tokenizer = tokenizer
        self.max_draft_len = max_draft_len
    
    def generate(self, prompt_ids, max_new_tokens=256):
        """带马尔可夫修正的推测解码生成"""
        generated = prompt_ids.clone()
        
        while len(generated) < len(prompt_ids) + max_new_tokens:
            # 1. 草稿阶段：生成 K 个候选 token
            draft_ids = self._generate_draft(generated, self.max_draft_len)
            
            # 2. 验证阶段：目标模型并行验证
            with torch.no_grad():
                target_logits = self.target_model(
                    torch.cat([generated, draft_ids])
                ).logits
            
            # 3. 修正阶段：马尔可夫头修正草稿 logits
            prev_embeds = self._get_embedding(
                torch.cat([generated[-1:], draft_ids[:-1]])
            )
            delta_logits = self.markov_head(prev_embeds)
            corrected_logits = self.draft_model(draft_ids).logits + delta_logits
            
            # 4. 接受/拒绝
            accepted = self._verify_and_accept(
                target_logits, corrected_logits, draft_ids
            )
            
            # 5. 更新生成的 token
            generated = torch.cat([generated, accepted])
        
        return generated
    
    def _generate_draft(self, prefix, k):
        """自回归草稿生成"""
        draft_ids = []
        current = prefix.clone()
        for _ in range(k):
            logits = self.draft_model(current).logits[:, -1]
            next_token = torch.argmax(logits, dim=-1, keepdim=True)
            draft_ids.append(next_token)
            current = torch.cat([current, next_token], dim=1)
        return torch.cat(draft_ids, dim=1)
    
    def _verify_and_accept(self, target_logits, draft_logits, draft_ids):
        """拒绝采样验证"""
        accepted = []
        for i in range(draft_ids.shape[1]):
            target_probs = F.softmax(target_logits[:, i], dim=-1)
            draft_probs = F.softmax(draft_logits[:, i], dim=-1)
            
            token = draft_ids[:, i]
            r = target_probs[0, token] / (draft_probs[0, token] + 1e-8)
            
            if torch.rand(1).item() < r.item():
                accepted.append(token)
            else:
                # 拒绝：从修正分布中采样一个 token
                new_token = torch.multinomial(target_probs, 1)
                accepted.append(new_token)
                break
        
        return torch.cat(accepted, dim=1)
    
    def _get_embedding(self, input_ids):
        """获取 token 嵌入"""
        return self.draft_model.get_input_embeddings()(input_ids)
