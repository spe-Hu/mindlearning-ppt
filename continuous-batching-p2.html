二、核心论文详细分析

论文 1：Orca — 连续批处理的奠基之作

论文信息：

  • 标题：Orca: A Distributed Serving System for Transformer-Based Generative Models
  • 作者：Gimelshein, Kharitonov, et al. (NVIDIA)
  • 发表：OSDI 2022 / arXiv 2022
  • 引用数：1500+

核心问题：

如何高效地为 Transformer 生成模型提供在线推理服务？传统的静态批处理导致 GPU 利用率极低（10-30%），因为 batch 中的所有请求必须同步完成。

核心贡献：

  1. **Iteration-level Scheduling（迭代级调度）**：

    2. Orca 的核心洞见是将调度粒度从"请求级"降低到"迭代级"。在传统的请求级调度中，一个 batch 从开始到结束（所有请求生成 EOS）是不可分割的整体。Orca 打破了这个约束——在每一次前向传播（iteration）中，调度器可以动态决定哪些请求参与计算。

    具体机制：

    • 每个 iteration 开始时，调度器检查当前 batch 中是否有请求已完成（生成 EOS）
    • 已完成的请求立即从 batch 中移除
    • 等待队列中的新请求被插入 batch
    • 所有请求仍然同步执行一次前向传播（每个请求生成 1 个 token）
    • **All-or-Nothing Batch Scheduling**：

    Orca 还提出了批调度的优化策略。当新请求到达时，调度器需要决定是否将其加入当前 batch。Orca 的分析表明：只有当 batch 中所有请求的"剩余生成长度"相近时，批处理才是高效的。如果新请求的预计生成长度远大于 batch 中已有请求，加入它会降低整体效率。

    1. **分布式架构**：

      2. Orca 设计了分布式推理服务架构，包括：

      • Coordinator：接收请求、分配到其他节点
      • Worker：执行模型推理
      • 动态负载均衡：根据各 worker 的队列长度和当前 batch 状态做负载均衡

      性能数据：

      • 在 GPT-175B 上，相比静态批处理，吞吐量提升 4 倍
      • P99 延迟降低 70%
      • GPU 利用率从 ~20% 提升至 ~80%

      局限性：

      • 没有解决 KV Cache 的显存管理问题。KV Cache 仍然是连续分配的，导致严重的显存碎片化。
      • 当请求的生成长度差异很大时，仍然存在一定的资源浪费。
      • 没有考虑 prefill 和 decode 阶段的异构性。

      历史地位：

      Orca 是连续批处理概念的正式提出者。它证明了"在 decode 阶段，每个请求每步的工作量相同（1 个 token），因此可以在 iteration 边界动态调度请求"这一核心洞见。后续所有连续批处理工作（vLLM、TensorRT-LLM、SGLang）都建立在 Orca 的迭代级调度思想之上。

      论文 2：vLLM / PagedAttention — 显存管理革命

      论文信息：

      • 标题：Efficient Memory Management for Large Language Model Serving with PagedAttention
      • 作者：Kwon, Li, et al. (UC Berkeley)
      • 发表：SOSP 2023
      • 引用数：5000+

      核心问题：

      大语言模型推理服务中，KV Cache 的显存管理是核心瓶颈。传统方案为每个请求预分配连续的 KV Cache 空间，导致 60-80% 的显存浪费（内部碎片 + 外部碎片）。这严重限制了可同时服务的请求数量，从而限制了吞吐量。

      核心贡献：

      1. **PagedAttention（分页注意力）**：

        2. 核心类比：KV Cache 的显存管理可以类比操作系统的虚拟内存管理。

        在操作系统中，进程的虚拟地址空间被分成固定大小的页（通常 4KB），物理内存也被分成同样大小的页框。通过页表（page table）将虚拟页映射到物理页框，使得进程的内存不需要连续物理空间。

        PagedAttention 采用完全相同的思路：

        • KV Cache 被分成固定大小的**块（block）**，每个块存储固定数量 token（如 16 个）的 KV 值
        • 物理显存被分成同样大小的**物理块**
        • 每个请求通过**块表（block table）**将其逻辑块映射到物理块
        • 逻辑块不需要映射到连续的物理块

        块的大小选择是一个工程权衡：

        • 太小（如 1 token）：页表开销大，管理复杂
        • 太大（如 256 token）：内部碎片严重
        • 实践中的最优值：16 token（vLLM 默认值）
        • **Copy-on-Write（写时复制）**：

        借鉴操作系统的 Copy-on-Write 机制，PagedAttention 支持多个请求共享同一个物理块的 KV Cache。

        应用场景：

        • **前缀共享**：多个请求使用相同的 system prompt 或前缀，可以共享前缀对应的物理块
        • **Beam Search**：beam search 的多条候选序列在分叉前共享相同的 KV Cache
        • **Parallel Sampling**：对同一 prompt 的多次采样共享前缀 KV Cache

        写时复制机制：当某个请求需要修改共享块的 KV 值时（如在新生成的 token 处写入 KV），系统先复制该块，再在副本上修改。这保证了其他共享该块的请求不受影响。

        1. **内存优化效果**：
        2. 内部碎片降低到 <5%（只有最后一个块可能有未使用的空间）
        3. 外部碎片几乎消除（所有物理块大小相同，可以任意分配）
        4. 前缀共享进一步减少了 50-80% 的重复 KV Cache 存储

          5. 4. 与 Orca 调度的集成：

          vLLM 将 PagedAttention 与 Orca 的迭代级调度无缝集成。PagedAttention 解决了显存管理问题，Orca 调度解决了请求调度问题。两者结合实现了：

          • 吞吐量比静态批处理提升 4-24 倍
          • 支持更大的 batch size（因为显存利用更高效）
          • 零浪费的 KV Cache 管理

          性能数据：

          • 在 OPT-175B 上，吞吐量比 FasterTransformer 提升 24 倍
          • 在 OPT-13B 上，吞吐量提升 14 倍
          • 在 ChatGPT 级别的模型上，显存浪费从 60-80% 降低到 <4%
          • 前缀共享场景下，KV Cache 存储减少 55%

          局限性：

          • 页表本身占用显存（但占比很小，通常 <1%）
          • 非连续物理块的访问可能引入少量的内存访问碎片（但通过预取和缓存可以缓解）
          • 对于极长序列（如 100K+ token），块表的查找开销可能变得显著

          历史地位：

          vLLM/PagedAttention 是连续批处理从学术概念走向工业级部署的关键推动力。它解决了 Orca 未解决的显存管理问题，使得连续批处理在实际部署中的效果大幅提升。vLLM 开源后迅速成为最流行的推理引擎，GitHub stars 超过 25k，被 OpenAI、Anthropic、Together AI 等主流 AI 公司采用。

          论文 3：DeepSpeed-FastGen / Dynamic SplitFuse

          论文信息：

          • 标题：DeepSpeed-FastGen: High-throughput Text Generation with DeepSpeed-MII and FastGen
          • 作者：Microsoft DeepSpeed Team
          • 发表：2024
          • 引用数：300+

          核心问题：

          Orca 和 vLLM 的连续批处理方案中，prefill 阶段和 decode 阶段在同一个 GPU 上混合执行。但这两个阶段的特性截然不同：

          • Prefill：计算密集型（处理完整 prompt，seq_len 可能很长）
          • Decode：内存带宽密集型（每次只处理 1 个 token）

          当一个长 prompt 请求进入 prefill 阶段时，它可能占用 GPU 数百毫秒，期间所有 decode 请求都被阻塞。这导致 decode 请求的延迟出现严重的"毛刺"（spike）。

          核心贡献：

          1. **Dynamic SplitFuse（动态分割融合）**：

            2. 核心思想是将 prefill 阶段的长 prompt 分成多个 chunk，每个 chunk 与 decode 请求混合执行。

            具体机制：

            • 调度器维护一个"预算"（budget），表示当前 iteration 可以处理的总 token 数量
            • 将 decode 请求的 token（每个请求 1 个）和 prefill chunk 的 token 组合成一个 batch
            • 例如：budget=2048，batch 中包含 128 个 decode token（128 个请求各 1 个）+ 1920 个 prefill chunk token

            这确保了每个 iteration 的总工作量是恒定的，避免了长 prefill 阻塞 decode 的问题。

            1. **Chunked Prefill 的调度策略**：

              2. SplitFuse 的关键设计决策是如何选择 chunk 大小。太小的 chunk 增加了调度开销和 prefill 的总时间（因为每个 chunk 都需要一次完整的前向传播），太大的 chunk 又回到了"阻塞 decode"的问题。

              FastGen 的策略：

              • 根据当前 decode 请求的数量动态调整 chunk 大小
              • decode 请求越多，chunk 越小（优先保证 decode 延迟）
              • decode 请求越少，chunk 越大（优先加速 prefill）
              • **与 vLLM 的对比**：
              • vLLM 的连续批处理在 decode 阶段做得很好，但 prefill 仍然是"原子性"的（一个 prompt 必须一次性处理完）
              • FastGen 将 prefill 也"连续化"了——长 prompt 被分成多个 chunk，跨多个 iteration 处理

              性能数据：

              • 在 LLaMA-2-70B 上，相比 vLLM，吞吐量提升 1.4-2.0 倍
              • P99 延迟降低 60%（消除了 prefill 导致的延迟毛刺）
              • 在混合长短 prompt 场景下效果最显著

              历史地位：

              DeepSpeed-FastGen 解决了连续批处理中最后一个"不连续"的环节——prefill 阶段。通过 Chunked Prefill，它实现了真正的"全阶段连续"。这一思想在 2025-2026 年被广泛采纳，成为推理引擎的标准特性。

              论文 4：SGLang / RadixAttention — 基于 Radix Tree 的 KV Cache 管理

              论文信息：

              • 标题：SGLang: Efficient Execution of Structured Language Model Programs
              • 作者：Zheng, Yin, et al. (Stanford)
              • 发表：OSDI 2024 / arXiv 2024
              • 引用数：800+

              核心问题：

              vLLM 的 PagedAttention 通过分页解决了显存碎片化问题，但前缀匹配仍然是"手动"的——用户需要显式指定哪些请求可以共享前缀。在实际场景中，前缀关系可能非常复杂（如多轮对话、共享 system prompt、相似的用户指令），需要自动化的前缀匹配和缓存管理。

              核心贡献：

              1. **RadixAttention**：

                2. 使用 Radix Tree（基数树/压缩前缀树） 来管理所有请求的 KV Cache。

                Radix Tree 的特性：

                • 每个节点代表一个 token 序列
                • 从根到任意节点的路径代表一个前缀
                • 共享前缀的请求自然汇聚到同一子树
                • 支持高效的 LCP（Longest Common Prefix）查找

                工作方式：

                • 当新请求到达时，在 Radix Tree 中查找其 prompt 的最长匹配前缀
                • 匹配到的前缀对应的 KV Cache 直接复用（无需重新计算）
                • 只有未匹配的部分需要执行 prefill
                • 生成的新 token 的 KV Cache 插入 Radix Tree，供后续请求使用
                • **自动前缀缓存（Automatic Prefix Caching）**：

                RadixAttention 自动缓存所有计算过的 KV Cache，并在后续请求中自动复用。用户无需任何额外配置。

                示例：

                
   请求1: "System: You are a helpful assistant. User: Hello" → 完整 prefill
   请求2: "System: You are a helpful assistant. User: How are you" → 复用 "System: You are a helpful assistant. User: " 的 KV Cache
   请求3: "System: You are a helpful assistant. User: Hello, tell me about AI" → 复用 "System: You are a helpful assistant. User: Hello" 的 KV Cache
                1. **LRU 缓存淘汰**：

                  2. 当显存不足时，Radix Tree 中的 KV Cache 节点按 LRU（Least Recently Used）策略淘汰。最近被访问的节点保留，最久未访问的节点优先释放。

                  4. 结构化生成支持：

                  SGLang 不仅优化了 KV Cache 管理，还支持结构化的生成模式（如 JSON schema 约束、正则表达式约束），通过约束解码（constrained decoding）确保输出符合指定格式。

                  性能数据：

                  • 在多轮对话场景下，相比 vLLM，吞吐量提升 1.5-3.0 倍（前缀缓存命中率高）
                  • 在共享 system prompt 场景下，首 token 延迟降低 80%
                  • Radix Tree 的管理开销 <2%

                  历史地位：

                  SGLang/RadixAttention 将 KV Cache 管理从"分页"提升到"智能缓存"。Radix Tree 的数据结构天然适合处理前缀关系复杂的场景（多轮对话、RAG、Agent 工作流），使得 KV Cache 的复用率大幅提升。

                  论文 5：Splitwise / DistServe — Prefill-Decode 分离架构

                  论文信息：

                  • 标题 1：Splitwise: Efficient Generative LLM Inference Using Phase Splitting
                  • 作者 1：Patel et al. (University of Washington)
                  • 发表 1：2024
                  • 标题 2：DistServe: Disaggregating Prefill and Decoding for Goodput-optimized Large Language Model Serving
                  • 作者 2：Zhong et al. (UC Berkeley)
                  • 发表 2：OSDI 2024

                  核心问题：

                  连续批处理将 prefill 和 decode 混合在同一 GPU 上执行，但这两个阶段的资源需求完全不同：

                  • Prefill 需要大量计算（高 FLOPs 利用率）
                  • Decode 需要大量显存带宽（高 memory bandwidth 利用率）

                  在同一 GPU 上混合执行意味着无法同时优化两个阶段。

                  核心贡献：

                  1. **PD Disaggregation（Prefill-Decode 分离）**：

                    2. 将 prefill 和 decode 分配到不同的 GPU 集群：

                    • Prefill 集群：优化计算性能（高 FLOPs GPU，如 H100）
                    • Decode 集群：优化显存带宽（高带宽 GPU，或相同 GPU 但更大的 batch size）

                    工作流程：

                    
   请求到达 → Prefill GPU 集群（处理 prompt，生成 KV Cache）
            → 通过高速网络传输 KV Cache 到 Decode GPU 集群
            → Decode GPU 集群（自回归生成 token）
            → 返回结果
                    1. **KV Cache 传输优化**：

                      2. PD 分离的关键挑战是如何高效传输 KV Cache。对于一个 70B 模型，seq_len=2048 的请求，KV Cache 大小约 10-20 GB。传输这么大的数据需要高速网络（如 InfiniBand/NVLink）。

                      优化策略：

                      • KV Cache 量化：传输前将 KV Cache 从 FP16 量化到 INT8，减少传输量
                      • 流水线传输：将 KV Cache 分成多个块，逐块传输，与 decode 的第一批 token 生成重叠
                      • 预取：预测请求的 KV Cache 大小，提前在 decode 集群预留显存
                      • **Goodput 优化**：

                      DistServe 提出了 Goodput 指标——在满足 SLO（Service Level Objective，如 P99 延迟 < 2s）约束下的有效吞吐量。PD 分离的目标是最大化 Goodput，而非简单的吞吐量。

                      性能数据：

                      • 在 LLaMA-2-70B 上，相比混合部署，Goodput 提升 1.5-2.0 倍
                      • P99 延迟降低 50%
                      • 在异构 GPU 集群中效果更显著（可以用低带宽 GPU 做 decode）

                      历史地位：

                      PD 分离代表了连续批处理架构的下一个演进方向。从"在单个 GPU 上做连续批处理"到"在集群级别做连续批处理"，PD 分离将调度优化的空间从单 GPU 扩展到了整个数据中心。

                      论文 6：TensorRT-LLM 技术白皮书

                      论文信息：

                      • 标题：NVIDIA TensorRT-LLM: Optimizing Large Language Model Inference
                      • 作者：NVIDIA
                      • 发表：GTC 2024 / 持续更新
                      • 类型：技术白皮书

                      核心贡献：

                      1. **In-flight Batching**：

                        2. NVIDIA 在 TensorRT-LLM 中实现了硬件级的连续批处理优化。In-flight Batching 是 Orca 迭代级调度的增强版：

                        • 利用 CUDA Graph 减少 kernel launch 开销
                        • 利用 Tensor Core 的异步特性，实现 batch 组成的动态调整
                        • 支持 FP8 量化下的连续批处理
                        • **Paged KV Cache 的硬件优化**：

                        TensorRT-LLM 在 H100/H200 上对 Paged KV Cache 做了硬件级优化：

                        • 利用 H100 的 TMA（Tensor Memory Accelerator）实现高效的非连续显存访问
                        • 利用 L2 cache 分区减少 KV Cache 访问的 cache miss
                        • 预取机制：根据当前 batch 的 block table，提前将下一 iteration 需要的 KV Cache 页加载到 L2 cache
                        • **FP8 + 连续批处理的联合优化**：

                        H100 原生支持 FP8 计算。TensorRT-LLM 将 FP8 量化与连续批处理结合：

                        • FP8 将模型权重大小减半，T_memory 减半
                        • 连续批处理在更小的 batch size 下即可达到高 GPU 利用率
                        • 两者协同：FP8 减少显存占用 → 支持更大 batch size → 更高吞吐量

                        性能数据：

                        • 在 LLaMA-70B (FP8) + H100 上，单节点吞吐量达到 2000+ tokens/s
                        • 相比 PyTorch 原生推理，吞吐量提升 8-12 倍
                        • 相比 vLLM，在相同硬件上吞吐量提升 1.3-1.8 倍（得益于硬件级优化）

                          四、技术演进脉络

                          
核心问题：如何最大化 GPU 在 LLM 推理中的利用率？

├── 子问题1：如何动态调度请求？
│   └── Orca (2022) → Iteration-level Scheduling
│       └── 每个 iteration 动态调整 batch 组成
│
├── 子问题2：如何高效管理 KV Cache 显存？
│   └── vLLM (2023) → PagedAttention
│       └── 分页 + Copy-on-Write + 前缀共享
│
├── 子问题3：如何避免 Prefill 阻塞 Decode？
│   └── FastGen (2024) → Dynamic SplitFuse
│       └── Chunked Prefill + 混合调度
│
├── 子问题4：如何智能复用 KV Cache？
│   └── SGLang (2024) → RadixAttention
│       └── Radix Tree + 自动前缀缓存
│
├── 子问题5：如何在集群级别优化？
│   └── Splitwise/DistServe (2024) → PD Disaggregation
│       └── Prefill/Decode 分离 + KV Cache 传输
│
└── 子问题6：如何在硬件级别优化？
    └── TensorRT-LLM (2024-25) → 硬件级连续批处理
        └── CUDA Graph + TMA + FP8
