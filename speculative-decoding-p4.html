卡片 1：核心定义卡

标题：推测解码 Speculative Decoding 副标题：用廉价的猜测换昂贵的验证

定义：推测解码是大语言模型推理加速的核心技术。它用一个轻量级"草稿模型"快速生成多个候选 token，再用目标大模型一次性并行验证这些 token 的正确性——接受正确的、拒绝错误的——从而在数学上保证输出分布与原始模型完全一致的前提下，实现 2-5 倍的推理加速。

核心公式：

接受概率 = min(1, p_target(x) / p_draft(x))
理论加速比 = 1 / (1 - alpha)，alpha = 接受率
最优草稿长度 K* ≈ -1 / ln(alpha)

关键洞见：推测解码是"免费午餐"——它不牺牲任何输出质量，纯粹利用 GPU 自回归解码时 99% 闲置的算力。前提是草稿模型的预测要足够准确。

一句话记忆：小模型快速猜，大模型一次验；猜对了白赚，猜错了不亏。

卡片 2：原理卡

标题：推测解码工作原理 副标题：草稿-验证的两阶段博弈

标准自回归解码：

Step 1: 输入 → 模型前向传播 → token_1 → 输出
Step 2: 输入+token_1 → 模型前向传播 → token_2 → 输出
Step 3: 输入+token_1+token_2 → 模型前向传播 → token_3 → 输出
...
每步都需要一次完整的前向传播，每次前向传播都要读取整个模型权重

推测解码：

草稿阶段：草稿模型自回归生成 K=5 个候选 token: [t1, t2, t3, t4, t5]
验证阶段：目标模型一次前向传播，并行验证所有 5 个 token
接受/拒绝：t1 接受 ✓, t2 接受 ✓, t3 接受 ✓, t4 拒绝 ✗
结果：一次前向传播产出 3 个 token（而非 1 个），加速 3x
从 t4 位置重新开始下一轮推测

为什么验证 K 个 token 的成本 ≈ 生成 1 个 token 的成本？ 因为瓶颈是内存搬运（读取模型权重），不是计算。验证 5 个 token 和验证 1 个 token 都需要读取相同的模型权重（几十 GB），额外的矩阵乘法计算在内存搬运时间内就能完成。

GPU 利用率分析：

自回归解码时 GPU 时间分布：
├── 内存搬运（读权重）：~99%
└── 矩阵乘法（实际计算）：~1%
推测解码利用的就是这 99% 时间里闲置的 1% 算力

卡片 3：公式卡

标题：推测解码核心公式 副标题：从接受率到加速比

加速比公式：

speedup = 1 / (1 - alpha)
其中 alpha = 平均接受率

alpha = 0.5 → speedup = 2x
alpha = 0.7 → speedup = 3.3x
alpha = 0.8 → speedup = 5x
alpha = 0.9 → speedup = 10x
alpha = 0.95 → speedup = 20x

最优草稿长度：

K* = -1 / ln(alpha)
alpha = 0.8 → K* ≈ 5
alpha = 0.5 → K* ≈ 1
alpha = 0.9 → K* ≈ 10

接受准则：

对于草稿模型生成的 token x_i：
  接受概率 = min(1, p_target(x_i) / p_draft(x_i))

直觉理解：
- 如果 p_target >> p_draft（目标模型比草稿模型更认可这个 token）→ 必定接受
- 如果 p_target ≈ p_draft（两者一致）→ 大概率接受
- 如果 p_target << p_draft（目标模型不认可草稿的猜测）→ 大概率拒绝

加速比推导：

每轮推测的成本 = 1 次草稿生成（K 步，小模型） + 1 次验证（大模型）
每轮推测的期望产出 = sum_{i=1}^{K} alpha^i ≈ alpha / (1-alpha)
当 K 足够大时，每轮产出 ≈ 1/(1-alpha) 个 token
每轮成本 ≈ 2 次大模型前向传播（草稿模型成本忽略）
标准解码产出 1 个 token 需 1 次前向传播
加速比 = (1/(1-alpha)) / 2 ≈ 1/(1-alpha)（忽略草稿模型成本时）

卡片 6：MTP 卡

标题：Multi-Token Prediction (MTP) 副标题：训练时就学会"一次看到多个未来"

标准训练 vs MTP 训练：

标准训练：
输入: [token_1, token_2, token_3]
目标: [token_2, token_3, token_4]
每次只预测下一个 token

MTP 训练：
输入: [token_1, token_2, token_3]
目标 1: [token_2]              (预测第 1 个未来 token)
目标 2: [token_2, token_3]     (预测第 1-2 个未来 token)
目标 3: [token_2, token_3, token_4]  (预测第 1-3 个未来 token)
同时预测多个未来 token

MTP 在推测解码中的角色：

推理时：
MTP 模块直接作为草稿生成器
无需额外的草稿模型
MTP 模块与主模型共享特征表示 → 接受率极高

MTP 的深层意义： - 改变了"先训练再优化"的范式 → "训练时就为推理做准备" - MTP 不是外挂，是内建 → 推测能力成为模型的一部分 - 训练-推理协同设计的开端

标题：推测解码 × 自动驾驶 副标题：实时推理的生死时速

为什么自动驾驶需要推测解码？ - 端到端模型的推理延迟问题：从传感器输入到控制输出，要求 <50ms - 车端 GPU 资源有限：Orin/Thor 的算力和内存都远小于云端 - 多传感器融合推理：摄像头 + 激光雷达 + 毫米波雷达的联合推理

自动驾驶场景的特殊性：

车端推理：延迟容忍度极低（<50ms），关注确定性
云端：batch size 大，推测解码加速比可能下降
车端：batch size = 1，推测解码加速比最大化

数据飞轮 + 推测解码： - 数据飞轮持续产生训练数据 → 模型持续更新 - 推测解码确保每次更新的模型都能高效推理 - MTP 训练使车端模型天生具备推测能力

车端部署约束：

功耗：车端 GPU 功耗受限（<100W）

内存：车载 GPU 内存有限（16-64GB）
实时性：硬实时要求，不能容忍偶发延迟

卡片 9：推测解码代码示例卡

标题：推测解码伪代码 副标题：从原理到实现

核心算法：

def speculative_decoding(target_model, draft_model, prompt, K=5):
    """
    target_model: 目标大模型
    draft_model: 草稿小模型
    K: 草稿长度
    """
    tokens = tokenize(prompt)

    while not done:
        # 草稿阶段：小模型自回归生成 K 个候选 token
        draft_tokens = []
        for i in range(K):
            p_draft = draft_model(tokens)
            x_i = sample(p_draft)
            draft_tokens.append(x_i)
            tokens.append(x_i)

        # 验证阶段：大模型一次性前向传播验证所有 K 个 token
        p_target_all = target_model(tokens[:-K])  # 并行计算

        # 接受-拒绝采样
        accepted = 0
        for i in range(K):
            p_t = p_target_all[i][draft_tokens[i]]
            p_d = draft_model_prob[i][draft_tokens[i]]
            accept_prob = min(1.0, p_t / p_d)
            if random() < accept_prob:
                accepted += 1
            else:
                # 拒绝：从修正分布中重新采样
                new_token = sample(corrected_distribution(p_t, p_d))
                tokens = tokens[:len(tokens)-K+i] + [new_token]
                break

        # 输出已接受的 token
        output(tokens[-accepted:])

关键实现细节： - 草稿模型的 KV Cache 可以复用（增量更新） - 目标模型的验证是一次前向传播（并行处理 K 个位置） - 接受-拒绝采样保证输出分布与原始模型完全一致

卡片 10：推测解码接受率分析卡

标题：接受率的决定因素 副标题：为什么有的场景接受率高，有的低？

影响接受率的三大因素：

1. 草稿模型质量
   - 草稿模型越接近目标模型 → 接受率越高
   - MTP 模块（共享表示）> 蒸馏模型 > 同系列小模型 > 不同系列小模型

2. 草稿长度 K
   - K 越大 → 后续 token 预测越不准 → 接受率越低
   - 最优 K ≈ -1/ln(alpha)

3. 输入文本特性
   - 低熵文本（自然语言续写）→ 接受率高（80-90%）
   - 高熵文本（代码/数学/创意写作）→ 接受率低（50-70%）

接受率实测数据：

| 草稿来源          | 自然语言 | 代码  | 数学  |
|-------------------|----------|-------|-------|
| 同系列小模型      | 65%      | 50%   | 40%   |
| Medusa 解码头     | 75%      | 60%   | 50%   |
| EAGLE 特征空间    | 82%      | 70%   | 58%   |
| MTP 模块          | 88%      | 78%   | 65%   |
