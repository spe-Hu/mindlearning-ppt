连续批处理 (Continuous Batching) 横纵分析报告

研究时间：2026-06-28 | 所属领域：AI/推理加速 | 研究对象类型：技术概念

关键词：Continuous Batching, Dynamic Batching, PagedAttention, vLLM, Orca, Memory-Bound, GPU Scheduling, KV Cache

一、一句话定义

连续批处理（Continuous Batching）是大语言模型推理服务的核心调度技术：在 GPU 执行每一次前向传播时，动态地将多个请求的不同位置 token 组合成一个 batch，让新请求随时插入、已完成的请求立即释放，从而实现 GPU 计算单元和显存的全时满载利用。 其本质是"搬一次也是搬，搬十次也是搬"——GPU 推理的瓶颈不是浮点运算（FLOPs），而是显存带宽（memory bandwidth），每一次前向传播都需要将模型权重从 HBM 搬运到计算核心，这个搬运成本与 batch 中的 token 数量几乎无关。连续批处理通过最大化每次搬运中"顺带处理"的 token 数量，将 GPU 利用率从静态批处理的 10-30% 提升至 80% 以上。从 2019 年 NVIDIA 的 Dynamic Batching 雏形，到 2022 年 Orca 的 Iteration-level Scheduling，到 2023 年 vLLM 的 PagedAttention，到 2025-2026 年 TensorRT-LLM/SGLang 的工业级部署，连续批处理已从学术构想演变为所有推理引擎的标配基础设施。

二、纵向分析：从静态批处理到连续批处理（2017-2026）

2.1 静态批处理时代（2017-2019）

Transformer 模型（Vaswani et al., 2017）的推理天然采用自回归解码：每一步生成一个 token，然后将这个 token 追加到序列中，再进行下一步。在早期部署中，推理服务采用最简单的静态批处理（Static Batching）策略：

工作方式：收集一批请求（如 batch_size=8），将这 8 个请求的 prompt 一起送入模型进行 prefill 阶段（处理完整输入序列），然后进入 decode 阶段（逐 token 生成）。关键约束是：整个 batch 中的所有请求必须同步前进——每一步，所有 8 个请求各生成 1 个 token。只有当 batch 中所有请求都生成了 EOS（结束符），这个 batch 才算完成，才能开始处理下一个 batch。

静态批处理的致命缺陷：

  1. **木桶效应**：batch 的处理时间取决于最慢的那个请求。如果 7 个请求在 20 步后完成，但第 8 个请求需要 200 步，那么前 7 个请求的 GPU 资源在 180 步中完全浪费。
  2. **新请求等待**：在一个 batch 处理期间到达的新请求必须排队等待，即使 GPU 有空闲的计算资源。
  3. **显存浪费**：已完成请求的 KV Cache 仍然占用显存，直到整个 batch 结束。
    4. 
静态批处理时间线：
请求1: [prefill][decode][decode]...[decode][EOS] ← 20步完成
请求2: [prefill][decode][decode]...[decode][EOS] ← 25步完成
请求3: [prefill][decode][decode]...[decode]......[decode][EOS] ← 200步完成
GPU:   |<---------- 必须等待所有请求完成 ----------->|
       |← 请求1&2完成后，GPU资源闲置180步 →|

量化分析：假设平均生成长度为 L，batch 中最大生成长度为 L_max。GPU 的有效利用率约为 L_avg / L_max。在长尾分布下（少数请求生成长度远超平均值），这个比值可能低至 10-30%。

2.2 动态批处理的萌芽（2019-2021）

NVIDIA 在 FasterTransformer（2019）中引入了动态批处理（Dynamic Batching）的初步概念：当一个请求完成（生成 EOS）后，立即用新请求替换它，而不是等待整个 batch 完成。

这解决了"新请求等待"的问题，但仍然有一个关键限制：替换只能在 iteration 边界进行，且所有请求的序列长度必须对齐（padding 到相同长度）。这意味着：

  1. 不同 prompt 长度的请求需要 padding 到 batch 内最大长度，浪费计算。
  2. KV Cache 的显存分配仍然是连续的、预分配的，导致显存碎片化。
  3. 调度粒度仍然是"整个 step"，无法在 token 级别做更精细的调度。
    4. 
动态批处理（早期）：
请求1: [prefill][decode][decode][EOS] → 完成，替换为请求4
请求2: [prefill][decode][decode]......[decode][EOS] → 完成
请求3: [prefill][decode][decode]......[decode]......[decode]...
请求4:            [prefill][decode]...[decode][EOS] → 完成

改进明显，但显存管理和调度粒度仍有根本性限制。

2.3 Orca：Iteration-level Scheduling（2022）

2022 年，Gimelshein et al.（来自 NVIDIA）发表 《Orca: A Distributed Serving System for Transformer-Based Generative Models》，这是连续批处理概念的正式提出。

Orca 的核心创新是迭代级调度（Iteration-level Scheduling）：调度的最小单位不再是"一个完整的请求"，而是"一次前向传播（iteration）"。在每一次 iteration 中，调度器可以：

  1. **插入新请求**：将新到达的请求加入当前 batch，从下一个 iteration 开始参与计算。
  2. **移除已完成请求**：生成 EOS 的请求立即从 batch 中移除，释放其占用的资源。
  3. **动态调整 batch 组成**：根据当前 batch 中各请求的状态，动态决定哪些请求参与本次 iteration。

    4. 关键洞见：在 decode 阶段，每个请求每步只生成 1 个 token。既然所有请求每步的工作量相同（都是 1 个 token 的前向传播），那么让不同请求在不同 iteration 中进出 batch 是完全自然的——不需要任何额外的同步开销。

    Orca 的性能提升：

    • 相比静态批处理，吞吐量提升 **2-4 倍**
    • P99 延迟降低 **50-70%**
    • GPU 利用率从 10-30% 提升至 **70-85%**

    但 Orca 仍有局限：它解决了调度问题，但没有解决显存管理问题。KV Cache 仍然是连续分配的，显存碎片化仍然严重。

    2.4 vLLM 与 PagedAttention（2023）

    2023 年，Kwon et al.（UC Berkeley）发表 《Efficient Memory Management for Large Language Model Serving with PagedAttention》，这是连续批处理的显存管理革命。

    vLLM 的核心洞见：KV Cache 的显存管理可以类比操作系统的虚拟内存管理。

    在传统方案中，每个请求的 KV Cache 需要一块连续的显存空间。这导致两个问题：

    1. **内部碎片**：预分配的显存通常大于实际需要的显存（因为生成长度不确定），浪费 60-80% 的 KV Cache 显存。
    2. **外部碎片**：不同请求释放的显存块大小不一，难以复用。

      3. PagedAttention 的解决方案：

      • 将 KV Cache 分成固定大小的**页（page/block）**，每页存储固定数量 token（如 16 个 token）的 KV 值。
      • 每个请求的 KV Cache 不再需要连续显存，而是通过**页表（page table）**映射到分散的物理页。
      • 按需分配：只有当请求实际生成了新 token，需要新的 KV Cache 空间时，才分配新页。
      • 页可以跨请求共享：对于具有相同前缀的请求（如使用相同 system prompt），可以共享前缀对应的物理页。
        • 
传统 KV Cache 分配：
请求1: [连续KV Cache ████████████████] ← 预分配200 token，实际使用45 token
请求2: [连续KV Cache ████████████████████████████] ← 预分配300 token，实际使用120 token
显存浪费: 60-80%

PagedAttention：
请求1: [页3][页7][页12] → 按需分配，无浪费
请求2: [页1][页5][页9][页14][页18] → 分散存储，页表映射
请求3: [页3][页7][页20] → 与请求1共享前缀页（页3、页7）

    vLLM + Orca 调度的组合效果：

    • 相比静态批处理，吞吐量提升 **4-24 倍**（取决于模型大小和请求模式）
    • 显存浪费从 60-80% 降低到 **<5%**
    • 支持更大的 batch size（因为显存利用更高效）
    • 前缀共享进一步减少了重复计算

      • 2.5 工业级部署与生态成熟（2024-2026）

2024-2025 年，连续批处理成为所有主流推理引擎的标配：

推理引擎 连续批处理实现 特色功能
vLLM PagedAttention + 迭代级调度 前缀缓存、自动前缀匹配
TensorRT-LLM in-flight batching + Paged KV Cache NVIDIA 硬件级优化、FP8 支持
SGLang RadixAttention 基于 Radix Tree 的 KV Cache 管理
DeepSpeed-FastGen Dynamic SplitFuse prefill 分块 + decode 混合调度
TGI (HuggingFace) Continuous batching 生态集成、模型兼容性

2025 年的关键进展：

  1. **Prefill-Decode 分离调度（PD Separation）**：将 prefill 阶段和 decode 阶段分配到不同的 GPU 集群。Prefill 是计算密集型（处理长 prompt），decode 是内存带宽密集型（逐 token 生成）。分离后可以分别优化。
  2. **Chunked Prefill**：将长 prompt 的 prefill 分成多个 chunk，每个 chunk 与 decode 请求混合调度。避免了长 prompt prefill 阻塞 decode 请求的问题。
  3. **多模型连续批处理**：在同一 GPU 上同时服务多个模型（如 MoE 的不同 expert），实现跨模型的 batch 复用。

    4. 2026 年的现状：

    连续批处理已经成为推理引擎的"操作系统级"基础设施。用户无需理解其内部机制，只需设置 max_batch_size 参数即可享受其加速效果。研究前沿已经转向更精细的调度策略：

    • 基于优先级和 SLO（Service Level Objective）的调度
    • 基于请求预测的智能批组合
    • 异构 GPU 集群的自适应调度

      • 三、GPU 运行特性：为什么 Batch 解码几乎不增加延迟

      3.1 Compute-Bound vs Memory-Bound

理解连续批处理的前提是理解 GPU 推理的瓶颈特性。

GPU 在执行一次前向传播时，有两个核心操作：

  1. **内存搬运（Memory Access）**：将模型权重从 HBM（高带宽显存）读取到 SRAM（片上缓存/计算核心）。
  2. **浮点计算（Computation）**：在计算核心上执行矩阵乘法（GEMM）。

    3. 关键参数：

    • HBM 带宽：NVIDIA A100 = 2 TB/s，H100 = 3.35 TB/s
    • HBM 容量：A100 = 80 GB，H100 = 80 GB
    • SRAM 带宽：A100 = ~12 TB/s（L2），H100 = ~12 TB/s
    • 计算能力：A100 = 312 TFLOPS (FP16)，H100 = 989 TFLOPS (FP16)

    自回归解码的瓶颈分析：

    在 decode 阶段，每一步只处理 1 个 token。对于一个 7B 参数的模型（FP16），模型权重约 14 GB。每一步前向传播需要：

    
内存搬运量 = 模型权重大小 = 14 GB
计算量 = 2 × 参数量 × 1 = 2 × 7B × 1 = 14 GFLOPs

内存搬运时间 = 14 GB / 2000 GB/s = 7 ms
计算时间 = 14 GFLOPS / 312 TFLOPS = 0.045 ms

GPU 利用率 = 计算时间 / (计算时间 + 内存搬运时间)
           = 0.045 / (0.045 + 7)
           = 0.64%

    结论：自回归解码时，GPU 的算力利用率不到 1%。99% 以上的时间花在等待显存搬运上。

    3.2 为什么 Batch 解码几乎不增加延迟

    当 batch size 从 1 增加到 N 时：

    
内存搬运量 = 模型权重大小 = 14 GB（不变！）
计算量 = 2 × 参数量 × N = 2 × 7B × N GFLOPs

内存搬运时间 = 14 GB / 2000 GB/s = 7 ms（不变！）
计算时间 = 2 × 7B × N / 312 TFLOPS = 0.045 × N ms

当 N = 10:
  总时间 = 7 + 0.45 = 7.45 ms（vs 7 ms，仅增加 6.4%）
  GPU 利用率 = 0.45 / 7.45 = 6%

当 N = 100:
  总时间 = 7 + 4.5 = 11.5 ms（vs 7 ms，增加 64%）
  GPU 利用率 = 4.5 / 11.5 = 39%

当 N = 200:
  总时间 = 7 + 9 = 16 ms（vs 7 ms，增加 129%）
  GPU 利用率 = 9 / 16 = 56%

    核心洞见：

    • 当 batch size 较小时（N < 50），增加 batch size 几乎不增加延迟，因为瓶颈仍在内存搬运。
    • 当 batch size 增大到一定程度后，计算量开始成为瓶颈，延迟线性增长。
    • **最优 batch size** 出现在内存搬运时间 ≈ 计算时间时，此时 GPU 利用率达到 ~50%。

      • 3.3 核心公式：GPU 利用率

      
GPU 利用率 = T_compute / (T_compute + T_memory)

其中：
  T_compute = 2 × P × B / FLOPS_peak
  T_memory = M_model / BW_memory

  P = 模型参数量
  B = batch size（token 数量）
  FLOPS_peak = GPU 峰值算力
  M_model = 模型权重大小（字节数）
  BW_memory = 显存带宽

最优 batch size（GPU 利用率 = 50%）：
  B_optimal = M_model × FLOPS_peak / (2 × P × BW_memory)

对于 7B 模型在 A100 上：
  B_optimal = 14 GB × 312 TFLOPS / (2 × 7B × 2 TB/s)
            = 14 × 312 / (2 × 7 × 2) × (10^9 / 10^9)
            ≈ 156

这意味着对于 7B 模型在 A100 上，batch size 约为 156 时 GPU 利用率达到 50%。而大多数在线推理服务的 batch size 远低于此，因此增加 batch size 几乎不增加单步延迟——这正是连续批处理的物理基础。

3.4 Prefill 阶段的差异

需要注意：上述分析针对 decode 阶段。Prefill 阶段（处理完整 prompt）的特性不同：


Prefill 阶段：
  内存搬运量 = 模型权重 = 14 GB
  计算量 = 2 × P × seq_len

当 seq_len = 1000:
  计算量 = 2 × 7B × 1000 = 14 TFLOPs
  计算时间 = 14 TFLOPS / 312 TFLOPS = 45 ms
  内存搬运时间 = 7 ms
  GPU 利用率 = 45 / (45 + 7) = 86.5%

Prefill 阶段是计算密集型（compute-bound）的，GPU 利用率很高。这就是为什么 2025 年的 PD 分离架构将 prefill 和 decode 分开处理——它们的瓶颈特性完全不同。

四、横向分析：与其他推理加速技术的关系

4.1 连续批处理 vs 推测解码

连续批处理和推测解码是互补关系，而非替代关系。

维度 连续批处理 推测解码
优化目标 提高 GPU 利用率（多请求共享） 减少解码步数（多 token 并行验证）
作用层面 请求级调度 算法级优化
依赖条件 多个并发请求 草稿模型 + 高接受率
加速效果 吞吐量 2-24x 单请求延迟 2-5x
是否可叠加 可以同时使用 可以同时使用

协同方式：推测解码在连续批处理的框架内运行。每个请求的 decode 步骤中，草稿模型生成多个候选 token，目标模型一次性验证。连续批处理确保 GPU 始终有足够的请求来填满 batch，推测解码确保每个请求的每个 decode 步骤能处理更多 token。

4.2 连续批处理 vs 量化

量化（Quantization）通过降低模型参数的精度（FP16 → INT8 → INT4）来减少模型大小和内存搬运量。

关系：

  • 量化**减小**了 M_model，从而**减少**了 T_memory。
  • 连续批处理通过增大 B 来**增加** T_compute，使其接近 T_memory。
  • 两者协同：量化后 T_memory 降低，最优 batch size 也随之降低，但连续批处理仍然可以在更小的 batch size 下实现高 GPU 利用率。
    • 
量化前（FP16）：M_model = 14 GB, T_memory = 7 ms
量化后（INT8）：M_model = 7 GB, T_memory = 3.5 ms
量化后（INT4）：M_model = 3.5 GB, T_memory = 1.75 ms

INT4 量化后，最优 batch size 从 ~156 降至 ~39
但 GPU 利用率的提升曲线形状不变——连续批处理仍然有效

    4.3 连续批处理 vs 模型并行

    模型并行（Tensor Parallelism, Pipeline Parallelism）将大模型分布到多个 GPU 上。

    关系：

    • 模型并行是连续批处理的**前提条件**——对于超过单 GPU 显存容量的模型，必须先做模型并行，才能进行推理。
    • 张量并行（TP）在每个 transformer 层内部切分矩阵乘法，需要 AllReduce 通信。TP 增加了通信开销，但不改变连续批处理的基本原理。
    • 流水线并行（PP）将不同层分配到不同 GPU，引入流水线气泡。PP 与连续批处理的交互更复杂——需要确保不同 stage 的 batch 组成一致。

      • 4.4 连续批处理 vs FlashAttention

      FlashAttention 通过 IO 感知的注意力计算（减少 HBM 访问次数）来加速注意力层。

      关系：

      • FlashAttention 降低了注意力计算的 T_memory，从而改变了整体的 compute-bound vs memory-bound 平衡。
      • 对于长序列（prefill 阶段），FlashAttention 的收益最大——注意力的 O(n^2) 计算量中，内存搬运的占比显著降低。
      • 对于短序列（decode 阶段），FlashAttention 的收益较小——注意力计算本身不是瓶颈。
      • 连续批处理和 FlashAttention 是**正交优化**：FlashAttention 优化单次前向传播的效率，连续批处理优化多次前向传播的调度。

        • 五、2026 年现状与未来趋势

        5.1 当前产业格局

        截至 2026 年中，连续批处理已成为所有主流推理引擎的标配。关键数据：

        • vLLM：GitHub 25k+ stars，被 OpenAI、Anthropic、 Together AI 等采用
        • TensorRT-LLM：NVIDIA 官方推理引擎，深度优化 H100/H200 特性
        • SGLang：Stanford 团队开发，RadixAttention 实现高效前缀缓存

          • 5.2 前沿方向

        • **Disaggregated Serving（分离式服务）**：Prefill 和 Decode 分别在不同的 GPU 集群上执行，通过高速网络传输 KV Cache。代表工作：Splitwise、DistServe。
        • **KV Cache 压缩**：通过量化、剪枝、蒸馏等手段压缩 KV Cache 大小，允许更大的 batch size。
        • **异构调度**：在混合 GPU 集群（如 A100 + H100 + L40S）中做自适应调度。

        4. 推测解码 + 连续批处理的联合优化：在连续批处理框架内智能调度推测解码的草稿/验证阶段。

        5.3 总结

        连续批处理的核心洞见——"GPU 推理的瓶颈是内存带宽而非算力，因此 batch 解码几乎不增加延迟"——已经成为理解所有推理加速技术的基础。无论是推测解码、量化、模型并行还是 FlashAttention，其价值和效果都需要在连续批处理的框架下理解。

        正如操作系统通过虚拟内存和进程调度让 CPU 资源被充分利用，连续批处理通过 PagedAttention 和迭代级调度让 GPU 资源被充分利用。它是推理时代的"操作系统"——看不见，但不可或缺。
