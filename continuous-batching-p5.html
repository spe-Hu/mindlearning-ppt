  • 维度一：历史 (History / Diachronic)

连续批处理从何而来？

它的历史是一部"GPU 利用率提升史"。2017 年 Transformer 诞生时，推理不是问题——模型小、用户少、GPU 够。2018 年 BERT/GPT-2 的推理延迟在可接受范围内。但 2020 年 GPT-3（1750 亿参数）改变了游戏规则——单次推理需要读取 350 GB 的模型权重，GPU 99% 的时间在等显存搬运。

关键节点：

  • 2017：Transformer 诞生，自回归解码成为主流
  • 2019：NVIDIA FasterTransformer 引入动态批处理雏形
  • 2022：Orca 正式提出"迭代级调度"——连续批处理的概念诞生
  • 2023：vLLM/PagedAttention 解决了显存管理问题，连续批处理走向实用
  • 2024：FastGen/SplitFuse 解决 prefill 阻塞，SGLang/RadixAttention 解决前缀匹配
  • 2025：PD 分离架构将连续批处理从单 GPU 扩展到集群级
  • 2026：连续批处理成为推理引擎标配，前沿转向异构调度和多模型共部署

历史规律：每一代都解决上一代遗留的"最后一个不连续的环节"。Orca 让 decode 连续了，但 prefill 还不连续；FastGen 让 prefill 也连续了，但前缀匹配还不智能；SGLang 让前缀匹配也智能了。这条演进链的方向是"彻底的连续性"——消除一切"等待"和"碎片"。

历史的讽刺：连续批处理的核心洞见（"搬一次也是搬，搬十次也是搬"）在 2017 年就已经成立——只要自回归解码是内存带宽受限的，batch 处理就几乎不增加延迟。但直到 2022 年才有人将它形式化为一个调度系统。为什么？因为 2017-2021 年模型还不够大，推理延迟问题不够痛。技术突破往往不是"新洞见"，而是"旧洞见 + 新痛点"。

  • 维度二：辩证 (Dialectics)

连续批处理的对立面是什么？

对立面是"确定性"（Determinism）。连续批处理追求"效率"——让 GPU 永远不闲置。但效率的反面是"可预测性"——每个请求的延迟取决于当前 batch 中有多少其他请求、它们的生成长度、是否有 prefill 在进行。连续批处理的延迟是"动态的"、"不可预测的"。

辩证张力：

  • 效率 vs 确定性：连续批处理最大化吞吐量，但引入了延迟波动
  • 共享 vs 隔离：多个请求共享 GPU 资源，但一个请求的异常可能影响其他请求
  • 动态 vs 静态：动态调度灵活高效，但实现复杂度高
  • 连续 vs 原子：连续批处理将推理"流式化"，但某些场景需要"原子性"（如 beam search 的完整候选集）

辩证综合：

  • Chunked Prefill：在效率（prefill 与 decode 混合）和确定性（decode 延迟不超标）之间找到平衡
  • SLO-aware Scheduling：在吞吐量（尽量多塞请求）和延迟保证（满足 SLO 约束）之间找到平衡
  • PD 分离：在资源共享（单 GPU 混合）和资源隔离（不同 GPU 不同阶段）之间找到平衡

更深层的辩证：连续批处理的本质是"用统计复用（statistical multiplexing）换取效率"——多个请求共享 GPU，就像多个用户共享网络带宽。统计复用的前提是"请求的波动性可以互相抵消"——有些请求快完成，有些刚开始，平均下来 GPU 利用率稳定。但如果所有请求同时到达、同时需要长 prefill，统计复用就失效了。这就是"效率"和"鲁棒性"的永恒辩证。

  • 维度三：现象 (Phenomenology)

连续批处理在"显现"什么？

当你打开 ChatGPT 输入一个问题，看到文字一个接一个地出现——这个"流式输出"的体验背后，就是连续批处理在工作。你看到的每一个 token，都是 GPU 在一次前向传播中"顺带"计算出来的。但 GPU 不只为你一个用户服务——它同时在为几十上百个用户生成 token。

现象学描述：

  • 用户视角：一个 token 接一个 token 地出现，流畅、连续
  • GPU 视角：一次前向传播处理几百个 token，来自不同用户、不同请求、不同位置
  • 调度器视角：一个不断变化的"batch 组合"——每毫秒都有请求进出
  • 显存视角：一页页 KV Cache 被分配、被共享、被释放，像城市的停车位一样不断周转

连续批处理"显现"的是一种"看不见的共享"。用户感知到的是"专属的 AI 助手"，实际发生的是"几百个用户共享同一块 GPU"。这种"共享的不可见性"是连续批处理成功的标志——就像操作系统的进程调度一样，你感觉不到你的 CPU 在几千个进程之间切换，但事实就是如此。

更深层的现象：连续批处理让"推理"从"一次性事件"变成了"持续流动"。传统推理是"接收请求 → 完整计算 → 返回结果"的离散过程。连续批处理将推理变成了"持续接收 → 持续计算 → 持续返回"的流式过程。这改变了推理的"时间性"——不再是"一个请求的时间"，而是"所有请求的叠加时间"。

  • 维度四：语言 (Linguistics)

"连续批处理"这个名字说了什么？又隐藏了什么？

"连续"（Continuous）：强调"不中断"。与"静态"（Static）对立——静态批处理是"一批处理完才开始下一批"，有"中断"（等待当前 batch 完成）。连续批处理"不中断"——新请求随时插入，完成请求立即释放。

"批处理"（Batching）：强调"批量"。与"逐个处理"（One-by-one）对立。批处理暗示"效率来自规模"——一次处理多个比逐个处理更高效。

名字隐藏了什么？

  • "连续批处理"没有暗示"显存管理"这个核心维度。PagedAttention 的贡献不体现在名字中。
  • "连续"暗示了"时间上的连续"，但没有暗示"空间上的分页"——KV Cache 的不连续分配恰恰是连续批处理的关键支撑。
  • "批处理"是一个古老的计算术语（batch processing from the 1960s），暗示了"批量提交、批量处理"的旧范式。但连续批处理实际上打破了传统批处理的"批量"概念——它不是"凑满一批再处理"，而是"有什么处理什么"。

名字的隐喻：

  • 高速公路隐喻：GPU 是高速公路，模型权重搬运是"修路"，token 是"车"。修路的成本固定，车越多越划算。连续批处理就是"不让高速公路空着"。
  • 餐厅隐喻：GPU 是厨房，模型权重是"菜谱"（每次做菜都要看），token 是"菜"。厨师看一次菜谱可以同时做几道菜——"看一次也是看，看十次也是看"。
  • 电梯隐喻：电梯上下一次的成本固定，载 1 人和载 10 人的成本几乎相同。连续批处理就是"不让电梯空跑"。
  • 维度五：形式 (Formalization)

连续批处理的数学结构是什么？

核心优化问题：

#+BEGIN_EXAMPLE

max Throughput = Σ tokens_generated / total_time

s.t.

∀ iteration i:

batch_i ⊆ active_requests

batch_i

Σ KV_cache(request_j) ≤ GPU_memory - model_size

latency(request_j) ≤ SLO_max

∀ request j:

arrival_time(j) ≤ start_time(j)

completion_time(j) - start_time(j) ≤ SLO_total(j)

#+END_EXAMPLE

GPU 利用率的形式化：

#+BEGIN_EXAMPLE

η(B) = T_compute(B) / (T_compute(B) + T_memory)

T_compute(B) = 2 × P × B / FLOPS_peak [线性增长]

T_memory = M_model / BW_memory [常数]

η(B) = (2PB/F) / (2PB/F + M/BW)

= 1 / (1 + MF / (2PB × BW))

当 B → 0: η → 0 (memory-bound)

当 B → ∞: η → 1 (compute-bound)

当 B = MF / (2P × BW): η = 0.5

dη/dB = (MF / (2P × BW)) / (B + MF / (2P × BW))^2

> 0 for all B > 0

即：GPU 利用率关于 batch size 单调递增，但增速递减（凹函数）

#+END_EXAMPLE

调度策略的形式化：

#+BEGIN_EXAMPLE

静态批处理: batch(t) = batch(t+1) = ... = batch(t+T) [T = max completion time]

动态批处理: batch(t+1) = (batch(t) \ {completed}) ∪ {new_requests}

连续批处理: batch(t+1) = schedule(batch(t), queue, memory_state, SLO_constraints)

其中 schedule() 是一个决策函数，输入：

  • batch(t): 当前 batch 组成
  • queue: 等待队列
  • memory_state: 显存使用状态（块表）
  • SLO_constraints: 延迟约束

最优调度是一个在线优化问题（online optimization），因为未来的请求到达是未知的。

#+END_EXAMPLE

PagedAttention 的形式化：

#+BEGIN_EXAMPLE

令 KV_cache(r) 为请求 r 的 KV Cache

传统方案: KV_cache(r) = contiguous_memory_block(size=seq_len(r) × kv_per_token)

PagedAttention: KV_cache(r) = {page_i : i ∈ block_table(r)}

block_table(r): logical_block → physical_block

physical_block: fixed_size (e.g., 16 tokens)

约束:

∀ r: |KV_cache(r)| = seq_len(r) × kv_per_token [完整性]

∀ r: block_table(r) 是单射 [每个逻辑块映射到唯一物理块]

可选: block_table(r1) ∩ block_table(r2) ≠ ∅ [前缀共享]

#+END_EXAMPLE

  • 维度六：存在 (Existentialism)

连续批处理改变了什么"存在方式"？

它改变了"推理"的存在方式。在连续批处理之前，推理是"离散的"——一个请求一个请求地处理，每个请求有自己的"开始"和"结束"。连续批处理之后，推理是"连续的"——没有明确的"开始"和"结束"，只有永恒的"流动"。

对开发者而言：连续批处理将"推理优化"从"手工艺术"变成了"工程基础设施"。以前优化推理需要理解 GPU 架构、手写 CUDA kernel、精心调参。现在只需 pip install vllm 然后设置 max_batch_size=256。这种"去技能化"（deskilling）让开发者可以专注于模型和业务，但也让他们对底层越来越无知。

对用户而言：连续批处理让"等待"变得可接受。没有连续批处理，ChatGPT 的响应延迟会高得不可接受（因为 GPU 资源不够）。有了连续批处理，即使同时有百万用户在用，每个用户的体验仍然是"流畅的"。连续批处理"隐藏"了规模——它让你感觉不到"百万用户共享同一块 GPU"的事实。

对 AI 产业而言：连续批处理改变了"推理成本"的经济学。没有连续批处理，推理成本与用户数量线性相关（每个用户需要专属 GPU）。有了连续批处理，推理成本与用户数量的关系是亚线性的（多个用户共享 GPU）——这意味着 AI 服务可以"规模化"而不"破产"。

  • 维度七：美学 (Aesthetics)

连续批处理的"美"在哪里？

结构之美：PagedAttention 将操作系统的虚拟内存管理"移植"到 GPU 显存管理。同样的分页、页表、写时复制——不同的领域，相同的结构。这种"跨领域的结构同构"是数学美的体现。

效率之美：一个 7B 模型在 A100 上，batch_size=1 时 GPU 利用率 0.64%，batch_size=156 时利用率 50%。从 0.64% 到 50%——同样的 GPU，同样的模型，只是改变了"调度方式"。这种"不增加任何硬件成本就提升 78 倍利用率"的效果，是工程美的体现。

演进之美：从 Orca 到 vLLM 到 FastGen 到 SGLang，每一代都精确地解决上一代遗留的"最后一个问题"。这种"问题 → 解决 → 新问题 → 新解决"的递进式演进，有一种"自然生长"的有机美感。

简洁之美：连续批处理的核心洞见极其简洁——"搬一次也是搬，搬十次也是搬"。十个字，概括了整个技术的物理基础。所有复杂的工程实现（页表、调度器、块分配、前缀缓存）都是这十个字的"展开"。

  • 维度八：元哲学 (Meta-philosophy)

连续批处理让我们思考什么更深层的问题？

"闲置"的本质：GPU 99% 的时间在闲置。连续批处理问的是："闲置"是一种浪费还是一种必然？从物理角度看，闲置是"产能过剩"——计算能力远超当前任务的需要。从经济角度看，闲置是"沉没成本"——GPU 已经买了，闲着也是闲着。连续批处理的哲学立场是："闲置是一种可以被消除的浪费"——通过更聪明的调度。

"共享"的悖论：连续批处理让多个请求共享 GPU。但"共享"意味着"互相影响"——一个请求的延迟取决于其他请求的行为。这在经济学中被称为"公地悲剧"（Tragedy of the Commons）——共享资源被过度使用。连续批处理通过"调度器"（类似"政府"）来管理共享资源，避免公地悲剧。

"连续"的代价：连续批处理追求"连续性"——消除一切中断和等待。但"连续"的代价是"复杂性"——调度器需要管理页表、处理请求进出、保证 SLO。这反映了一个普遍的哲学命题："连续性"和"简单性"不可兼得。你想要更高的效率，就必须接受更高的复杂性。

"搬运"的隐喻：连续批处理的核心隐喻是"搬运"——GPU 的主要工作是"搬运"模型权重，而不是"计算"。这颠覆了我们对"计算"的理解——我们以为 GPU 在"思考"，实际上它在"搬砖"。这个隐喻的深层含义是：在很多系统中，"搬运"（数据传输、资源调度）比"生产"（计算、生成）更重要。

  • 压缩 (Epiphany)

八维解剖后，连续批处理的精神内核是：

"搬一次也是搬，搬十次也是搬" —— 在固定成本（内存搬运）主导的系统中，边际成本（增加 batch size）趋近于零，因此最大化边际利用是最优策略。

这个洞见适用于一切"固定成本主导 + 边际成本低"的系统：高速公路（修路成本高，多一辆车成本为零）、餐厅厨房（厨师看一次菜谱可以同时做几道菜）、电梯（上下一次的成本固定，载多人成本几乎相同）。

连续批处理的技术价值在于将这个朴素的洞见形式化为一套完整的调度系统——包括迭代级调度、分页显存管理、智能前缀缓存、PD 分离。它的哲学价值在于提醒我们：在很多系统中，"搬运"比"生产"更重要，"调度"比"堆砌"更有效，"共享"比"独占"更经济。
