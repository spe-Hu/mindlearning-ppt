  • 1 核心问题 + 挠痒处

推测解码在答一个问题：如何打破自回归解码的速度瓶颈？更直白地说——GPU 在推理时 99% 的时间都在闲置，怎么把这些浪费的算力利用起来？

挠痒处：2024-2026 年，大模型推理加速成为产业焦点。训练成本可以通过 MoE、混合精度等手段控制，但推理成本几乎与请求量成正比——用户越多，GPU 越不够用。推测解码的诱惑在于它承诺了一种"免费午餐"：不牺牲任何输出质量，纯粹通过更聪明的调度来加速。但"免费"二字总是可疑的。草稿模型要占内存，验证过程有开销，接受率不是 100%——真正的"免费"在哪里？

答案藏在 GPU 的架构特性里。自回归解码时，每一步只处理一个 token，但需要把整个模型权重（几十 GB）从 HBM 读到计算单元。这个内存搬运操作占据了 99% 的时间，而实际的矩阵乘法只用了不到 1% 的时间。推测解码的洞见是：既然内存搬运的时间已经付出了（读模型权重），那搬运期间闲置的计算单元不用白不用——让它同时验证多个候选 token，反正不额外增加内存搬运。

这就是"内存带宽换算力"的真正含义。不是真的用带宽换算力，而是利用已经付出的带宽成本中未被利用的算力部分。前提是：草稿模型的预测要足够准。如果接受率太低，验证的开销反而拖慢了速度。

从 2018 年 Stern 的块级并行解码到 2022 年 Leviathan 的形式化，到 2024 年 Medusa 的自草稿、EAGLE 的特征空间推测，到 2025 年 DeepSeek DSpark 的工业级部署——推测解码走过了一条从学术构想到产业标配的道路。DSpark 的突破在于把推测解码从"推理时的后处理"提升为"训练时的深度优化"——MTP 模块让模型在训练时就学会预测多个未来 token，从根本上提高了推测的质量。

  • 2 基础假设

推测解码立在三条不证之物上。

第一条：小模型预测 + 大模型验证的接受率足够高。这是整条推测解码的生命线。如果草稿模型生成的 token 和目标模型会生成的 token 完全不一致，推测解码就退化成了"先随机生成一堆 token 再让目标模型全部拒绝"——不仅没加速，反而更慢。接受率取决于草稿模型和目标模型之间的"相似度"——两个模型越像，接受率越高。这就是为什么 Medusa 用目标模型自身的解码头做草稿（特征分布完全一致），EAGLE 在特征空间做推测（特征空间比 token 空间更平滑、更可预测）。接受率的理论上限是 100%（草稿模型完美预测），实际中 70-90% 是合理范围。

第二条：目标模型可以一次性并行验证多个 token，且验证 K 个 token 的成本约等于生成 1 个 token 的成本。这是推测解码能加速的硬件前提。Transformer 的自回归解码之所以慢，不是因为计算量大，而是因为内存搬运量大。验证 K 个 token 需要读取的模型权重与生成 1 个 token 相同（都是整个模型），只是计算量从"1 个 token 的矩阵乘法"增加到"K 个 token 的矩阵乘法"——但计算量的增加几乎不增加时间，因为瓶颈在内存搬运而非计算。这个假设在 batch size = 1 时几乎完美成立。在 batch size 较大时，计算量开始成为瓶颈，推测解码的加速比会下降。

第三条：接受-拒绝采样保证输出分布不变。这是推测解码"无损"的数学基础。接受准则 min(1, p_target(x) / p_draft(x)) 确保了每个 token 被接受的概率恰好补偿了草稿模型的偏差。数学上可以证明，经过足够多次的接受-拒绝采样后，推测解码生成的 token 序列的分布与目标模型直接生成的分布完全一致。这不是近似，是精确的数学等价。但注意：这个证明依赖于一个隐含假设——草稿模型的概率 p_draft(x) 在所有位置都是正的（即草稿模型不会给目标模型认为可能的 token 分配零概率）。实际中，这通过给草稿分布加一个小常数 epsilon 来保证。

还有一条隐含的：GPU 的内存带宽是固定成本。无论你是否做推测解码，读取模型权重的时间都是一样的。推测解码不增加内存搬运量，只是利用搬运期间闲置的算力。这个假设在当前 GPU 架构（HBM 带宽有限但计算单元充裕）下成立。但如果未来 GPU 的内存带宽大幅提升（如 CXL 内存池化），推测解码的前提条件可能不再成立——因为内存搬运不再是瓶颈，计算本身成为瓶颈。

  • 3 分析框架

推测解码的取景框是"草稿-验证"博弈：草稿模型负责快速猜测，目标模型负责精确验证。

框架里有几件独占的东西：

接受率（acceptance rate）——推测解码的核心指标。接受率 alpha 决定了理论加速比 = 1/(1-alpha)。alpha=0.5 时加速 2x，alpha=0.8 时加速 5x，alpha=0.9 时加速 10x。接受率取决于三个因素：(1) 草稿模型的质量——越接近目标模型越好；(2) 草稿长度 K——K 越大，后续 token 的预测越不准确，接受率越低；(3) 输入文本的特性——低熵文本（如自然语言续写）接受率高，高熵文本（如代码生成、数学推理）接受率低。

草稿长度（draft length）——每次推测生成的候选 token 数量。K 太小，加速效果不明显（每次只多验证几个 token）；K 太大，后续 token 的预测不准确，接受率下降，浪费验证计算。最优 K 取决于接受率：K ≈ -1/ln(alpha)。当 alpha=0.8 时，K ≈ 5；当 alpha=0.5 时，K* ≈ 1。EAGLE 的动态草稿长度机制就是根据实时接受率自动调整 K。

草稿模型选择——推测解码的关键设计决策。三种路线： 1. 独立小模型（Leviathan 2022）：用同系列的小模型（如 LLaMA-7B 作为 LLaMA-70B 的草稿）。优点：简单直接。缺点：额外内存开销，特征分布不完全匹配。 2. 自草稿（Medusa 2024）：在目标模型上添加解码头。优点：无额外内存，特征完全匹配。缺点：解码头容量有限，预测能力受限。 3. 特征空间推测（EAGLE 2024）：在隐藏层特征空间做自回归预测。优点：特征空间更平滑，预测更准确。缺点：需要额外训练特征预测模型。

DSpark/MTP 代表了第四种路线：训练时集成。在预训练阶段就加入多 token 预测目标，使 MTP 模块成为模型的有机组成部分。这不是推理时的"外挂"，而是训练时的"内建"。

树状验证 vs 线性验证——验证策略的选择。线性验证（Leviathan）按顺序验证 K 个 token，遇到第一个拒绝就停止。树状验证（Medusa, EAGLE）同时生成多条候选路径，用树状注意力一次性验证所有路径。树状验证的优势：即使某条路径在早期被拒绝，其他路径可能继续被接受。这显著提高了"至少接受 K 个 token"的概率。

  • 4 核心观点

推测解码是 GPU 内存墙时代的必然产物。只要自回归解码仍然是内存带宽受限的操作，推测解码就有存在的理由。它的核心价值不在于某种具体的算法实现，而在于一个更深层的洞见：自回归解码中存在大量未被利用的算力，这些算力可以用来"投机"——用廉价的猜测换昂贵的验证。

从 Leviathan 到 DSpark，推测解码经历了三次范式转换： 1. 独立草稿 → 自草稿（Medusa）：消除了草稿模型的额外开销 2. Token 空间 → 特征空间（EAGLE）：提高了草稿的预测质量 3. 推理时优化 → 训练时优化（DSpark/MTP）：从根本上提升了推测能力

每次范式转换都带来了显著的加速提升：从 2-3x 到 3-4x 再到工业级的 60-85% 速度提升。

推测解码的"无损"特性值得深入思考。在 AI 推理优化的众多方法中（量化、蒸馏、剪枝），推测解码是唯一一个在数学上保证输出分布完全不变的方法。这意味着你可以无脑开启推测解码，不需要担心"加速后模型变笨了"。这种"免费午餐"在工程中极为罕见。

但推测解码也有其边界。当 batch size 较大时，计算量成为瓶颈，推测解码的加速效果减弱。当模型已经在 prefill 阶段（处理长 prompt）而非 decode 阶段时，推测解码不适用。当 GPU 的内存带宽不再是瓶颈时（如未来架构），推测解码的前提条件可能不再成立。

DSpark 的跨模型通用性是一个重要信号。它说明推测解码的价值不仅在于特定模型的优化，而在于一种通用的推理范式。DSpark 在 Qwen3 上也能实现 30% 加速，说明其核心机制（动态调度、内存优化）不依赖于 DeepSeek 的特定架构。

从产业角度看，推测解码正在成为推理引擎的标配。vLLM、TensorRT-LLM、SGLang 都内置了推测解码支持。这意味着即使你不理解推测解码的数学原理，也能享受其加速效果——就像你不需要理解 TCP 协议也能上网一样。

  • 5 精神内核

一个洞见——"推测解码的核心不是'怎么猜得更准'，而是'为什么猜了也不亏'。"

从 2018 年到 2026 年，推测解码的技术路线在不断演化——从独立草稿到自草稿到特征空间推测到训练时集成。但底层逻辑从未改变：GPU 在自回归解码时大量闲置，推测解码利用这些闲置资源来"投机"，即使投机失败（拒绝），损失也很小（验证的开销几乎为零），而投机成功时收益很大（一次验证通过多个 token）。

这是一个不对称的赌局：输了不亏，赢了大赚。而提高接受率的所有技术——更好的草稿模型、特征空间推测、动态草稿长度、训练时集成——都是在增加"赢"的概率。

带走了这一件，推测解码的精神内核就在手：在资源闲置的系统中，投机是理性的——只要投机的成本足够低，而成功的收益足够高。

这个洞见超越了自动驾驶。在任何"安全关键 + 实时约束"的系统中（航空航天、医疗设备、工业控制），推测思维都需要与安全约束相结合。推测解码在自动驾驶中的实践，为所有安全关键系统的推理优化提供了范本。

  • 6 延伸思考：推测解码在自动驾驶中的未来

推测解码在自动驾驶中的应用仍处于早期阶段。几个值得关注的方向：

第一，多模态推测解码。当前的推测解码主要处理单一模态（文本或驾驶决策）。未来的自动驾驶推测解码需要同时处理多种模态——图像、点云、雷达信号、驾驶决策——的推测和验证。跨模态的推测解码是一个全新的技术挑战。

第二，推测解码与强化学习的结合。自动驾驶的决策不仅是"预测下一个 token"，还需要考虑长期奖励（安全性、效率、舒适性的综合优化）。推测解码可以与强化学习结合——用推测解码加速策略评估（value function 的推测计算），用强化学习的奖励信号指导推测策略。

第三，车-云协同推测。车端用轻量级 MTP 做快速推测，云端用大模型做验证和修正。这种协同架构可以在车端资源受限的情况下，仍然获得高质量的推测加速。5G/V2X 的低延迟通信使得车-云协同推测成为可能。

第四，推测解码在仿真测试中的应用。自动驾驶的仿真测试需要生成大量的驾驶场景。推测解码可以加速场景生成——用轻量级场景模型生成场景草稿，用复杂物理引擎验证和修正。这可以大幅提高仿真测试的效率。

从更宏观的视角看，推测解码在自动驾驶中的应用代表了"安全关键 AI"的一个新范式：不是"更快的模型"，而是"更聪明的调度"。通过利用 GPU 的闲置算力，推测解码在不增加硬件成本的情况下实现了推理加速。这种"免费午餐"对于成本敏感的汽车行业尤为重要。
