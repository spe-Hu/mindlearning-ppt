推测解码 (Speculative Decoding) 横纵分析报告

研究时间：2026-06-28 | 所属领域：AI/推理加速 | 研究对象类型：技术概念 关键词：Speculative Decoding, Draft Model, Verification, Memory-Bound, DSpark, MTP

一、一句话定义

推测解码（Speculative Decoding）是大语言模型推理加速的核心技术：用一个轻量级"草稿模型"快速生成多个候选 token，再用目标大模型一次性并行验证这些 token 的正确性，接受正确的、拒绝错误的，从而在数学上保证输出分布与原始模型完全一致的前提下，实现 2-3 倍的推理加速。 其本质是"用廉价的猜测换昂贵的验证"——将自回归解码中 GPU 高达 99% 的空置算力利用起来，用内存带宽换取吞吐量。从 2018 年 Blockwise Parallel Decoding 的思想萌芽，到 2022 年 Leviathan 等人的形式化提出，到 2024 年 EAGLE 的自适应草稿，到 2025 年 DeepSeek DSpark 的工业级部署，推测解码已从学术构想演变为产业级推理加速的基石。

二、纵向分析：从并行解码到工业级部署（2018-2026）

2.1 思想萌芽：块级并行解码（2018）

推测解码的思想根源可以追溯到 2018 年 Google 的论文 《Blockwise Parallel Decoding for Deep Autoregressive Models》（Stern et al., NeurIPS 2018）。

这篇论文的核心洞见是：自回归解码不一定非要一步一步来。如果能把序列分成若干"块"（block），每个块内并行生成多个候选 token，然后用某种机制验证哪个块是正确的，就能实现并行加速。

具体方法：训练一个浅层的"辅助"模型（jam model），用它来预测未来若干步的 token 序列。然后让目标模型一次性验证整个序列。如果辅助模型的预测足够准确，就能大幅减少目标模型的前向传播次数。

然而，2018 年的技术条件限制了这一思想的传播。当时的模型规模还不够大，推理延迟问题还不突出。GPT-2（15 亿参数，2019 年）的推理延迟在可接受范围内，学术界和工业界的主要精力还集中在"如何让模型更强"而非"如何让模型更快"。

2.2 形式化提出：Speculative Decoding（2022-2023）

2022 年，Leviathan et al. 发表 《Fast Inference from Transformers via Speculative Decoding》，正式将这一技术命名为"Speculative Decoding"。

核心贡献： 1. 理论框架：将推理加速问题形式化为"草稿生成-验证"的两阶段框架。草稿模型（draft model）自回归生成 K 个候选 token，目标模型（target model）一次性前向传播验证所有 K 个 token。 2. 无损保证：通过精心设计的接受-拒绝采样（acceptance-rejection sampling），保证推测解码的输出分布与原始目标模型的输出分布完全一致——不是近似，是数学上的精确一致。 3. 加速比公式：理论加速比 = 1 / (1 - alpha)，其中 alpha 是接受率。当接受率为 80% 时，加速比为 5x；当接受率为 50% 时，加速比为 2x。

同期，DeepMind 的 Chen et al. 发表 《Accelerating Large Language Model Decoding with Speculative Sampling》，独立提出了相同的想法，并给出了等价的理论保证。两篇论文的同时出现说明推测解码是技术成熟到一定程度的必然产物。

2.3 Medusa：自草稿时代的开启（2024）

2024 年，Cai et al.（UC Berkeley）发表 《Medusa: Simple LLM Inference Acceleration Framework with Multiple Decoding Heads》。

Medusa 的核心创新是消除独立的草稿模型。它不再训练一个单独的小模型来做草稿，而是在目标模型的最后几层上添加多个额外的解码头（decoding heads），每个头负责预测未来第 k 步的 token。

标准自回归: h → head → token_1 → h' → head → token_2 → h'' → head → token_3
Medusa:     h → head_0 → token_1
              → head_1 → token_2  (并行)
              → head_2 → token_3  (并行)

优势： - 无需维护独立草稿模型：减少了内存开销和模型加载时间 - 特征共享：解码头共享目标模型的隐藏层表示，草稿质量更高 - 训练简单：只需在已有模型上微调额外的解码头，成本极低

Medusa 在 Vicuna-7B 上实现了 2.2x 的加速，在 Vicuna-30B 上实现了 2.5x 的加速。

2.4 EAGLE：自适应推测解码（2024）

2024 年，Han et al. 发表 《EAGLE: Speculative Sampling Requires Rethinking Feature Space》。

EAGLE 的关键洞见是：Medusa 的解码头虽然简单，但在特征空间（feature space）中做预测比在 token 空间（token space）中做预测更有效。

核心创新： 1. 特征级推测：不是在 token 概率分布上做推测，而是在隐藏层特征（hidden states）上做推测。特征空间比 token 空间更连续、更可预测。 2. 动态草稿长度：根据输入的难度动态调整每次推测的 token 数量。简单的文本（如"the cat sat on the"）用长草稿，复杂的文本（如数学证明）用短草稿。 3. 注意力增强：利用目标模型的注意力模式来指导草稿生成，提高接受率。

EAGLE 在 Vicuna-7B 上实现了 3.0x 加速，在 LLaMA-2-Chat 70B 上实现了 2.8x 加速，且保持无损。

2.5 DSpark：工业级推测解码（2025）

2025 年，DeepSeek 发布 DSpark 推理加速框架（梁文锋署名 + 北京大学合作），这是推测解码从学术走向工业级部署的里程碑。

DSpark 的核心创新： 1. Multi-Token Prediction (MTP) 模块：在 DeepSeek-V3 的预训练阶段就加入多 token 预测目标，使模型在训练时就学会"一次看到多个未来"。这不是推理时的权宜之计，而是训练时的深度优化。 2. 跨模型通用性：DSpark 不仅在 DeepSeek-V4-Flash/Pro 上有效（60%-85% 速度提升），在 Qwen3 等第三方模型上也能实现约 30% 的加速，证明了框架的通用性。 3. 内存带宽优化：DSpark 深入分析了 GPU 推理的内存墙问题。自回归解码时，GPU 的算力利用率不到 1%——99% 的时间都在等待内存读写。DSpark 通过批量验证和内存访问优化，将空置的算力利用起来。 4. 动态调度策略：根据序列的上下文复杂度动态调整推测策略——高熵区域（如代码生成、数学推理）使用保守策略，低熵区域（如自然语言续写）使用激进策略。

DSpark 的性能数据： | 模型 | 场景 | 基线速度 | DSpark 速度 | 提升 | |------|------|----------|-------------|------| | DeepSeek-V4-Flash | 通用对话 | 100 tok/s | 185 tok/s | 85% | | DeepSeek-V4-Pro | 代码生成 | 60 tok/s | 99 tok/s | 65% | | Qwen3-32B | 通用对话 | 80 tok/s | 104 tok/s | 30% |

2.6 现状与趋势（2026）

截至 2026 年中，推测解码已经成为大模型推理引擎的标配组件：

  • vLLM 从 v0.5 开始内置推测解码支持
  • TensorRT-LLM 集成了 Medusa 和 EAGLE 风格的推测解码
  • SGLang 实现了基于 jump-forward 的推测解码
  • DeepSeek DSpark 在工业级部署中验证了跨模型通用性

技术趋势： 1. 训练-推理协同设计：MTP 代表了在训练阶段就为推测解码做优化的趋势 2. 自适应策略：从固定草稿长度到动态调整，从静态草稿模型到上下文感知的草稿生成 3. 硬件协同：推测解码与 Flash Attention、PagedAttention 等内存优化技术的深度结合 4. 多模态扩展：推测解码从纯文本扩展到视觉-语言模型、语音模型的推理加速

3.2 推测解码 vs 量化

量化（如 GPTQ、AWQ、SmoothQuant）通过降低权重精度来减少内存占用和计算量。

核心区别： - 量化是有损的——INT4 量化的模型输出与 FP16 原始模型不完全一致 - 推测解码是无损的——输出分布与原始模型数学上完全一致 - 量化减少的是单次前向传播的成本 - 推测解码减少的是前向传播的次数

互补性：两者可以叠加使用。先量化草稿模型和目标模型，再用推测解码加速。实际部署中，量化+推测解码的组合可以实现 5-10x 的综合加速。

3.3 推测解码 vs 蒸馏

蒸馏训练一个小模型来模仿大模型的行为。

核心区别： - 蒸馏后的模型是独立的——能力必然低于原始大模型 - 推测解码不改变目标模型——输出质量完全不变 - 蒸馏适合边缘部署（手机、IoT 设备） - 推测解码适合云端部署（有足够内存运行两个模型）

融合趋势：用蒸馏模型作为推测解码的草稿模型——蒸馏模型既小又"像"大模型，是理想的草稿模型候选。

3.4 推测解码 vs 剪枝

剪枝移除模型中"不重要"的参数或结构。

核心区别： - 剪枝改变模型结构，推测解码不改变 - 剪枝的加速来自减少计算量，推测解码的加速来自减少前向传播次数 - 剪枝可能导致不可逆的质量下降，推测解码完全可逆

3.5 推测解码 vs Flash Attention / PagedAttention

核心区别： - Flash Attention 优化的是单次前向传播中注意力计算的内存访问模式 - PagedAttention 优化的是KV Cache 的内存管理 - 推测解码优化的是前向传播的总次数

三者解决的是不同层面的问题，完全可以组合使用。实际上，现代推理引擎（如 vLLM）通常同时使用这三种优化。

五、产业影响

5.1 对推理服务商的影响

推测解码直接降低了推理成本。对于 API 服务商（如 OpenAI、Anthropic、DeepSeek），推测解码意味着： - 相同的 GPU 资源可以服务更多请求：2x 加速等于有效吞吐量翻倍 - 延迟降低：用户感知的首 token 延迟和生成延迟显著降低 - 成本下降：每百万 token 的推理成本降低 30%-50%

5.2 对模型设计的影响

DSpark/MTP 的出现改变了模型设计的思路： - 训练时就要考虑推理效率：不再是"先训练再优化"，而是"训练时就为高效推理做准备" - 模型架构需要支持多 token 预测：MTP 模块成为模型架构的一部分 - 草稿模型和目标模型的协同设计：不再是独立训练，而是联合优化

5.3 对硬件设计的影响

推测解码揭示了 GPU 推理的核心瓶颈——内存墙： - 自回归解码时，GPU 99% 的时间在等待内存读写 - 算力（TFLOPS）不是瓶颈，内存带宽（GB/s）才是 - 这推动了硬件设计的调整：HBM（高带宽内存）的容量和带宽成为关键指标 - NVIDIA Blackwell 架构的 HBM3e（带宽 8TB/s）正是针对这一瓶颈

5.4 核心洞见：内存带宽换算力

推测解码的本质是一个经济学问题：

GPU 推理的成本模型：
- 算力成本：每 TFLOPS 的价格
- 内存带宽成本：每 GB/s 的价格
- 自回归解码的特征：计算密集度（compute intensity）极低
  - 每步只处理 1 个 token
  - 需要读取整个模型权重（几十 GB）
  - 实际计算量极小（几 MFLOPS）
  - 计算密集度 = 计算量 / 读取数据量 ≈ 几 FLOP / 几十 GB ≈ 极低

结论：自回归解码是内存带宽受限（memory-bandwidth-bound）的操作
推测解码的解法：既然内存带宽已经用了（读模型权重），空置的算力不用白不用
→ 用空置的算力来验证草稿 → 内存带宽没增加，但有效算力提升了

这就是"内存带宽换算力"的深层含义：不是真的用带宽换算力，而是利用已经付出的带宽成本中未被利用的算力部分。这是一种"免费午餐"——前提是草稿模型的预测足够准确。

六、总结

推测解码从 2018 年的学术构想到 2025 年的工业级部署，经历了七年的演化。它的核心价值在于：在不牺牲任何输出质量的前提下，实现 2-5 倍的推理加速。这一加速不是通过近似或压缩实现的，而是通过巧妙地利用 GPU 的空置算力——一种真正的"免费午餐"。

DSpark 的出现标志着推测解码进入新阶段：从"推理时的权宜之计"演变为"训练时的深度优化"。MTP 模块的引入意味着未来的大模型将在训练阶段就为推测解码做准备，实现训练-推理的协同设计。

从更宏观的视角看，推测解码是"推理效率革命"的核心组成部分。与量化、Flash Attention、PagedAttention、MoE 等技术一起，推测解码正在将大模型推理的成本降低一个数量级，使 AI 服务更加普惠。
