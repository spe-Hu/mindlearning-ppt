滑动窗口注意力——不看完所有，只看附近

核心问题：长序列注意力的计算复杂度如何降低？

问题的提出

想象你在一本 1000 页的书中寻找某个信息的关联。全量注意力的做法是：把每一页都与其他所有页进行比较——这需要 1000 × 1000 = 1,000,000 次比较。滑动窗口注意力的做法是：每页只与前后 10 页比较——只需要 1000 × 10 = 10,000 次比较。计算量减少了 100 倍。

但问题是：只看附近，会不会错过远处的关键信息？

问题的本质

长序列注意力的核心矛盾是：信息的全局性与计算效率之间的冲突。

  • 全量注意力：每个位置都能看到所有其他位置 → 信息流通完美，但计算量 O(n^2)
  • 完全不关注：每个位置只看自己 → 计算量 O(n)，但丢失所有上下文
  • 滑动窗口注意力：每个位置看附近 w 个位置 → 计算量 O(n × w)，是两者的折中

关键洞见是：在大多数自然序列中，局部信息比全局信息更重要。一个词的含义主要由其上下文（前后几个词）决定，而不是由文档中距离很远的内容决定。

基础假设：局部上下文比全局上下文更重要

假设的内容

滑动窗口注意力的有效性建立在一个关键假设之上：在自然序列（文本、图像、音频等）中，相邻位置之间的相关性远高于远距离位置之间的相关性。

这个假设有坚实的统计学基础：

自然语言中的距离-相关性关系:
距离 1 (相邻词): 相关性 ~0.8-0.9
距离 5: 相关性 ~0.5-0.6
距离 20: 相关性 ~0.2-0.3
距离 100: 相关性 ~0.05-0.1
距离 1000+: 相关性 ~0.01

注意力分数的衰减:
Attention(i, j) ∝ exp(-α × |i-j|)  (近似指数衰减)

当 |i-j| > w 时，注意力分数通常接近零
→ 这些位置的贡献可以忽略
→ 只计算 |i-j| ≤ w 的位置是合理的

假设的语言学证据

例 1: 局部依赖
"那只 [黑色的] 猫 坐在 [温暖的] 垫子上"
形容词 "黑色的" 修饰 "猫"，"温暖的" 修饰 "垫子"
这些依赖关系都在局部窗口内 (距离 ≤ 3)

例 2: 长距离依赖
"那个 [昨天在公园里遇到的] 穿红衣服的 女孩 说 她 [很喜欢] 冰淇淋"
"女孩" 和 "很喜欢" 之间有较长距离
但这种长距离依赖在自然语言中的比例较低

统计:
- 句法依赖的平均距离: ~3-5 个词
- 90% 的依赖关系在 20 个词以内
- 只有 ~5% 的依赖关系超过 50 个词

假设何时被打破

长距离依赖在以下场景中变得重要：

  1. 指代消解："John went to the store. He bought milk." — "He" 指代 "John"，距离可能较远
  2. 文档级推理：需要综合文档不同部分的信息才能得出结论
  3. 代码理解：函数的定义和调用可能相距很远
  4. 数学证明：定理的引用可能在文档的其他位置

对于这些场景，纯滑动窗口注意力不够，需要引入全局 token 或稀释注意力来补充长距离信息。

分析框架：局部性 × 效率 × 感受野

三维分析框架

                    感受野 (Receptive Field)
                         ↑
                         |
                    全局 (n)
                         |
                         |
                    局部 (w) ───────→ 效率 (Efficiency)
                   /                  O(n × w) → O(n^2)
                  /
                 /
            无信息

维度 1：局部性 - 滑动窗口注意力是"局部"的——每个位置只关注附近的 w 个位置 - 局部性越强（w 越小），计算效率越高，但信息越不完整 - 最优的 w 取决于任务和数据

维度 2：效率 - 计算效率: O(n × w) vs O(n^2) - 内存效率: KV Cache O(w) vs O(n) - 硬件效率: 与 FlashAttention 完美兼容

维度 3：感受野 - 单层感受野: w - L 层堆叠感受野: L × (w-1) + 1 - 加上全局 token: 有效感受野 = 全局

窗口大小 w 的选择

不同任务的推荐窗口大小:

文本理解:
- 句子级: w = 64-128 (覆盖大多数句子)
- 段落级: w = 256-512 (覆盖大多数段落)
- 文档级: w = 512-1024 + 全局 token

代码理解:
- 函数级: w = 256-512
- 文件级: w = 1024-2048

图像处理 (ViT):
- 局部特征: w = 7×7 = 49 patches
- 中等特征: w = 14×14 = 196 patches

自动驾驶:
- 短期时序: w = 3-10 帧
- 中期时序: w = 10-30 帧

Mistral 7B 的选择: w = 4096
- 32 层堆叠后感受野: 32 × 4095 = 131,040 tokens
- 覆盖了 128K 上下文窗口的绝大部分

核心结论：局部注意力是长序列处理的最优折中

为什么滑动窗口注意力是最优折中

在所有稀疏注意力方案中，滑动窗口注意力以其简洁性和高效性脱颖而出：

方案对比:
1. 随机稀疏: 不可预测，硬件不友好
2. 哈希稀疏 (Reformer): 需要额外计算哈希，硬件效率低
3. 路由稀疏 (Routing): 需要负载均衡，实现复杂
4. 滑动窗口: 固定模式，硬件极度友好 ✓
5. 线性注意力: 改变了注意力的数学定义，质量有损

滑动窗口注意力的优势:
- 注意力模式固定 → 可以高度优化
- 不需要额外计算 → 纯效率提升
- 与 FlashAttention 完美兼容 → 硬件效率极高
- KV Cache 固定大小 → 内存可预测
- 多层堆叠扩大感受野 → 不损失长距离信息

数学证明的支撑

BigBird 的理论结果提供了坚实的数学基础：

定理: 滑动窗口 + 全局注意力 = Universal Approximator

含义:
- 理论上，滑动窗口注意力可以逼近任何全量注意力能表达的函数
- 不需要随机连接（实践中）
- 只需要少量全局 token

这是一个非常强的结论:
它意味着滑动窗口注意力不是"妥协"，而是"等价"
唯一的代价是需要更多层来扩大感受野

精神内核："不看完所有，只看附近"

哲学内涵

滑动窗口注意力体现了一种深刻的认知哲学：你不需要看到所有事情才能做出好的决策。

这与人类认知的方式惊人地相似： - 人类阅读时，眼睛每次只聚焦在一小段文字上（约 7-8 个字符） - 通过快速移动视线（saccade），我们有效地"滑动窗口"过整个文本 - 我们的大脑通过多次"局部注视"构建了对整个文档的理解

人类阅读 vs 滑动窗口注意力:

人类眼睛:
- 每次注视 (fixation): ~200ms
- 每次看到: ~7-8 个字符 (≈ "窗口大小")
- 通过 saccade 移动窗口
- 大脑整合多次注视的信息

滑动窗口注意力:
- 每个位置: 关注 w 个邻居 ("窗口大小")
- 通过多层堆叠整合信息
- 全局 token 类似人类的"余光"
- 最终获得全局理解

工程哲学

在工程层面，滑动窗口注意力体现了"分而治之"的经典思想：

分而治之的三个层次:

1. 空间分治: 将长序列分成局部窗口
   - 每个窗口独立处理
   - 减少计算量

2. 时间分治: 多层堆叠逐步扩大感受野
   - 第 1 层: 看到 w 个位置
   - 第 2 层: 看到 2w 个位置
   - 第 L 层: 看到 L×w 个位置

3. 信息分治: 全局 token 传递跨窗口信息
   - 局部窗口处理局部信息
   - 全局 token 传递全局信息
   - 两者互补

一句话总结

滑动窗口注意力的智慧在于：承认我们无法同时关注一切，但通过巧妙地组合局部观察，我们可以理解整体。这不是妥协，而是一种更高级的认知策略。

实际案例分析

Mistral 7B 的滑动窗口设计

Mistral 7B 是第一个使用滑动窗口注意力的主流大语言模型，其设计选择值得深入分析：

Mistral 7B 配置:
- 层数: 32
- 窗口大小: 4096
- 注意力头数: 32
- head_dim: 128
- 上下文长度: 8K (训练), 128K (推理)

设计分析:
1. 为什么 w=4096?
   - 32 层 × 4096 = 131,072 感受野
   - 覆盖了 128K 上下文窗口的绝大部分
   - 每层的 KV Cache: 4096 × 2 × 32 × 128 × 2 = ~64 MB
   - 总 KV Cache: 64 MB × 32 = ~2 GB (远小于全量的 ~16 GB)

2. 为什么不用全量注意力?
   - 全量 KV Cache (128K): ~16 GB
   - 滑动窗口 KV Cache (w=4096): ~2 GB
   - 缩减 8 倍，质量几乎无损

3. 质量验证:
   - Mistral 7B 在 MMLU 上达到 60.1%
   - 同等规模的全量注意力模型: ~60-61%
   - 差距 < 1%，但内存节省 8 倍

Longformer 在长文档中的应用

场景: 法律文档分析 (平均长度 16K tokens)

全量注意力:
- 注意力矩阵: 16K × 16K = 256M → 需要 ~1 GB 内存
- 在单 GPU 上勉强可行，但很紧张

Longformer (w=512, 全局 token=16):
- 注意力矩阵: 每行只有 512+16 = 528 个非零值
- 总非零值: 16K × 528 = 8.4M → 只需 ~32 MB
- 内存减少 32 倍
- 速度提升 ~20 倍

质量对比:
- 全量注意力: F1 = 82.3%
- Longformer: F1 = 80.1%
- 差距 2.2%，但效率提升巨大

Mistral 的设计对行业的启示

Mistral 7B 的成功证明了滑动窗口注意力在主流 LLM 中的可行性。此后，越来越多的模型开始采用类似的设计： - Mixtral 8x7B: 继承 Mistral 的滑动窗口设计 - DeepSeek-V2: 在 MLA 基础上使用局部注意力 - 多个开源模型在微调时默认启用滑动窗口注意力

这一趋势表明，滑动窗口注意力正在从"长序列的特殊需求"变为"所有 Transformer 模型的默认选择"。
