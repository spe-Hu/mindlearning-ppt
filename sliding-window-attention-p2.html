论文 1：Longformer: The Long-Document Transformer (Beltagy et al., 2020)

基本信息

  • 标题：Longformer: The Long-Document Transformer
  • 作者：Iz Beltagy, Matthew E. Peters, Arman Cohan
  • 发表：arXiv:2004.05150
  • 意义：滑动窗口注意力的奠基之作，提出了局部+全局的混合注意力模式

论文核心内容

Longformer 的核心贡献是提出了一种高效的注意力机制，使 Transformer 能够处理长达数千乃至数万个 token 的文档。

核心设计：

Longformer 的注意力由两个部分组成：

1. 滑动窗口注意力 (Sliding Window Attention):
   每个位置 i 只关注 [i-w, i+w] 范围内的 token
   其中 w 是窗口大小（半宽）

   数学表达:
   A_i = Attention(Q_i, K_{max(0,i-w):min(n,i+w+1)}, V_{max(0,i-w):min(n,i+w+1)})

   计算复杂度: O(n × w) 每层
   内存复杂度: O(n × w) 每层

2. 全局注意力 (Global Attention):
   选定的全局 token 关注所有位置，所有位置也关注这些全局 token

   全局 token 的选择:
   - [CLS] token (用于分类任务)
   - 段落分隔符 (用于文档理解)
   - 问题 token (用于 QA 任务)

   计算复杂度: O(n × g) 每层 (g 为全局 token 数)

信息流分析：

单层: 每个 token 只能看到 w 个邻居
两层: 感受野扩展到 2w
L 层: 感受野扩展到 L × w

全局 token 的作用:
- 全局 token 在每一层都能看到所有位置
- 其他 token 通过全局 token 间接获取远距离信息
- 信息流路径: token_i → 全局token → token_j (2 跳即可传递全局信息)

稀疏注意力矩阵的可视化：

全量注意力矩阵 (n × n):

████████████████
████████████████
████████████████

Longformer 注意力矩阵:

▓██▓█..............   ▓ = 滑动窗口

最后一列和某些行是全局 token

代码示例

import torch
import torch.nn.functional as F

def sliding_window_attention(Q, K, V, window_size):
    """
    滑动窗口注意力的简化实现
    Q, K, V: [batch, seq_len, heads, dim]
    window_size: 半窗口宽度 w
    """
    batch, n, heads, dim = Q.shape

    # 创建滑动窗口掩码
    # mask[i, j] = True if |i - j| <= window_size
    indices = torch.arange(n, device=Q.device)
    mask = (indices.unsqueeze(0) - indices.unsqueeze(1)).abs() <= window_size
    # mask: [n, n] 的布尔矩阵

    # 计算注意力分数
    # Q: [batch, n, heads, dim] -> [batch, heads, n, dim]
    q = Q.permute(0, 2, 1, 3)
    k = K.permute(0, 2, 1, 3)
    v = V.permute(0, 2, 1, 3)

    scores = torch.matmul(q, k.transpose(-2, -1)) / (dim ** 0.5)
    # scores: [batch, heads, n, n]

    # 应用滑动窗口掩码
    scores = scores.masked_fill(~mask.unsqueeze(0).unsqueeze(0), float('-inf'))

    # Softmax
    attn_weights = F.softmax(scores, dim=-1)

    # 输出
    output = torch.matmul(attn_weights, v)
    # output: [batch, heads, n, dim]

    return output.permute(0, 2, 1, 3)

深远影响

Longformer 的影响远超长文档处理： 1. 确立了滑动窗口注意力作为高效注意力的标准方案 2. 启发了 BigBird、BigBird-Pegasus 等后续工作 3. 其稀疏注意力模式被 FlashAttention 原生支持 4. 成为 Mistral、DeepSeek 等模型中滑动窗口注意力的理论源头

论文 2：Big Bird: Transformers for Longer Sequences (Zaheer et al., 2020)

基本信息

  • 标题：Big Bird: Transformers for Longer Sequences
  • 作者：Manzil Zaheer, Guru Guruganesh, Avinava Dubey, et al. (Google Research)
  • 发表：NeurIPS 2020
  • 意义：为滑动窗口注意力提供了理论基础

论文核心内容

BigBird 的核心贡献在于从理论角度证明了稀疏注意力的有效性。

BigBird 的注意力模式：

BigBird 结合了三种注意力:
1. 滑动窗口 (Sliding): 每个 token 关注 w 个邻居
2. 全局 (Global): 少数 token 关注所有位置
3. 随机 (Random): 每个 token 随机关注 c 个其他位置

总注意力: A = A_sliding + A_global + A_random
每行非零元素数: w + g + c (远小于 n)

理论贡献：

BigBird 论文最重要的贡献是以下定理：

定理 (Zaheer et al., 2020):
设 G 是一个注意力图，如果 G 包含:
(a) 滑动窗口连接 (每个节点连接 w 个邻居)
(b) 全局连接 (g 个节点连接所有其他节点)
(c) 随机连接 (每个节点随机连接 c 个其他节点)

则基于 G 的稀疏注意力是 universal approximator:
对于任意连续函数 f 和 ε > 0，存在一个基于 G 的 Transformer
使得 |Transformer(x) - f(x)| < ε

关键参数选择:
- w = O(1) (常数窗口大小即可)
- g = O(1) (常数个全局 token 即可)
- c = O(log n) (随机连接数只需对数级)

这意味着: 稀疏注意力在理论上与全量注意力等价!

实践中的简化：

实际应用中，随机连接通常被省略:
- BigBird (完整版): 滑动 + 全局 + 随机 → 理论最优
- BigBird (简化版): 滑动 + 全局 → 实践中效果接近
- Longformer: 滑动 + 全局 → 与简化版 BigBird 等价

随机连接的价值:
- 理论上: 保证 universal approximation
- 实践中: 提升 < 1%，但增加了实现的复杂性
- 结论: 大多数应用中使用滑动 + 全局即可

实验结果

BigBird 在多个长序列任务上进行了评测：

任务 序列长度 BERT Longformer BigBird
摘要 (arXiv) 4096 29.1 30.4 31.2
QA (QASper) 4096 42.3 45.8 47.1
分类 (AG News) 4096 91.2 91.5 91.8
基因组序列 8192 OOM 可行 可行

深远影响

BigBird 的理论贡献为整个稀疏注意力领域提供了信心： 1. 证明了稀疏注意力不是"退而求其次"，而是理论上等价 2. 指导了注意力模式的设计（哪些连接是必要的） 3. 启发了后续对注意力稀疏性的更深入理解

论文 3：FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness (Dao et al., 2022)

基本信息

  • 标题：FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness
  • 作者：Tri Dao, Daniel Y. Fu, Stefano Ermon, Atri Rudra, Christopher Ré
  • 发表：NeurIPS 2022
  • 意义：通过 IO 感知的 GPU 优化，使注意力计算（包括滑动窗口）在硬件上高效执行

论文核心内容

FlashAttention 的核心洞察是：注意力计算的瓶颈不是计算量，而是内存访问（IO）。

GPU 内存层次：

SRAM (片上缓存):
- 容量: ~20 MB (A100)
- 带宽: ~19 TB/s
- 延迟: ~ns

HBM (高带宽内存):
- 容量: 40-80 GB
- 带宽: ~2 TB/s
- 延迟: ~100ns

关键观察:
- SRAM 带宽是 HBM 的 ~10000 倍
- 但 SRAM 容量远小于 HBM
- 传统实现在 HBM 中做计算 → 带宽瓶颈
- FlashAttention 在 SRAM 中做计算 → 利用高带宽

Tiling 算法：

传统注意力实现:
1. 在 HBM 中加载 Q, K, V
2. 在 HBM 中计算 S = QK^T (n × n 矩阵)
3. 在 HBM 中计算 P = softmax(S)
4. 在 HBM 中计算 O = PV
5. 需要存储 n × n 的中间矩阵 → 内存 O(n^2)

FlashAttention:
1. 将 Q, K, V 分成块 (tiles)，每块大小 B_r × B_c
2. 每次加载一个 tile 到 SRAM
3. 在 SRAM 中计算局部注意力
4. 使用 online softmax 技巧累积结果
5. 不需要存储完整的 n × n 矩阵 → 内存 O(n)

FlashAttention + 滑动窗口：

FlashAttention 2/3 原生支持滑动窗口注意力：

在 tiling 过程中:
- 对于位置 i 的 tile，只需要加载 [i-w, i+w] 范围内的 K, V tiles
- 跳过窗口外的 tiles → 直接减少计算和内存访问
- 实际加速: 与 w/n 成正比

性能对比 (n=32K, w=256):
全量注意力 + FlashAttention: ~12 ms
滑动窗口 + FlashAttention: ~0.1 ms
加速比: ~120x

深远影响

FlashAttention 改变了整个注意力计算的实现范式： 1. 使长序列 Transformer 训练和推理成为可能 2. 原生支持滑动窗口注意力，两者协同效果极佳 3. 被几乎所有主流框架（PyTorch、DeepSpeed、Megatron）采用 4. 启发了 FlashAttention-2（2x 加速）和 FlashAttention-3（Hopper 优化）

论文 4：DeepSeek DSpark (DeepSeek, 2025)

基本信息

  • 标题：DeepSeek DSpark: Efficient Parallel Backbone for Long-Context Processing
  • 作者：DeepSeek-AI
  • 发表：2025 (技术报告)
  • 意义：将滑动窗口注意力作为并行主干的核心，实现高效长上下文处理

论文核心内容

DSpark 是 DeepSeek 在长上下文处理方面的重要创新。其核心思想是将滑动窗口注意力与 MLA 结合，构建一个高效的并行主干。

架构设计：

DSpark 架构:
1. 输入分块: 将长序列分成多个块 (chunks)
2. 并行处理: 每个块独立通过滑动窗口注意力处理
3. 跨块通信: 通过全局 token 或交叉注意力传递信息
4. 合并输出: 将所有块的输出合并

关键创新:
- 滑动窗口注意力作为"局部处理器"
- 全局 token 作为"信息中继站"
- MLA 压缩 KV Cache
- 三者协同，实现高效长上下文处理

与 MLA 的协同：

MLA 压缩 KV: 每 token 的 KV 从 2 × h × d_k 压缩到 d_c
滑动窗口: KV Cache 从 n 个 token 减少到 w 个 token

两者叠加:
KV Cache = w × d_c × L × dtype_size

例: w = 256, d_c = 512, L = 60, FP16
KV Cache = 256 × 512 × 60 × 2 = ~15.7 MB

对比全量注意力 + MHA:
KV Cache = 128K × 2 × 128 × 128 × 80 × 2 = ~41 GB

缩减比: ~2600x!

深远影响

DSpark 代表了滑动窗口注意力的最新应用范式： 1. 滑动窗口注意力不再是"妥协"，而是"设计选择" 2. 与 MLA 的协同展示了多种优化技术的叠加效应 3. 为未来高效长上下文模型的设计提供了蓝图
