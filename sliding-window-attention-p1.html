滑动窗口注意力 (Sliding Window Attention) 横纵深度分析

纵轴：从全量注意力到滑动窗口的演进史

第一纪元：全量注意力的辉煌与困境（2017-2019）

2017 年 Transformer 的诞生带来了自注意力机制（Self-Attention），其核心公式为：

Attention(Q, K, V) = softmax(QK^T / √d_k) V

这个机制让序列中的每个位置都能直接关注所有其他位置，完美解决了 RNN/LSTM 的长距离依赖问题。然而，这个"全连接"的设计也带来了严重的效率问题：

计算复杂度：对于长度为 n 的序列，注意力矩阵的大小为 n × n，计算复杂度为 O(n^2 × d)。

内存复杂度：需要存储 n × n 的注意力矩阵，内存复杂度为 O(n^2)。

在 2017-2019 年的应用场景中（机器翻译，序列长度 ~50-200），这些复杂度完全不是问题。但当序列长度增长到数千乃至数十万时，O(n^2) 的复杂度成为不可逾越的障碍。

序列长度与计算/内存需求:
n = 512:    n^2 = 262K     → 微不足道
n = 2048:   n^2 = 4.2M     → 开始注意
n = 8192:   n^2 = 67M      → 明显瓶颈
n = 32K:    n^2 = 1.07B    → 严重瓶颈
n = 128K:   n^2 = 16.8B    → 几乎不可行
n = 1M:     n^2 = 1T       → 完全不可行

第二纪元：稀疏注意力的探索（2019-2021）

面对 O(n^2) 的复杂度困境，研究者们开始探索"稀疏注意力"（Sparse Attention）——不是让每个位置关注所有位置，而是只关注一部分位置。

关键思路：如果注意力矩阵是稀疏的（大部分位置的注意力分数接近零），那么只计算非零部分就可以大幅降低复杂度。

早期稀疏注意力方案：

  1. Sparse Transformer (Child et al., 2019)：
  2. 将注意力分解为"局部"和"全局"两种模式
  3. 局部注意力：每个位置关注附近的 w 个位置
  4. 全局注意力：少数"枢纽"位置关注所有位置

  5. 复杂度：O(n × √n)

  6. Routing Transformer (Roy et al., 2020)：

  7. 使用哈希将相似的 Q 和 K 分到同一组
  8. 只在组内计算注意力

  9. 复杂度：O(n × k)，k 为每组大小

  10. Reformer (Kitaev et al., 2020)：

  11. 使用局部敏感哈希（LSH）实现稀疏注意力
  12. 复杂度：O(n × log(n))

这些方案虽然降低了复杂度，但引入了额外的复杂性（哈希、路由等），且在实际硬件上的加速效果有限。

第三纪元：滑动窗口注意力的确立（2020-2022）

2020 年，两篇论文几乎同时确立了滑动窗口注意力作为最高效的稀疏注意力方案之一的地位。

Longformer (Beltagy et al., 2020)：

Longformer 提出了三种注意力模式的混合：

1. 滑动窗口注意力 (Sliding Window Attention):
   - 每个位置只关注前后 w 个位置
   - 复杂度: O(n × w)
   - 捕获局部上下文

2. 全局注意力 (Global Attention):
   - 选定的"全局"token（如 [CLS]、段落标记）关注所有位置
   - 所有位置也关注这些全局 token
   - 复杂度: O(n × g)，g 为全局 token 数
   - 实现全局信息流通

3. 稀释注意力 (Dilated Attention): [可选]
   - 每隔 d 个位置关注一次
   - 扩大感受野而不增加计算量

滑动窗口注意力的核心优势在于其简洁性： - 不需要哈希、路由等额外机制 - 注意力模式固定，可以高度优化 - 与 FlashAttention 等硬件优化完美兼容 - KV Cache 大小固定为 w，不随序列长度增长

BigBird (Zaheer et al., 2020)：

BigBird 从理论角度证明了滑动窗口注意力 + 全局注意力的组合具有 universal approximator 的性质：

BigBird 的注意力模式:
1. 滑动窗口 (Sliding): 每个 token 关注 w 个邻居
2. 全局 (Global): 少数 token 关注所有位置
3. 随机 (Random): 每个 token 随机关注 c 个其他位置

理论结果:
- 滑动窗口 + 全局 + 随机 = universal approximator
- 去掉随机成分，只用滑动 + 全局 = 仍然有效（实践中）
- 这为 Longformer/BigBird 的设计提供了理论基础

第四纪元：FlashAttention 与硬件感知优化（2022-2024）

2022 年，Dao et al. 提出的 FlashAttention 彻底改变了注意力计算的实现方式。虽然 FlashAttention 本身不改变注意力的数学定义（仍然是 O(n^2)），但它通过 IO 感知的 GPU 内存优化，使得滑动窗口注意力等稀疏模式在硬件上的执行效率大幅提升。

FlashAttention 的核心思想:
- 传统实现: 在 HBM 中计算完整的 n × n 注意力矩阵 → 内存瓶颈
- FlashAttention: 在 SRAM 中分块 (tiling) 计算注意力 → 减少 HBM 访问

FlashAttention + 滑动窗口:
- 注意力矩阵只有 w × n 的非零元素（每行只有 w 个非零值）
- FlashAttention 的 tiling 可以直接跳过零元素
- 实际计算量: O(n × w)
- 内存访问: O(n × w) 而非 O(n^2)

第五纪元：DSpark 与并行主干中的应用（2025-2026）

DeepSeek 的 DSpark 架构将滑动窗口注意力作为并行主干（parallel backbone）的核心组件。这标志着滑动窗口注意力从"长序列处理的妥协方案"升级为"高效推理的核心架构"。

DSpark 的架构设计:
1. 并行主干 (Parallel Backbone):
   - 使用滑动窗口注意力处理局部上下文
   - 多个窗口并行处理，提高吞吐量
   - KV Cache 大小固定为 w

2. 全局信息聚合:
   - 通过少量全局 token 或额外的聚合层传递全局信息
   - 不依赖完整的全局注意力

3. 与 MLA 的协同:
   - 滑动窗口注意力减少了需要缓存的 KV 数量
   - MLA 进一步压缩了每个 token 的 KV 大小
   - 两者叠加，KV Cache 被极度压缩

横轴：滑动窗口注意力与其他方案的对比

1. 与 FlashAttention 的关系

FlashAttention 和滑动窗口注意力不是替代关系，而是互补关系：

维度 FlashAttention 滑动窗口注意力
本质 计算优化（IO 感知） 算法优化（稀疏模式）
改变的是什么 注意力的实现方式 注意力的计算范围
复杂度变化 不改变 O(n^2) 降到 O(n × w)
质量影响 无损（数学等价） 有损（丢失远距离信息）
可组合性 可与任何注意力组合 可与任何实现组合

最佳实践：FlashAttention + 滑动窗口注意力 = 最优的硬件效率。

2. 与线性注意力的对比

线性注意力（如 Mamba、RWKV）通过核方法或状态空间模型将注意力复杂度降到 O(n)：

维度 滑动窗口注意力 线性注意力 (Mamba)
复杂度 O(n × w) O(n)
长距离依赖 有限 (w 范围内) 通过隐状态间接支持
KV Cache 固定大小 w 固定大小隐状态
并行训练 支持 支持
硬件效率 极高 (FlashAttention 兼容) 高 (顺序扫描)
模型质量 接近全量注意力 略低于 Transformer
实现复杂度 低 中

3. 与全量注意力的对比

维度 全量注意力 滑动窗口注意力
复杂度 O(n^2) O(n × w)
信息流 全局 局部 + 全局 token
KV Cache O(n) O(w)
质量 最优 接近最优 (w 足够大时)
适用场景 短序列 (< 8K) 长序列 (> 8K)

4. 与其他稀疏注意力的对比

方案 复杂度 感受野 实现难度 硬件效率
全量注意力 O(n^2) 全局 低 中

核心优势：将 O(n^2) 降到 O(n × w)

数学分析

滑动窗口注意力的核心是将每个位置的注意力范围从全部 n 个位置缩小到局部 w 个位置：

全量注意力:
对于位置 i, 计算 Attention(Q_i, K_{1:n}, V_{1:n})
每行计算量: O(n × d)
总计算量: O(n^2 × d)

滑动窗口注意力:
对于位置 i, 计算 Attention(Q_i, K_{i-w/2:i+w/2}, V_{i-w/2:i+w/2})
每行计算量: O(w × d)
总计算量: O(n × w × d)

加速比: n/w
当 n = 128K, w = 256: 加速比 = 500x

KV Cache 的影响

滑动窗口注意力对 KV Cache 的影响是革命性的：

全量注意力 KV Cache:
- 大小: O(n × L × d) 随序列长度线性增长
- n = 128K: ~41 GB (LLaMA-2 70B)

滑动窗口注意力 KV Cache:
- 大小: O(w × L × d) 固定，不随序列长度增长
- w = 256: ~82 MB (LLaMA-2 70B 配置)
- 缩减比: ~500x

这意味着:
- 同样的 GPU 显存可以支持 500 倍的并发请求
- 或者同样的并发下支持 500 倍的序列长度

感受野分析

滑动窗口注意力的一个关键问题是：它能否捕获长距离依赖？

单层滑动窗口: 感受野 = w
两层堆叠: 感受野 = 2w - 1 ≈ 2w
L 层堆叠: 感受野 = L × (w-1) + 1 ≈ L × w

例: w = 256, L = 32
感受野 = 32 × 255 + 1 = 8,161 tokens

如果 w = 512, L = 80 (LLaMA-2 70B):
感受野 = 80 × 511 + 1 = 40,881 tokens

结论: 通过堆叠多层，滑动窗口注意力的有效感受野可以覆盖很长的序列

与全局 token 的协同

为了弥补滑动窗口注意力的局部性，通常引入少量全局 token：

全局 token 的作用:
1. [CLS] token: 聚合全局信息
2. 段落标记: 每个段落的第一个 token 关注所有位置
3. 特殊 token: 用于传递跨窗口信息

全局 token 的计算量:
- g 个全局 token × n 个位置 = O(g × n)
- 当 g << w 时，全局注意力的计算量远小于滑动窗口
- 总复杂度: O(n × w + g × n) ≈ O(n × w) (当 w > g)

2026 年现状总结

滑动窗口注意力在 2026 年的地位：

  1. 长序列处理的标准方案：几乎所有支持 128K+ 上下文的模型都使用了某种形式的滑动窗口注意力或局部注意力。

  2. 与 FlashAttention 深度绑定：FlashAttention 2/3 原生支持滑动窗口注意力，两者结合是当前最高效的注意力计算方案。

  3. DSpark 等新架构的核心组件：滑动窗口注意力不再只是"长序列的妥协"，而是高效推理架构的核心设计元素。

  4. KV Cache 优化的天然盟友：滑动窗口注意力将 KV Cache 从 O(n) 降到 O(w)，与 GQA/MLA 等优化形成叠加效应。

  5. 多模态和自动驾驶的关键技术：在处理高分辨率图像和视频时，滑动窗口注意力是控制计算量的关键手段。

滑动窗口注意力的故事告诉我们：在效率和质量之间，最优的平衡往往不是"全部"或"没有"，而是"恰到好处"。只看附近（局部窗口），加上少量全局信息，就能在 O(n × w) 的复杂度下获得接近全量注意力的质量。
